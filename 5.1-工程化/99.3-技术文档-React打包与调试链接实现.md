## 第一部分：顶层鸟瞰

### 背景问题
这批截图在解决一个问题：**如何使用 Rollup 打包 React Monorepo 项目，通过 pnpm link 将本地 React 包链接到测试项目中，实现本地开发调试和验证 jsxDEV 函数的功能。**

### 宏观图

```
jsxDEV 源码（TypeScript）→ Rollup 编译 → 输出（JavaScript + package.json）
                                          ↓
                                   pnpm link --global
                                          ↓
                             全局 node_modules/react
                                          ↓
                            测试项目 pnpm link react --global
                                          ↓
                              测试项目可以使用本地 React
```

**本节实现目标：**
1. ✅ 实现 jsxDEV 方法并导出
2. ✅ 打包并通过 pnpm link 链接到测试项目
3. ✅ 验证 JSX 转换功能正常工作
4. ✅ 掌握热更新调试流程


### 开发流程

编写 jsxDEV → Rollup 打包 → pnpm link 链接 → 测试项目验证

```bash
# 步骤 1：实现 jsxDEV 方法
# 编写 packages/react/src/jsx.ts

# 步骤 2：在 big-react 项目中打包
pnpm build:dev

# 步骤 3：链接到全局
cd dist/node_modules/react
pnpm link --global

# 步骤 4：在测试项目中使用
cd react-demo
pnpm link react --global

# 步骤 5：启动测试项目
npm start
```

**关键技术栈：**

| 技术 | 作用 |
|------|------|
| **jsxDEV** | 开发环境的 JSX 转换方法（children 在 config 中，不需要 ...maybeChildren） |
| **Rollup** | 打包工具，将 TypeScript 编译成 JavaScript（配置已实现） |
| **generatePackageJson** | Rollup 插件，自动生成 package.json |
| **pnpm link --global** | 创建全局符号链接，用于本地调试 |

### 步骤 1：实现 jsxDEV 方法

#### 1.1 packages/react/src/jsx.ts（jsxDEV 实现）

**路径：** `packages/react/src/jsx.ts`

**完整实现：**

```typescript
import { REACT_ELEMENT_TYPE } from 'shared/ReactSymbols';
import {
  Type,
  Key,
  Ref,
  Props,
  ReactElementType,
  ElementType
} from 'shared/ReactTypes';

// ReactElement 工厂函数
const ReactElement = function (
  type: Type,
  key: Key,
  ref: Ref,
  props: Props
): ReactElementType {
  const element = {
    $$typeof: REACT_ELEMENT_TYPE,
    type,
    key,
    ref,
    props,
    __mark: 'Ygm'                        // ← 自定义标记
  };
  return element;
};

// jsxDEV 方法（开发环境）
export const jsxDEV = (type: ElementType, config: any) => {
  let key: Key = null;
  const props: Props = {};
  let ref: Ref = null;

  // 处理 config 中的属性
  for (const prop in config) {
    const val = config[prop];
    
    // 提取 key
    if (prop === 'key') {
      if (val !== undefined) {
        key = '' + val;
      }
      continue;
    }
    
    // 提取 ref
    if (prop === 'ref') {
      if (val !== undefined) {
        ref = val;
      }
      continue;
    }
    
    // 其他属性放入 props（安全检查）
    if ({}.hasOwnProperty.call(config, prop)) {
      props[prop] = val;
    }
  }

  // 返回 ReactElement
  return ReactElement(type, key, ref, props);
};
```

**关键点说明：**

| 实现细节 | 说明 |
|---------|------|
| **函数签名** | `jsxDEV = (type, config)` - 不接收 `...maybeChildren` 参数 |
| **children 处理** | children 已经在 `config.children` 中，通过 `for...in` 循环自动赋值 |
| **开发环境特性** | jsxDEV 是专门为开发环境设计的，编译器会将 children 打包到 config 中 |
| **`{}.hasOwnProperty.call()`** | 安全检查，避免原型链污染 |
| **`__mark: 'Ygm'`** | 自定义标记，用于调试 |

**jsxDEV 与 jsx 的区别：**

| 特性 | jsx（生产环境） | jsxDEV（开发环境） |
|------|----------------|------------------|
| **函数签名** | `jsx(type, config, ...maybeChildren)` | `jsxDEV(type, config)` |
| **children 来源** | 通过剩余参数 `...maybeChildren` 接收 | 已经在 `config.children` 中 |
| **编译器处理** | Babel 将 children 作为单独的参数传递 | Babel 将 children 放在 config 对象中 |
| **使用场景** | 生产环境（React 17+） | 开发环境（更多调试信息） |

**为什么 jsxDEV 不需要处理 children 参数？**

```jsx
// JSX 代码
<div>hello<span>world</span></div>

// 编译结果（生产环境 - jsx）
jsx('div', {}, 'hello', jsx('span', {}, 'world'))

// 编译结果（开发环境 - jsxDEV）
jsxDEV('div', {
  children: ['hello', jsxDEV('span', { children: 'world' })]
})
```

在开发环境中，Babel 会将 children 直接放到 config 对象中，所以 jsxDEV 函数不需要额外的 `...maybeChildren` 参数。

#### 1.2 packages/react/index.ts（导出 jsxDEV）

**路径：** `packages/react/index.ts`

```typescript
import { jsxDEV } from './src/jsx';

// React
export default {
  version: '0.0.0',
  createElement: jsxDEV          // ← 导出 jsxDEV 作为 createElement
};
```

**说明：**
- 默认导出包含 `version` 和 `createElement`
- 开发环境中，`createElement` 实际指向 `jsxDEV`
- 这样做是为了保持 API 的兼容性

## 第三部分：问题排查与解决

### 常见问题：pnpm link react --global 失败

#### 问题现象

在 CRA（Create React App）项目中执行 `pnpm link react --global` 时报错：

```bash
ERROR  Symlink path is the same as the target path 
(/Users/xxx/Library/pnpm/global/5/node_modules/react)
```

#### 根本原因

**pnpm 10.x 在 npm-based 项目（如 CRA）中存在 bug**

1. **环境冲突**：
   - CRA 默认使用 npm 管理依赖（有 `package-lock.json`）
   - node_modules 是 npm 的扁平结构
   - pnpm 10.x 在处理混合环境时，路径解析逻辑有问题

2. **技术细节**：
   - pnpm 检测到 `node_modules/react` 已存在（npm 安装的）
   - 尝试创建符号链接时，误认为"符号链接路径"和"目标路径"相同
   - 触发防重复检查，导致报错

#### 解决方案对比

| 方案 | 可行性 | 适用场景 | 配置复杂度 |
|------|--------|----------|-----------|
| **方案1：npm link** | ✅ 完美 | 所有 CRA 项目 | ⭐ 简单 |
| **方案2：pnpm 8 + React 18** | ✅ 完美 | 想严格使用 pnpm | ⭐⭐ 中等 |
| **方案3：Vite + pnpm 10** | ✅ 可行 | 新项目 | ⭐⭐⭐ 较高 |
| ❌ pnpm 9/10 + CRA | ❌ 失败 | - | - |

---

#### 解决方案1：使用 npm link（推荐）

**适用场景**：只想快速调试，不在意工具链细节

**步骤：**

```bash
# 1. 在 React-P1 打包后创建全局链接
cd React-P1/dist/node_modules/react
npm link

# 2. 在 react-demo 中链接
cd react-demo
npm link react

# 3. 验证
ls -la node_modules/react  # 应该是符号链接
```

**优点：**
- ✅ 最简单、最稳定
- ✅ 适用所有 npm/pnpm 版本
- ✅ 与 pnpm link 效果完全一样

**缺点：**
- ⚠️ 不是"纯 pnpm"工作流（但功能相同）

**适用场景**：团队规范要求 pnpm，或想严格按文档使用 pnpm link

**完整步骤：**

#### 技术原理：为什么 npm link 和 pnpm link 效果一样

**符号链接（Symlink）的本质：**

```bash
# npm link 创建的链接
node_modules/react -> ../../React-P1/dist/node_modules/react

# pnpm link 创建的链接
node_modules/react -> ../../../../Library/pnpm/global/5/node_modules/react
                   -> ../../React-P1/dist/node_modules/react
```

**关键点：**
1. 两者都是文件系统级别的符号链接
2. Node.js 的 `require()` 和 `import` 会透明地跟随符号链接
3. 最终都指向同一个真实文件：`React-P1/dist/node_modules/react`
4. **功能完全一致**，只是路径层级不同

**验证链接是否正确：**

```bash
cd react-demo

# 检查是否是符号链接
ls -la node_modules/react

# 查看链接目标
readlink node_modules/react

# 验证版本（应该是 1.0.0，不是 18.x 或 19.x）
cat node_modules/react/package.json | grep version

# 验证自定义标记
grep "__mark" node_modules/react/index.js
```
