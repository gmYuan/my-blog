# Monorepo 工程化配置完整指南

---

## 第一部分：顶层鸟瞰

### 背景问题
这批截图在解决一个问题：**如何从零搭建一个标准的 pnpm monorepo 项目，并配置完整的代码规范工具链（ESLint + Prettier + Husky + CommitLint）**。

### 宏观图

```
big-react 项目根目录
├── .husky/                  # Git hooks 钩子
│   ├── pre-commit          # 提交前检查（运行 lint）
│   └── commit-msg          # 提交信息检查（commitlint）
├── .vscode/                # VSCode 配置
│   └── tsconfig.json       # TypeScript 配置
├── packages/               # Monorepo 子包目录
│   └── test.js            # 测试文件
├── scripts/               # 构建脚本
│   └── rollup/            # rollup 构建配置
│       └── .commitlintrc.js  # commitlint 配置
├── .eslintrc.json         # ESLint 配置
├── .gitignore             # Git 忽略规则
├── .prettierrc.json       # Prettier 配置
├── package.json           # 根 package.json
├── pnpm-lock.yaml         # pnpm 依赖锁定文件
├── pnpm-workspace.yaml    # pnpm workspace 配置
└── tsconfig.json          # TypeScript 配置
```

**这个工程化方案解决的问题：**

| 工具 | 解决什么问题 |
|------|-------------|
| **pnpm workspace** | Monorepo 包管理（多个子包共享依赖） |
| **ESLint** | 代码质量检查（找出代码错误和潜在问题） |
| **Prettier** | 代码格式统一（避免团队风格不一致） |
| **Husky** | Git hooks 管理（在 commit 前自动执行检查） |
| **CommitLint** | commit 信息规范检查（统一 commit 格式） |
| **Rollup** | 打包工具（用于构建生产代码） |

## 第二部分：实战步骤

---

### 步骤 1：Pnpm 初始化

**做了什么：** 全局安装 pnpm 并初始化项目。

**完整代码：**

```bash
# 全局安装 pnpm
npm install -g pnpm

# 初始化 pnpm 项目
pnpm init
```

**注意点：**
- 这会在项目根目录生成 `package.json` 文件
- pnpm 是高性能的 npm 替代品，节省磁盘空间且安装更快

---

### 步骤 2：配置 pnpm workspace

**做了什么：** 创建 `pnpm-workspace.yaml` 文件，启用 monorepo 模式。

**完整代码：**

创建文件 `pnpm-workspace.yaml`：

```yaml
packages:
  - 'packages/*'
```

**注意点：**
- `packages:` 后面有冒号，这是 YAML 语法
- `'packages/*'` 表示 packages 目录下所有子目录都是独立的包
- 配置后，pnpm 会识别 packages 下的所有子包

---

### 步骤 3：安装 ESLint

**做了什么：** 安装 ESLint 到 workspace 根目录。

**完整代码：**

```bash
pnpm i eslint -D -w
```

**参数说明：**

| 参数 | 作用 |
|------|------|
| `-D` | 安装为 devDependencies（开发依赖） |
| `-w` | 安装到 workspace 根目录 |

---

### 步骤 4：初始化 ESLint 配置

**做了什么：** 运行 ESLint 初始化向导，生成配置文件。

**完整代码：**

```bash
npx eslint --init
```

**初始化过程的选项：**

| 问题 | 选择 |
|------|------|
| What type of modules does your project use? | **esm**（ES 模块） |
| Which framework does your project use? | **none** |
| Does your project use TypeScript? | **Yes** |
| Where does your code run? | **browser** |
| What format do you want your config file to be in? | **JSON** |
| Would you like to install them now? | **Yes** |
| Which package manager do you want to use? | **pnpm** |

**注意点：**
- 会自动安装 `@typescript-eslint/parser` 和 `@typescript-eslint/eslint-plugin`
- 底部显示红色警告，这是因为依赖安装过程中的正常提示

---

### 步骤 5：处理 TypeScript 依赖警告

**做了什么：** 手动安装 TypeScript 相关依赖以解决警告。

**完整代码：**

```bash
# 安装 TypeScript ESLint 插件和解析器
pnpm i -D -w @typescript-eslint/eslint-plugin @typescript-eslint/parser

# 安装 TypeScript
pnpm i -D -w typescript
```

**易错点：**
- ⚠️ **警告原因：** pnpm 在 workspace 模式下对依赖路径检查很严格
- ⚠️ **解决方案：** 手动安装这些依赖即可，不影响使用
- ⚠️ **peer dependencies 警告：** 可以忽略

---

### 步骤 6：配置 ESLint 基础规则

**做了什么：** 编辑 `.eslintrc.json`，配置基础的 ESLint 规则（第一版，不含 Prettier）。

**完整代码 `.eslintrc.json`（第一版）：**

```json
{
  "env": {
    "browser": true,
    "es2021": true,
    "node": true
  },
  "extends": [
    "eslint:recommended",
    "plugin:@typescript-eslint/recommended",
    "prettier",
    "plugin:prettier/recommended"
  ],
  "parser": "@typescript-eslint/parser",
  "parserOptions": {
    "ecmaVersion": "latest",
    "sourceType": "module"
  },
  "plugins": ["@typescript-eslint", "prettier"],
  "rules": {
    "prettier/prettier": "error",
    "no-case-declarations": "off",
    "no-constant-condition": "off",
    "@typescript-eslint/ban-ts-comment": "off"
  }
}
```

**注意点：**
- ⚠️ **这是改造前的版本**：配置已经包含 prettier 相关配置
- 实际操作时可能先创建了不含 prettier 的版本，然后逐步添加

### 步骤 8：安装 Prettier

**完整代码：**

### 步骤 9：安装 ESLint-Prettier 集成插件

**做了什么：** 安装 `eslint-config-prettier` 和 `eslint-plugin-prettier`。

**完整代码：**

```bash
# 安装 ESLint 和 Prettier 集成插件
pnpm i eslint-config-prettier eslint-plugin-prettier -D -w
```

**插件说明：**

| 插件 | 作用 |
|------|------|
| `eslint-config-prettier` | 关闭 ESLint 中与 Prettier 冲突的格式化规则 |
| `eslint-plugin-prettier` | 将 Prettier 作为 ESLint 规则运行 |

---

### 步骤 10：创建 Prettier 配置文件

**做了什么：** 创建 `.prettierrc.json` 文件，配置 Prettier 格式化规则。

**完整代码 `.prettierrc.json`：**

```json
{
  "printWidth": 80,
  "tabWidth": 2,
  "useTabs": false,
  "singleQuote": true,
  "semi": true,
  "trailingComma": "none",
  "bracketSpacing": true
}
```

**配置说明：**

| 配置项 | 作用 | 值 |
|--------|------|-----|
| `printWidth` | 单行最大字符数 | 80 |
| `tabWidth` | 缩进空格数 | 2 |
| `useTabs` | 使用 tab 还是空格 | false（用空格） |
| `singleQuote` | 使用单引号 | true |
| `semi` | 语句末尾加分号 | true |
| `trailingComma` | 尾随逗号 | none（不加） |
| `bracketSpacing` | 对象花括号内加空格 | true |

### 步骤 11：配置 package.json 的 lint 脚本

**做了什么：** 在 `package.json` 中添加 lint 命令。

**完整代码：**

在 `package.json` 的 `scripts` 字段中添加：

```json
{
  "scripts": {
    "lint": "eslint --ext .ts,.js,.jsx,.tsx --fix --quiet ./packages"
  }
}
```

**参数说明：**

| 参数 | 作用 |
|------|------|
| `--ext .ts,.js,.jsx,.tsx` | 指定检查的文件扩展名 |
| `--fix` | 自动修复可修复的问题 |
| `--quiet` | 只显示错误，不显示警告 |
| `./packages` | 检查 packages 目录下的文件 |

### 步骤 13：安装 Husky

**做了什么：** 安装 Husky 用于管理 Git hooks。

**完整代码：**

```bash
# 安装 husky
pnpm i husky -D -w
```

**注意点：**
- Husky 用于在 git commit 等操作前自动执行脚本
- 可以实现 commit 前自动运行 lint 检查

### 步骤 14：初始化 Husky 并配置 prepare 脚本

**做了什么：** 初始化 Husky 并添加 prepare 脚本。

**完整代码：**

```bash
# 初始化 husky（会创建 .husky 目录和 git hooks）
npx husky install

# 设置 prepare 脚本（在 npm install 后自动初始化 husky）
npm pkg set scripts.prepare="husky install"
```

或者手动在 `package.json` 中添加：

```json
{
  "scripts": {
    "prepare": "husky install"
  }
}
```

**注意点：**
- `husky install` 会创建 `.husky/` 目录
- `prepare` 脚本会在团队成员执行 `npm install` 或 `pnpm install` 后自动运行

### 步骤 15：添加 pre-commit hook

**做了什么：** 创建 pre-commit hook，在 commit 前自动执行 lint。

**完整代码：**

```bash
# 添加 pre-commit hook，执行 pnpm lint
npx husky add .husky/pre-commit "pnpm lint"
```

**这会创建文件 `.husky/pre-commit`：**

```bash
#!/usr/bin/env sh
. "$(dirname -- "$0")/_/husky.sh"

pnpm lint
```

**注意点：**
- 现在每次 `git commit` 前都会自动运行 `pnpm lint`
- 如果 lint 检查失败，commit 会被阻止
- 这能确保提交的代码都符合代码规范

### 步骤 16：安装 CommitLint

**做了什么：** 安装 CommitLint 用于规范 commit 信息格式。

**完整代码：**

```bash
# 安装 commitlint 核心和约定式配置
pnpm i @commitlint/cli @commitlint/config-conventional -D -w
```

**注意点：**
- `@commitlint/cli`：commitlint 命令行工具
- `@commitlint/config-conventional`：约定式提交（Conventional Commits）规范配置

---

### 步骤 17：创建 CommitLint 配置文件

**做了什么：** 创建 `.commitlintrc.js` 配置文件。

**完整代码：**

创建文件 `.commitlintrc.js`（可以放在根目录或 `scripts/` 目录下）：

```javascript
module.exports = {
  extends: ['@commitlint/config-conventional']
};
```

**注意点：**
- 这个配置使用约定式提交规范（Conventional Commits）
- 支持的 commit 类型：`feat`、`fix`、`chore`、`docs`、`perf`、`refactor`、`test` 等

**Conventional Commits 格式：**

```
<type>: <subject>
```

**常用的 type 值：**

| type | 说明 | 示例 |
|------|------|------|
| `feat` | 添加新功能 | `feat: 添加用户登录功能` |
| `fix` | 修复 Bug | `fix: 修复用户名显示错误` |
| `chore` | 构建过程或辅助工具的变动 | `chore: 更新依赖版本` |
| `docs` | 文档修改 | `docs: 更新 README` |
| `perf` | 性能优化 | `perf: 优化渲染性能` |
| `refactor` | 代码重构 | `refactor: 重构登录模块` |
| `test` | 添加测试 | `test: 添加单元测试` |

---

### 步骤 18：添加 commit-msg hook

**做了什么：** 创建 commit-msg hook，在 commit 时检查 commit 信息格式。

**完整代码：**

```bash
# 添加 commit-msg hook
npx husky add .husky/commit-msg "npx --no-install commitlint --edit $HUSKY_GIT_PARAMS"
```

或者使用新版本的命令：

```bash
npx husky add .husky/commit-msg "npx commitlint --edit $1"
```

**注意点：**
- 现在每次 `git commit` 时都会检查 commit message 是否符合规范
- 如果不符合规范，commit 会被拒绝
- 例如：`git commit -m "update code"` 会失败，需要改成 `git commit -m "chore: update code"`

---

### 步骤 19：配置 TypeScript

**做了什么：** 创建 `tsconfig.json` 配置 TypeScript 编译选项。

**完整代码 `tsconfig.json`：**

```json
{
  "compileOnSave": true,
  "compilerOptions": {
    "target": "ESNext",
    "useDefineForClassFields": true,
    "module": "ESNext",
    "lib": ["ESNext", "DOM"],
    "moduleResolution": "Node",
    "strict": true,
    "sourceMap": true,
    "resolveJsonModule": true,
    "isolatedModules": true,
    "esModuleInterop": true,
    "noEmit": true,
    "noUnusedLocals": true,
    "noUnusedParameters": true,
    "noImplicitReturns": false,
    "skipLibCheck": true,
    "baseUrl": "./packages"
  }
}
```

**关键配置说明：**

| 配置项 | 作用 |
|--------|------|
| `target: "ESNext"` | 编译目标为最新 ES 标准 |
| `module: "ESNext"` | 模块系统使用 ES 模块 |
| `lib: ["ESNext", "DOM"]` | 包含 ES 最新特性和 DOM API 类型定义 |
| `strict: true` | 开启所有严格类型检查选项 |
| `sourceMap: true` | 生成 source map 文件（用于调试） |
| `noEmit: true` | 不输出编译文件（只做类型检查） |
| `baseUrl: "./packages"` | 模块解析的基础路径 |
| `skipLibCheck: true` | 跳过声明文件的类型检查（加快编译速度） |

## 易错点汇总

| 错误 | 原因 | 解决方案 |
|------|------|----------|
| ⚠️ `Local dependency not found` | ESLint 初始化时路径不对 | 手动安装 `@typescript-eslint/parser` 等依赖 |
| ⚠️ `Issues with peer dependencies` | pnpm 严格检查 peer dependencies | 忽略警告或按提示安装 peer dependencies |
| ⚠️ ESLint 报错 `8.25.0` | ESLint 版本号提示 | 正常提示，不影响使用 |
| ⚠️ commit 时没有触发 hook | husky 未初始化 | 运行 `npx husky install` |
| ⚠️ commit 信息格式错误被拦截 | commitlint 检查不通过 | 使用规范格式：`feat: xxx` 或 `fix: xxx` |
