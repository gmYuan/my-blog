## 一、pnpm：npm 的痛点终结者

### 1.1 npm 的三大痛点

| 痛点 | 具体表现 | 影响 |
|------|---------|------|
| 😰 **慢** | 每次 `npm install` 都要重新下载 | 浪费时间 |
| 💾 **占空间** | 10 个项目 = 10 份 `lodash` | 浪费硬盘（可能几十 GB） |
| 👻 **幽灵依赖** | 能引用没声明的包（见下方） | 代码不可靠，换环境就崩 |

**幽灵依赖示例**：

> **💡 以下内容回答了你的疑问：场景 1 和场景 3 是什么意思？**

```javascript
// 你的 package.json（只声明了 A）
{
  "dependencies": { "A": "1.0.0" }
}

// npm 安装后的结构（扁平化）
node_modules/
  ├── A/        ← 你装的
  └── B/        ← A 的依赖，被 npm 提升到根目录了

// 你的代码（错误示范）
import B from 'B'; // ✅ 能跑，但这是隐患！

// 为什么是隐患？

// 场景 1：同事没装 A
你的 package.json:
{
  "dependencies": { "A": "1.0.0" }  // 只声明了 A，没声明 B
}

你的代码:
import A from 'A';  // ✅ 正确
import B from 'B';  // ❌ 错误！你没声明依赖 B

同事克隆你的项目：
npm install  // 只会安装 A（因为 package.json 里只有 A）
             // A 会带来 B，但这是"偶然的"
             
// 如果同事用 pnpm install（严格模式）：
node_modules/
  ├── A/       ← 能访问
  └── .pnpm/
      └── B/   ← 藏在这里，你访问不到！
      
// 结果：你的代码报错 "Cannot find module 'B'"
// 问题根源：你用了 B，但没在 package.json 里声明依赖 B

// 场景 2：A 升级后不再依赖 B
// 某天 A 升级到 2.0.0，内部重构不再用 B 了
{
  "dependencies": { "A": "2.0.0" }  // 升级 A
}

npm install
node_modules/
  └── A/       ← A 2.0.0 不再依赖 B
  // B 消失了！

你的代码:
import B from 'B';  // ❌ 报错！B 不存在了

// 场景 3：换到生产环境
// 开发环境（你的电脑）：
npm install  // 使用扁平化策略，B 被提升
node_modules/
  ├── A/
  └── B/     ← B 在根目录，你能用

// 生产环境（CI/CD 服务器）：
npm ci --legacy-peer-deps  // 使用不同的安装策略
node_modules/
  └── A/
      └── node_modules/
          └── B/     ← B 被嵌套在 A 里面了，不在根目录
          
你的代码:
import B from 'B';  // ❌ 报错！找不到 B（B 不在根目录）

// ✅ 正确做法：
// package.json 中明确声明 B
{
  "dependencies": {
    "A": "1.0.0",
    "B": "1.0.0"  // 明确声明！
  }
}
```

**核心问题**：你依赖了一个"没有明确声明"的包，这个包的存在是"偶然的"（恰好被 npm 提升了），而不是"必然的"（你声明了依赖）。

### 1.2 pnpm 如何解决？一图看懂

> **💡 以下内容回答了你的疑问：幽灵依赖"换环境报错"是什么意思？**

**先理解 2 个概念**：

| 概念 | 通俗解释 | 作用 |
|------|---------|------|
| **硬链接** | 同一个文件有多个名字（共享同一份数据） | 不占额外磁盘空间 |
| **符号链接** | 快捷方式（指向另一个文件） | 方便访问 |

**举例说明**：

```bash
# 硬链接示例
原文件: /store/lodash.js (100MB)
硬链接: /project1/node_modules/lodash.js → 指向同一份数据
硬链接: /project2/node_modules/lodash.js → 指向同一份数据
# 磁盘占用：100MB（而不是 300MB）

# 符号链接示例
符号链接: /project1/node_modules/lodash → /store/lodash
# 作用：让你能用 import 'lodash'，实际访问的是 /store/lodash
```

**pnpm 的三层结构**：

```bash
【第 1 层：全局 Store】~/.pnpm-store/
└── v3/files/
    └── 00/1a2b3c...  ← lodash@4.17.21 的实际内容（全局唯一）
           ↑
           | 硬链接（共享磁盘）
           |
【第 2 层：虚拟存储】项目/node_modules/.pnpm/
└── lodash@4.17.21/
    └── node_modules/
        └── lodash/  ← 硬链接到全局 Store
              ↑
              | 符号链接（方便访问）
              |
【第 3 层：你能看到的】项目/node_modules/
└── lodash/  ← 符号链接到第 2 层

# 你的代码
import lodash from 'lodash';  // ✅ 实际访问路径：
// 第 3 层 lodash → 第 2 层 .pnpm/lodash@4.17.21 → 第 1 层全局 Store
```

**为什么要三层？**

| 层级 | 作用 | 解决的问题 |
|------|------|----------|
| 第 1 层（全局 Store） | 全局只存一份 | 节省磁盘空间 |
| 第 2 层（.pnpm） | 隔离间接依赖 | 防止幽灵依赖 |
| 第 3 层（node_modules） | 只暴露直接依赖 | 让你的代码能 import |

**对比 npm**：

```bash
npm（扁平化，有问题）:
node_modules/
  ├── A/           ← 你装的
  ├── B/           ← A 的依赖（幽灵依赖！）
  └── lodash/      ← 每个项目都复制一份

pnpm（三层结构，安全）:
node_modules/
  ├── A/           ← 符号链接（你能访问）
  ├── .pnpm/
  │   ├── A@1.0.0/
  │   │   └── node_modules/
  │   │       ├── A/       ← 硬链接到全局
  │   │       └── B/       ← A 的依赖（藏在这里）
  │   └── lodash@4.17.21/  ← 硬链接到全局（只存一份）
  └── lodash/      ← 符号链接（你能访问）

# 你只能访问 A 和 lodash（你装的）
# 你不能访问 B（A 的依赖，被隔离了）
```

### 1.3 pnpm vs npm 对比表（面试必背）

| 对比维度 | npm | pnpm | pnpm 优势原因 |
|---------|-----|------|-------------|
| **速度** | 慢 | 快 2-3 倍 | 不重复下载，并行更激进 |
| **磁盘** | 10 个项目 10 份 | 10 个项目 1 份 | 硬链接共享 |
| **安全性** | 有幽灵依赖 | 无幽灵依赖 | 严格的嵌套结构 |
| **兼容性** | 100% | 99%+ | 极少数包依赖扁平化 |

### 1.4 工作常用命令（只需记住这些）

| 场景 | npm 命令 | pnpm 命令 | 说明 |
|------|---------|----------|------|
| 初始化 | `npm init` | `pnpm init` | 创建 package.json |
| 安装全部 | `npm install` | `pnpm install` | 简写：`pnpm i` |

### 1.5 必知的 2 个配置

**配置 1：`.npmrc`（项目根目录创建）**

**`.npmrc` 是什么？**
- npm/pnpm 的配置文件（类似 Git 的 `.gitconfig`）
- 控制包管理器的行为（镜像源、严格模式等）

> **💡 以下内容回答了你的疑问：.npmrc 干嘛用的？peer 依赖是什么？**

**核心配置解释**：

```bash
# .npmrc 文件内容

# 1. peer 依赖相关
strict-peer-dependencies=false
# 解释：peer 依赖缺失时，只警告不报错（默认 pnpm 会报错）

auto-install-peers=true
# 解释：自动安装缺失的 peer 依赖（省得手动装）

# 2. 镜像源（国内加速）
registry=https://registry.npmmirror.com
# 解释：从淘宝镜像下载包，比官方源快
```

**什么是 Peer Dependencies（对等依赖）？**

| 依赖类型 | 含义 | 示例 |
|---------|------|-----|
| **dependencies** | 你的代码直接用到的包 | `lodash` |
| **devDependencies** | 开发时用的工具 | `vite`, `typescript` |
| **peerDependencies** | 你依赖的"宿主环境"的包 | React 组件库依赖 `react` |

**Peer 依赖的场景**：

> **💡 以下内容回答了你的疑问：为什么 peer 不会重复打包 React？**

```json
// 你在开发一个 React 组件库 my-ui-library
{
  "name": "my-ui-library",
  "peerDependencies": {
    "react": ">=18.0.0"  // 要求使用者必须装 react 18+
  }
}

// 为什么用 peer？
// 1. 避免多个 React 版本冲突（你用 18，用户用 17，会崩溃）
// 2. 让用户自己决定用哪个版本
// 3. 不会重复打包 React（减小体积）← 重点解释

// 用户安装你的组件库：
npm install my-ui-library
# npm 警告：需要 react >= 18.0.0，但你没装
# pnpm 报错：必须先装 react（strict-peer-dependencies=true）
```

**为什么不会重复打包 React？详细解释**：

**场景对比**：

```javascript
// ❌ 错误方式：把 react 放在 dependencies
{
  "name": "my-ui-library",
  "dependencies": {
    "react": "18.2.0"  // 错误！会重复打包
  }
}

// 用户项目的 package.json
{
  "dependencies": {
    "react": "18.3.0",        // 用户装的 react
    "my-ui-library": "1.0.0"  // 你的组件库
  }
}

// npm install 后的结构
node_modules/
  ├── react/                    ← 用户的 react 18.3.0
  └── my-ui-library/
      └── node_modules/
          └── react/            ← 你的组件库带的 react 18.2.0（重复了！）

// 问题：
// 1. 磁盘占用：React 被装了 2 次（18.2.0 和 18.3.0）
// 2. 打包体积：最终打包后，React 代码可能被包含 2 次（几百 KB）
// 3. 运行时错误：2 个 React 实例会导致 Hooks 报错
import React from 'react';  // 用户代码用的是 18.3.0
import { Button } from 'my-ui-library';  // Button 内部用的是 18.2.0
// 结果：React 检测到多个实例，报错 "Invalid hook call"

// ✅ 正确方式：把 react 放在 peerDependencies
{
  "name": "my-ui-library",
  "peerDependencies": {
    "react": ">=18.0.0"  // 正确！不打包 React
  }
}

// npm install 后的结构
node_modules/
  ├── react/                    ← 只有 1 份 React（用户的 18.3.0）
  └── my-ui-library/
      // 没有 node_modules/react/，组件库共用用户的 React

// 好处：
// 1. 磁盘占用：React 只装 1 次
// 2. 打包体积：React 只打包 1 次（减小几百 KB）
// 3. 运行时正确：只有 1 个 React 实例，不会报错

// 组件库的代码（打包时的处理）
// webpack.config.js 或 rollup.config.js
export default {
  external: ['react', 'react-dom'],  // 标记为外部依赖，不打包进去
  // ...
}

// 最终效果：
// my-ui-library.js 不包含 React 代码（只有你的组件代码）
// 运行时从用户项目的 node_modules/react 引用
```

**通俗类比**：

```
不用 peer（错误）：
你开发一个"房子装修工具包"，里面自带了"水泥、砖头"
用户买了你的工具包，自己也买了"水泥、砖头"
结果：家里有 2 份水泥、2 份砖头（浪费空间）

用 peer（正确）：
你开发一个"房子装修工具包"，说明书写着"需要用户自备水泥、砖头"
用户买了你的工具包，自己准备"水泥、砖头"
结果：家里只有 1 份水泥、1 份砖头（节省空间，共用）
```

**配置 2：`package.json` 强制使用 pnpm**
```json
{
  "engines": {
    "pnpm": ">=8.0.0"  // 声明需要 pnpm 8+
  },
  "scripts": {
    "preinstall": "npx only-allow pnpm"  // 禁止 npm/yarn
  }
}
```

### 1.6 如何声明 Peer Dependencies？

> **💡 以下内容回答了你的新疑问：怎么把依赖声明为 peerDependencies？**

**方式 1：手动编辑 `package.json`（推荐）**

```json
{
  "name": "my-ui-library",
  "version": "1.0.0",
  
  // ❌ 错误：放在 dependencies（会打包进去）
  "dependencies": {
    "react": "^18.0.0"
  },
  
  // ❌ 错误：放在 devDependencies（开发用，但不是 peer）
  "devDependencies": {
    "react": "^18.0.0"
  },
  
  // ✅ 正确：放在 peerDependencies（不打包，要求用户提供）
  "peerDependencies": {
    "react": ">=18.0.0",
    "react-dom": ">=18.0.0"
  },
  
  // 可选：为了开发时能用 react，同时也放在 devDependencies
  "devDependencies": {
    "react": "^18.0.0",      // 开发时用（不会打包）
    "react-dom": "^18.0.0"
  },
  "peerDependencies": {
    "react": ">=18.0.0",     // 要求用户提供（不会打包）
    "react-dom": ">=18.0.0"
  }
}
```

**方式 2：使用命令（npm 7+）**

```bash
# npm 没有直接命令添加 peer 依赖，需要手动编辑 package.json

# 但可以用工具自动化
npx add-peerDependency react@18.0.0
```

**常见场景对比**：

| 依赖类型 | 什么时候用 | 会不会打包进去 | 示例 |
|---------|-----------|-------------|-----|
| **dependencies** | 你的代码直接用的包 | ✅ 会 | `lodash`, `axios` |
| **devDependencies** | 开发/构建时用的工具 | ❌ 不会 | `vite`, `typescript`, `eslint` |
| **peerDependencies** | 你依赖的"宿主环境"的包 | ❌ 不会 | React 组件库依赖 `react` |

**实际示例：React 组件库的 package.json**

```json
{
  "name": "@my/ui-components",
  "version": "1.0.0",
  
  "dependencies": {
    // 你组件库内部用到的工具（会打包）
    "classnames": "^2.3.2",
    "date-fns": "^2.30.0"
  },
  
  "devDependencies": {
    // 开发和构建工具（不打包）
    "react": "^18.2.0",        // 开发时需要，但不打包
    "react-dom": "^18.2.0",
    "typescript": "^5.0.0",
    "vite": "^4.0.0",
    "@types/react": "^18.0.0"
  },
  
  "peerDependencies": {
    // 要求用户必须提供（不打包）
    "react": ">=18.0.0",
    "react-dom": ">=18.0.0"
  }
}
```

**构建配置（确保不打包 peer 依赖）**

```javascript
// vite.config.js 或 rollup.config.js
export default {
  build: {
    rollupOptions: {
      // 标记为外部依赖，不打包进最终文件
      external: ['react', 'react-dom'],
      output: {
        globals: {
          react: 'React',          // 全局变量名
          'react-dom': 'ReactDOM'
        }
      }
    }
  }
}
```

**验证是否正确配置**：

```bash
# 1. 构建你的组件库
npm run build

# 2. 查看打包后的文件大小
ls -lh dist/

# 如果配置正确：
# - 文件应该很小（几十 KB）
# - 不应该包含 React 代码（React 本身几百 KB）

# 3. 查看打包后的代码
cat dist/my-ui-library.js
# 应该看到类似：
# import React from 'react';  // 引用外部的 react，而不是打包进来
```

**总结对比**：

```javascript
// 场景 1：工具库（如 lodash-plus）
{
  "dependencies": {
    "lodash": "^4.17.21"  // ✅ 正确：打包进去
  }
}

// 场景 2：React 组件库
{
  "dependencies": {
    "classnames": "^2.3.2"  // ✅ 正确：工具库，打包进去
  },
  "peerDependencies": {
    "react": ">=18.0.0"     // ✅ 正确：不打包，要求用户提供
  },
  "devDependencies": {
    "react": "^18.2.0"      // ✅ 正确：开发时用
  }
}

// 场景 3：Vue 插件
{
  "peerDependencies": {
    "vue": ">=3.0.0"        // ✅ 正确：不打包 Vue
  }
}
```

## 二、Monorepo：多项目管理的银弹

### 2.1 什么问题让 Monorepo 诞生？

**场景**：你在开发一个产品，包含：
- 🌐 官网（Web）
- 📱 移动端 App
- 📦 共享的 UI 组件库
- 🔧 共享的工具函数

**传统方案（Polyrepo）的痛点**：

| 痛点 | 具体表现 |
|------|---------|
| 😰 **代码复用难** | 改个组件要发 npm、等 CI、再更新版本，太慢 |
| 🔀 **版本地狱** | Web 用 UI@1.0，App 用 UI@2.0，互相冲突 |
| 🔧 **重构噩梦** | 重命名一个函数，要改 3 个仓库，容易漏 |
| ⚙️ **工具不统一** | 每个仓库都配置一遍 ESLint、Prettier |

**Monorepo 的解决方案**：

```
一个仓库 = 多个项目

my-product/
├── packages/
│   ├── web/         ← 官网项目
│   ├── mobile/      ← 移动端项目
│   ├── ui/          ← UI 组件库
│   └── utils/       ← 工具函数库
└── package.json     ← 统一配置
```

### 2.2 Monorepo 的优缺点（面试必问）

| 优点 | 具体好处 | 场景举例 |
|------|---------|---------|
| ✅ **代码复用简单** | 直接 `import`，不用发 npm | 改组件立即生效 |
| ✅ **版本统一** | 所有项目用相同版本的依赖 | 不会出现版本冲突 |
| ✅ **原子提交** | 一次 commit 改多个包，保证一致性 | 重构 API 时同步更新调用方 |
| ✅ **重构方便** | 重命名自动更新所有引用 | IDE 全局搜索替换 |
| ✅ **工具统一** | 一套 lint/test/build 配置 | 新人只需学一次 |

| 缺点 | 具体问题 | 缓解方案 |
|------|---------|---------|
| ❌ **仓库大** | 代码多、历史长 | Git Shallow Clone（只克隆最近几次提交） |
| ❌ **权限难控制** | 难以限制某些人只能改某个包 | CODEOWNERS 文件（GitHub 功能） |
| ❌ **CI/CD 慢** | 改一行代码，跑全部测试太慢 | 只测试变更的包（后面讲） |

> **💡 以下内容回答了你的疑问：Git Shallow Clone 和 CODEOWNERS 是什么？**

**术语解释**：
- **Git Shallow Clone**：`git clone --depth=1`，只下载最新的提交历史，不下载完整历史（仓库小 10 倍+）
- **CODEOWNERS**：GitHub 的文件，定义谁可以审核/修改哪些目录（如 `packages/ui/* @ui-team`）

### 2.3 Monorepo 适用场景判断表

| 你的情况 | 建议 |
|---------|-----|
| ✅ 多个项目共享大量代码（如 React 生态） | 强烈推荐 Monorepo |
| ✅ 需要确保依赖版本一致 | 推荐 Monorepo |
| ✅ 需要频繁跨项目重构 | 推荐 Monorepo |
| ❌ 项目完全独立，无代码共享 | 不需要 Monorepo |
| ❌ 团队技术栈完全不同（Java + Python + Go） | 不适合 Monorepo |

## 三、pnpm Workspace：Monorepo 的实现工具

### 3.0 pnpm 为什么能支持 Monorepo？

> **💡 以下内容回答了你的疑问：pnpm 为什么能够支持 Monorepo？**

**核心能力对比**：

| 能力 | npm/yarn | pnpm | 为什么 pnpm 更适合 |
|------|----------|------|------------------|
| **工作区（Workspace）支持** | ✅ 有 | ✅ 有 | 三者都支持 |
| **本地包热更新** | ✅ 支持 | ✅ 支持 | 通过符号链接实现 |
| **速度** | 慢 | 快 | 硬链接 + 不重复下载 |
| **磁盘占用** | 大 | 小 | 全局共享依赖 |
| **依赖隔离** | ❌ 弱 | ✅ 强 | 严格的三层结构 |

**pnpm 的独特优势**：

1. **速度快**：Monorepo 依赖多，pnpm 的硬链接优势明显
2. **严格性**：防止子包之间的幽灵依赖（npm 容易出现）
3. **内置支持**：不需要额外工具（npm 需要配合 Lerna）

**示例场景**：

```bash
# Monorepo 有 10 个子包，每个都依赖 React

npm/yarn:
- 可能重复安装 10 次 React（扁平化不完美时）
- 磁盘占用：React x 10 = 几百 MB

pnpm:
- React 只下载一次，10 个包通过硬链接共享
- 磁盘占用：React x 1 = 几十 MB
```

### 3.1 核心概念：Workspace（工作区）

**Workspace 是什么？** 
让一个仓库里的多个子项目能互相引用，就像在同一个项目里一样。

**解决什么问题？**
- ❌ 传统：改了组件库，要 `npm publish` → `npm update` → 重启项目
- ✅ Workspace：改了组件库，项目里立即生效（热更新）

### 3.2 3 步配置 Workspace

### 3.2 3 步配置 Workspace

**Step 1：创建 `pnpm-workspace.yaml`（根目录）**

```yaml
packages:
  - 'packages/*'  # packages 下的所有文件夹都是独立的包
```

**Step 2：创建目录结构**

```bash
my-monorepo/
├── pnpm-workspace.yaml  ← 告诉 pnpm 这是 Monorepo
├── package.json          ← 根配置
├── packages/
│   ├── ui/               ← 子包 1（组件库）
│   │   ├── package.json
│   │   └── src/
│   ├── utils/            ← 子包 2（工具库）
│   │   ├── package.json
│   │   └── src/
│   └── app/              ← 子包 3（应用）
│       ├── package.json
│       └── src/
└── node_modules/
```

**Step 3：配置子包的 `package.json`**

```json
// packages/ui/package.json
{
  "name": "@my-project/ui",     // ⚠️ 必须有名字
  "version": "1.0.0"
}

// packages/app/package.json
{
  "name": "@my-project/app",
  "dependencies": {
    "@my-project/ui": "workspace:*"  // ⚠️ 关键：引用本地包
  }
}
```

**`workspace:*` 的含义**：
- 告诉 pnpm：这是本地包，不要从 npm 下载
- 修改 `ui` 包的代码，`app` 包立即生效（热更新）
- 发布时自动替换为真实版本号

### 3.3 Workspace 常用命令（工作核心）

| 场景 | 命令 | 说明 |
|------|------|-----|
| 安装所有包的依赖 | `pnpm install` | 在根目录执行一次即可 |
| 给根目录添加依赖 | `pnpm add typescript -w` | `-w` = workspace root |
| 给特定包添加依赖 | `pnpm add react --filter app` | `--filter` 指定包名 |
| 给特定包添加本地包 | `pnpm add @my/ui --filter app -w` | 建立内部依赖 |
| 运行特定包的脚本 | `pnpm --filter app dev` | 只运行 app 的 dev 命令 |
| 运行所有包的脚本 | `pnpm -r run build` | `-r` = recursive（递归） |
| 只运行变更包的脚本 | `pnpm --filter ...[origin/main] test` | 只测试有改动的包 |

**记忆技巧**：
- `-w`：操作根目录
- `--filter <包名>`：操作指定包
- `-r`：操作所有包

## 四、实战案例：10 分钟搭建 Mini React 项目

### 4.1 项目目标

模拟 React 源码的 Monorepo 结构：
- `react`：核心包（提供 API）
- `react-dom`：渲染器（操作 DOM）
- `shared`：共享工具

### 4.2 操作流程（跟着敲）

```bash
# 1. 创建项目
mkdir mini-react && cd mini-react
pnpm init

# 2. 配置 Workspace
cat > pnpm-workspace.yaml << EOF
packages:
  - 'packages/*'
EOF

# 3. 创建子包目录
mkdir -p packages/{react,react-dom,shared}

# 4. 初始化子包
cd packages/react && pnpm init
cd ../react-dom && pnpm init
cd ../shared && pnpm init
cd ../..

# 5. 建立依赖关系（重点理解）
# 命令格式：pnpm add <包名> --filter <给谁装> -w

> **💡 以下内容回答了你的疑问：这几个 pnpm add 命令是什么意思？**

pnpm add @mini-react/shared --filter react -w
# 解释：给 react 包添加 shared 包的依赖
# --filter react: 指定在 react 包中操作

# 解释：给 react-dom 包添加 react 包的依赖

pnpm add @mini-react/shared --filter react-dom -w
# 解释：给 react-dom 包添加 shared 包的依赖

# 6. 安装全部依赖

```

**命令详解**：

| 命令部分 | 含义 | 作用 |
|---------|------|-----|
| `pnpm add @mini-react/shared` | 安装 shared 包 | 指定要添加的依赖 |
| `--filter react` | 过滤器：只对 react 包操作 | 指定目标包 |
| `-w` | workspace（工作区） | 从本地查找而非 npm |

**依赖关系图**：

```
shared（工具库）
  ↑
  |
  ├─── react（核心）← 依赖 shared
  |      ↑
  |      |
  └─── react-dom（渲染器）← 依赖 react 和 shared
```

### 4.3 配置文件示例

**根目录 `package.json`**：
```json
{
  "name": "mini-react-monorepo",
  "private": true,
  "scripts": {
    "build": "pnpm -r run build",

  }
}
```

**子包 `packages/react-dom/package.json`**：

{
  "name": "@mini-react/react-dom",

### Q1: pnpm 为什么比 npm 快？

**回答思路**（3 个点）：
1. **硬链接共享**：同一个包全局只下载一次，其他项目通过硬链接复用
2. **并行下载**：pnpm 的并行策略比 npm 更激进
3. **智能缓存**：基于内容寻址（content-addressable），精确复用

**面试加分**：可以补充"npm 的扁平化算法很耗时，pnpm 不需要"

### Q2: 什么是幽灵依赖？pnpm 如何解决？

**回答思路**：

**问题**（what）：
- npm 会把依赖的依赖提升到 `node_modules` 根目录（扁平化）
- 你的代码可以引用没有声明的包（幽灵依赖）
- 换个环境可能就报错了

**解决**（how）：
- pnpm 不使用扁平化结构
- `node_modules` 里只有你直接安装的包的符号链接
- 间接依赖藏在 `.pnpm` 目录里，无法直接访问

**示例代码**（可以在面试时画图）：
```
npm 结构（有问题）:          pnpm 结构（安全）:
node_modules/               node_modules/
├── A/                      ├── A/  ← 符号链接
└── B/  ← 幽灵依赖！         └── .pnpm/
                                └── A/
                                    └── node_modules/
                                        └── B/  ← 藏起来了
```

### Q5: 如何只构建变更的包？

**回答思路**（命令 + 原理）：

**命令**：
```bash
# 只构建相对于 main 分支有变更的包
pnpm --filter ...[origin/main] run build
```

**原理**：
- `...` 表示包含该包及其所有依赖
- `[origin/main]` 表示相对于 origin/main 分支的变更
- pnpm 会自动对比 Git 历史，找出变更的包

**面试加分**：可以提到"大型 Monorepo 常用 Turborepo 或 Nx 做增量构建"

> **💡 以下内容回答了你的疑问：Turborepo、Nx 增量构建是什么意思？**

**增量构建工具解释**：

| 工具 | 作用 | 适用场景 |
|------|------|---------|
| **Turborepo** | 智能缓存构建结果，只重新构建变更的包 | 大型 Monorepo（几十个包） |
| **Nx** | 依赖图分析 + 增量构建 + 任务编排 | 超大型 Monorepo（上百个包） |

**举例**：
```bash
# 场景：100 个包的 Monorepo，你只改了 1 个包

# 普通方式（慢）：
pnpm -r run build  # 构建全部 100 个包，耗时 10 分钟

# Turborepo（快）：
turbo run build    # 只构建变更的包 + 依赖它的包（2 个包），耗时 1 分钟
# 原理：Turborepo 缓存了之前的构建结果，没变的包直接复用
```

### Q6: pnpm 的 peer dependencies 如何处理？

### Q7: dependencies 和 devDependencies 的本质区别是什么？

> **💡 以下内容回答了你的新疑问：dependencies 和 devDependencies 到底有什么区别？什么时候用哪个？**

**回答思路（3 个维度）**：

#### 1. 本质区别（最重要）

| 维度 | dependencies | devDependencies |
|------|--------------|-----------------|
| **安装时机** | `npm install` 时总是安装 | `npm install --production` 时不安装 |
| **会不会被打包** | 取决于你的代码是否 import | 不影响（构建工具决定） |
| **会不会被依赖者安装** | ✅ 会（传递依赖） | ❌ 不会 |

**核心本质**：

```bash
# 场景：用户安装你的包

# 你的包 my-package 的 package.json
{
  "name": "my-package",
  "dependencies": {
    "lodash": "^4.17.21"  // 生产依赖
  },
  "devDependencies": {
    "typescript": "^5.0.0"  // 开发依赖
  }
}

# 用户安装你的包
npm install my-package

# 结果：
node_modules/
  ├── my-package/
  ├── lodash/          ← ✅ lodash 被自动安装（传递依赖）
  └── typescript/      ← ❌ typescript 不会被安装

# 本质区别：
# - dependencies: 你的包运行时需要的依赖（用户也要装）
# - devDependencies: 你的包开发时需要的依赖（用户不需要装）
```

#### 2. 什么放在 dependencies？什么放在 devDependencies？

**判断标准**：问自己"**用户运行我的代码时，需要这个包吗？**"

| 判断条件 | 放在哪里 | 示例 |
|---------|---------|-----|
| 你的代码 `import` 了这个包 | dependencies | `lodash`, `axios`, `react` |
| 只在构建/测试/开发时用 | devDependencies | `typescript`, `vite`, `jest` |
| 类型定义文件（`@types/*`） | devDependencies | `@types/node`, `@types/react` |

**详细示例**：

```javascript
// 你的库代码 src/index.js

// 场景 1: 你的代码直接用到了 lodash
import _ from 'lodash';  // ← lodash 应该放在 dependencies

export function myFunction(data) {
  return _.uniq(data);  // 运行时需要 lodash
}

// 场景 2: 你用 TypeScript 写的，但发布的是编译后的 JS
// src/index.ts
export function add(a: number, b: number) {
  return a + b;
}

// 编译后的 dist/index.js
export function add(a, b) {  // 没有类型，不需要 typescript
  return a + b;
}
// ← typescript 应该放在 devDependencies（用户运行时不需要）

// 场景 3: 测试代码
// tests/index.test.js
import { myFunction } from '../src';
import { expect, test } from 'vitest';  // ← vitest 放在 devDependencies

test('myFunction works', () => {
  expect(myFunction([1, 1, 2])).toEqual([1, 2]);
});
```

#### 3. 常见包分类速查表

| 包名 | 放在哪里 | 原因 |
|------|---------|-----|
| `lodash` | dependencies | 运行时需要的工具库 |
| `axios` | dependencies | 运行时需要的 HTTP 客户端 |
| `react` | dependencies（或 peerDependencies） | 运行时需要的框架 |
| `express` | dependencies | 运行时需要的服务器框架 |
| `typescript` | devDependencies | 只在开发时编译用 |
| `vite` | devDependencies | 只在开发/构建时用 |
| `eslint` | devDependencies | 只在开发时检查代码 |
| `prettier` | devDependencies | 只在开发时格式化代码 |
| `jest` / `vitest` | devDependencies | 只在开发时测试 |
| `@types/node` | devDependencies | 只在开发时提供类型 |
| `rollup` / `webpack` | devDependencies | 只在构建时打包 |

#### 4. 特殊场景详解

**场景 A：库项目（会被别人安装）**

```json
// 你在开发一个工具库 my-utils
{
  "name": "my-utils",
  "dependencies": {
    "lodash": "^4.17.21"     // ✅ 代码里用了，放这里
  },
  "devDependencies": {
    "typescript": "^5.0.0",  // ✅ 只是编译工具，放这里
    "rollup": "^3.0.0",      // ✅ 只是打包工具，放这里
    "jest": "^29.0.0"        // ✅ 只是测试工具，放这里
  }
}

// 用户安装你的库
npm install my-utils
// 只会安装 lodash，不会安装 typescript/rollup/jest
```

**场景 B：应用项目（最终部署的项目，如网站）**

```json
// 你在开发一个 Vue 应用
{
  "name": "my-vue-app",
  "dependencies": {
    "vue": "^3.3.0",         // ✅ 运行时需要
    "vue-router": "^4.0.0",  // ✅ 运行时需要
    "axios": "^1.0.0"        // ✅ 运行时需要
  },
  "devDependencies": {
    "vite": "^4.0.0",        // ✅ 只在开发/构建时用
    "@vitejs/plugin-vue": "^4.0.0",  // ✅ 只在构建时用
    "eslint": "^8.0.0"       // ✅ 只在开发时用
  }
}

// 部署到生产环境
npm install --production
// 只安装 dependencies（vue, vue-router, axios）
// 不安装 devDependencies（vite, eslint 等）
// 因为生产环境只需要运行打包后的代码，不需要构建工具
```

**场景 C：全栈项目（Node.js 后端）**

```json
// 你在开发一个 Express 后端
{
  "name": "my-backend",
  "dependencies": {
    "express": "^4.18.0",    // ✅ 运行时需要
    "mongoose": "^7.0.0",    // ✅ 运行时需要
    "jsonwebtoken": "^9.0.0" // ✅ 运行时需要
  },
  "devDependencies": {
    "nodemon": "^2.0.0",     // ✅ 只在开发时自动重启
    "@types/express": "^4.17.0",  // ✅ 只在开发时提供类型
    "ts-node": "^10.0.0"     // ✅ 只在开发时运行 TS
  }
}

// 生产环境部署
npm install --production
// 只安装 express, mongoose, jsonwebtoken
// 不安装 nodemon, @types/express, ts-node
```

#### 5. 常见误区

| 误区 | 错误示例 | 正确做法 |
|------|---------|---------|
| ❌ 把类型定义放在 dependencies | `"@types/node": "^18.0.0"` 在 dependencies | 应该放在 devDependencies |
| ❌ 把运行时依赖放在 devDependencies | `"axios": "^1.0.0"` 在 devDependencies | 应该放在 dependencies |
| ❌ 库项目把 React 放在 dependencies | `"react": "^18.0.0"` 在 dependencies | 应该放在 peerDependencies |

#### 6. 验证是否放对了

**方法 1：检查打包后的代码**

```bash
# 构建你的项目
npm run build

# 查看打包后的文件
cat dist/index.js

# 如果看到某个包的代码被打包进去，那个包应该在 dependencies
# 如果没有被打包进去，可以放在 devDependencies
```

**方法 2：模拟用户安装**

```bash
# 打包你的库
npm pack
# 生成 my-package-1.0.0.tgz

# 在另一个目录测试
mkdir test-install && cd test-install
npm init -y
npm install ../my-package-1.0.0.tgz

# 查看安装了哪些依赖
ls node_modules/
# dependencies 里的包会被安装
# devDependencies 里的包不会被安装
```

#### 7. 面试答题模板

**简洁版**（1 分钟）：
- **dependencies**：运行时需要的依赖，用户安装你的包时会自动安装
- **devDependencies**：开发时需要的依赖，用户安装你的包时不会安装
- **判断标准**：问自己"用户运行我的代码时，需要这个包吗？"

**详细版**（2-3 分钟）：
1. **本质区别**：dependencies 有传递性（用户会安装），devDependencies 没有
2. **使用场景**：dependencies 放运行时依赖（lodash、axios），devDependencies 放工具（typescript、vite）
3. **生产部署**：`npm install --production` 只安装 dependencies
4. **举例**：TypeScript 编译后不需要，放 devDependencies；Lodash 运行时需要，放 dependencies

## 六、常见问题速查

> **💡 以下内容回答了你的疑问：pnpm Store、@changesets/cli、Lerna/Nx/Turborepo 是什么？**

| 问题 | 解决方案 | 详细说明 |
|------|---------|---------|
| **pnpm Store 在哪？** | `pnpm store path` | Store 是全局存储目录，所有项目共享依赖的地方 |
| **如何清理缓存？** | `pnpm store prune` | 删除未被引用的包（安全） |
| **Windows 符号链接权限问题？** | 开启开发者模式或以管理员运行 | Windows 默认不允许普通用户创建符号链接 |
| **如何禁止团队使用 npm/yarn？** | `package.json` 加 `"preinstall": "npx only-allow pnpm"` | 强制团队统一使用 pnpm |
| **Monorepo 如何发布包？** | 用 `@changesets/cli`（推荐） | 见下方详解 |

**pnpm Store 详解**：

```bash
# 查看 Store 位置
pnpm store path
# 输出示例：/Users/username/.pnpm-store/v3

# Store 目录结构
~/.pnpm-store/
└── v3/
    └── files/
        ├── 00/
        │   └── 1a2b3c...  ← lodash@4.17.21 的实际文件
        ├── 01/
        │   └── 4d5e6f...  ← react@18.2.0 的实际文件
        └── ...

# 作用：
# 1. 所有项目共享这一份存储（节省空间）
# 2. 通过内容寻址（哈希值）确保文件唯一性
# 3. 项目中的 node_modules 通过硬链接指向这里
```

**@changesets/cli 详解**：

**是什么？**
- Monorepo 的版本管理和发布工具
- 解决"改了多个包，怎么统一发版本号和 changelog"的问题

**使用流程**：

```bash
# 1. 安装
pnpm add -Dw @changesets/cli

# 2. 初始化
pnpm changeset init
# 生成 .changeset/ 目录

# 3. 记录变更（开发时）
pnpm changeset
# 交互式选择：改了哪些包？版本升级类型（major/minor/patch）？

# 4. 生成版本号和 changelog
pnpm changeset version
# 自动更新 package.json 的 version 和 CHANGELOG.md

# 5. 发布到 npm
pnpm changeset publish
```

**为什么需要它？**

| 场景 | 不用 changesets | 用 changesets |
|------|----------------|--------------|
| 改了 5 个包 | 手动改 5 个 package.json 版本号 | 自动更新 |
| 写 changelog | 手动写 | 自动生成 |
| 决定版本号 | 容易搞错（patch 还是 minor？） | 交互式引导 |

**Monorepo 工具对比**：

| 工具 | 定位 | 核心功能 | 适用场景 |

| **pnpm** | 包管理器 + Workspace | 依赖管理、本地包引用 | 中小型 Monorepo（入门首选） |

**选择建议**：

```

│   └── 推荐：pnpm Workspace（本文讲的）
├── 中型（10-50 个包）
│   └── 推荐：pnpm + Turborepo
└── 大型（50+ 个包）
    └── 推荐：pnpm + Nx

发布需求：
└── 需要版本管理
    └── 加上 @changesets/cli
```
