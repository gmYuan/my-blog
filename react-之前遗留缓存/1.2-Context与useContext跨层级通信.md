# Context与useContext跨层级通信

## 1. 层级关系图

```
        祖先组件 (Provider)
        提供 value
            |
      ┌─────┼─────┐
      |     |     |
   父组件  父组件  父组件
      |     |     |
   子组件  子组件  子组件 (Consumer)
                  消费 value
            
跳过中间层，直接传递数据
```

## 2. 各自的职责

### createContext
- **职责**: 创建Context对象，定义默认值
- **输入**: defaultValue（可选）
- **输出**: Context对象（含Provider和Consumer）
- **类比**: 建立一个广播电台，设置默认频道

### Provider
- **职责**: 在组件树上层提供数据
- **输入**: value（要共享的数据）
- **输出**: 子树中所有Consumer都能访问
- **类比**: 电台发射信号，覆盖整个区域

### useContext
- **职责**: 函数组件中消费Context数据
- **输入**: Context对象
- **输出**: 当前Context的value值
- **类比**: 收音机接收电台信号

## 3. 调用链路

**场景：主题切换（祖先提供主题，后代使用）**

```
1. 顶层创建 ThemeContext = createContext('light')
   ↓
2. App 组件用 <ThemeContext.Provider value={theme}>
   ↓
3. 中间层组件（不需要theme）直接透传 {children}
   ↓
4. 深层组件调用 const theme = useContext(ThemeContext)
   ↓
5. 直接获取到 theme 值，不需要层层传props
   ↓
6. Provider 的 value 变化
   ↓
7. 所有使用 useContext(ThemeContext) 的组件自动重新渲染
```

## 4. 为什么需要Context？

**原因1**: 避免props drilling - 跨多层传递props会让中间组件代码冗余
- 例子：主题色要传给20层深的按钮，不用Context就要在19个中间组件都加theme props

**原因2**: 半全局数据共享 - 适合"不必用Redux但需要跨层共享"的场景
- 例子：用户信息、语言设置、UI主题这类数据，全局只有一份但不需要复杂状态管理

## 5. 类比锚定

想象公司内部广播系统：
- **createContext** 是安装广播设备，设置默认音量
- **Provider** 是总部通过广播发布通知（value），所有楼层都能听到
- **useContext** 是各部门收听广播，直接获取通知内容
- 中间楼层不需要通知的就不听，但信号依然覆盖整栋楼

## 6. 代码示例

### 基础用法

```jsx
import { createContext, useContext, useState } from 'react';

// 1. 创建Context
const ThemeContext = createContext('light');

// 2. 顶层组件提供值
function App() {
  const [theme, setTheme] = useState('light');
  
  return (
    <ThemeContext.Provider value={theme}>
      <Toolbar />
      <button onClick={() => setTheme(theme === 'light' ? 'dark' : 'light')}>
        切换主题
      </button>
    </ThemeContext.Provider>
  );
}

// 中间组件（不关心theme，直接透传children）
function Toolbar() {
  return (
    <div>
      <ThemedButton />
    </div>
  );
}

// 3. 深层组件消费值
function ThemedButton() {
  const theme = useContext(ThemeContext);  // 直接获取
  
  return (
    <button style={{ background: theme === 'light' ? '#fff' : '#333' }}>
      当前主题: {theme}
    </button>
  );
}
```

### 传递多个值

```jsx
const UserContext = createContext(null);

function App() {
  const [user, setUser] = useState({ name: '张三', role: 'admin' });
  
  // 传递对象，包含数据和更新函数
  const value = {
    user,
    updateUser: setUser
  };
  
  return (
    <UserContext.Provider value={value}>
      <Profile />
    </UserContext.Provider>
  );
}

function Profile() {
  const { user, updateUser } = useContext(UserContext);
  
  return (
    <div>
      <p>用户: {user.name}</p>
      <button onClick={() => updateUser({ ...user, name: '李四' })}>
        修改用户名
      </button>
    </div>
  );
}
```

### 优化：用useMemo稳定value引用

```jsx
function App() {
  const [theme, setTheme] = useState('light');
  const [count, setCount] = useState(0);
  
  // 避免count变化时，theme Consumer也重新渲染
  const themeValue = useMemo(() => ({ theme, setTheme }), [theme]);
  
  return (
    <ThemeContext.Provider value={themeValue}>
      <button onClick={() => setCount(count + 1)}>Count: {count}</button>
      <ThemedButton />
    </ThemeContext.Provider>
  );
}
```

## 7. 深入理解

### Context的更新机制与性能陷阱

**关键认知：Provider的value变化会触发所有Consumer重新渲染**

```jsx
// ❌ 性能问题：每次Parent渲染都创建新对象，即使theme没变
function Parent() {
  const [theme, setTheme] = useState('light');
  const [count, setCount] = useState(0);
  
  // 每次渲染都是新对象 {}
  const value = { theme, setTheme };
  
  return (
    <ThemeContext.Provider value={value}>
      <Child />
      <button onClick={() => setCount(count + 1)}>Count: {count}</button>
    </ThemeContext.Provider>
  );
}

// Child会在count变化时重新渲染，即使theme没变
function Child() {
  const { theme } = useContext(ThemeContext);
  console.log('Child渲染了');  // count变化也会打印
  return <div>{theme}</div>;
}
```

**优化方案1：useMemo稳定引用**

```jsx
function Parent() {
  const [theme, setTheme] = useState('light');
  const [count, setCount] = useState(0);
  
  // 只有theme变化时才创建新对象
  const value = useMemo(() => ({ theme, setTheme }), [theme]);
  
  return (
    <ThemeContext.Provider value={value}>
      <Child />
      <button onClick={() => setCount(count + 1)}>Count: {count}</button>
    </ThemeContext.Provider>
  );
}
```

**优化方案2：拆分Context**

```jsx
// 拆成两个Context，避免相互影响
const ThemeContext = createContext();
const ThemeUpdateContext = createContext();

function ThemeProvider({ children }) {
  const [theme, setTheme] = useState('light');
  
  return (
    <ThemeContext.Provider value={theme}>
      <ThemeUpdateContext.Provider value={setTheme}>
        {children}
      </ThemeUpdateContext.Provider>
    </ThemeContext.Provider>
  );
}

// 只读取theme的组件
function Display() {
  const theme = useContext(ThemeContext);  // setTheme变化不会触发渲染
  return <div>{theme}</div>;
}

// 只修改theme的组件
function Button() {
  const setTheme = useContext(ThemeUpdateContext);  // theme变化不会触发渲染
  return <button onClick={() => setTheme('dark')}>切换</button>;
}
```

**优化方案3：useContextSelector（第三方库）**

```jsx
import { createContext, useContextSelector } from 'use-context-selector';

const UserContext = createContext();

function UserProvider({ children }) {
  const [user, setUser] = useState({ name: '张三', age: 25 });
  return (
    <UserContext.Provider value={{ user, setUser }}>
      {children}
    </UserContext.Provider>
  );
}

// 只订阅name，age变化不会触发渲染
function NameDisplay() {
  const name = useContextSelector(UserContext, v => v.user.name);
  return <div>{name}</div>;
}
```

### Context vs Redux：何时选择谁？

| 对比维度 | Context | Redux |
|---------|---------|-------|
| **适用场景** | 半全局、变化不频繁的数据 | 复杂全局状态、高频变化 |
| **典型数据** | 主题、语言、用户信息、权限 | 购物车、表单、实时数据 |
| **更新粒度** | Provider下所有Consumer都更新 | connect的组件按需更新 |
| **调试工具** | React DevTools | Redux DevTools（时间旅行） |
| **样板代码** | 少 | 多（actions、reducers） |
| **中间件** | 无 | 有（redux-thunk、saga） |
| **学习成本** | 低 | 中等 |

**选择建议**：
- **先用Context**：如果数据简单、变化不频繁，Context足够
- **再考虑Redux**：如果出现性能问题、需要复杂异步逻辑、需要时间旅行调试
- **混合使用**：Context管理主题/语言，Redux管理业务状态

### Context的默认值与Provider嵌套

**defaultValue的作用**

```jsx
// 创建时提供默认值
const ThemeContext = createContext('light');

// 如果组件树中没有Provider，useContext会返回默认值
function Button() {
  const theme = useContext(ThemeContext);  // 返回 'light'
  return <button>{theme}</button>;
}

// 没有Provider包裹也能用
<Button />  // 显示 'light'
```

**Provider嵌套（内层覆盖外层）**

```jsx
const ThemeContext = createContext('light');

function App() {
  return (
    <ThemeContext.Provider value="dark">
      <Header />  {/* 使用 'dark' */}
      
      <ThemeContext.Provider value="light">
        <Sidebar />  {/* 使用 'light'，覆盖了外层 */}
      </ThemeContext.Provider>
      
      <Footer />  {/* 使用 'dark' */}
    </ThemeContext.Provider>
  );
}
```

### 多个Context的组合使用

**问题：Provider地狱**

```jsx
// ❌ 可读性差
function App() {
  return (
    <ThemeProvider>
      <LanguageProvider>
        <UserProvider>
          <AuthProvider>
            <RouterProvider>
              <Content />
            </RouterProvider>
          </AuthProvider>
        </UserProvider>
      </LanguageProvider>
    </ThemeProvider>
  );
}
```

**解决方案：组合Provider**

```jsx
// ✅ 封装组合Provider
function AppProviders({ children }) {
  return (
    <ThemeProvider>
      <LanguageProvider>
        <UserProvider>
          <AuthProvider>
            {children}
          </AuthProvider>
        </UserProvider>
      </LanguageProvider>
    </ThemeProvider>
  );
}

function App() {
  return (
    <AppProviders>
      <Content />
    </AppProviders>
  );
}
```

### Context与组件组合的对比

**场景：Page组件需要user数据**

**方案1：Context（适合跨多层）**

```jsx
<UserContext.Provider value={user}>
  <Layout>
    <Sidebar>
      <Nav>
        <UserMenu />  {/* 深层组件，用useContext(UserContext) */}
      </Nav>
    </Sidebar>
  </Layout>
</UserContext.Provider>
```

**方案2：组件组合（适合层级浅）**

```jsx
// 直接传递完整的组件，而不是数据
<Layout
  sidebar={<Sidebar user={user} />}
  content={<Content user={user} />}
/>

// Layout不需要知道user
function Layout({ sidebar, content }) {
  return (
    <div>
      <div className="sidebar">{sidebar}</div>
      <div className="content">{content}</div>
    </div>
  );
}
```

**选择原则**：
- 能用组件组合就不用Context（减少隐式依赖）
- 真正需要跨多层、多个地方消费，才用Context

## 8. 一句话总结

Context通过Provider向下广播数据、useContext在任意层级消费，避免props层层传递，但需注意性能优化避免不必要渲染。

## 9. 面试回答路径

### 问题：Context是什么？解决什么问题？

**回答结构（90秒）**：

**第一步：定义**
> "Context是React提供的跨层级数据共享机制，由createContext创建、Provider提供数据、useContext消费数据三部分组成。"

**第二步：解决的问题**
> "主要解决props drilling问题。当数据需要传递给很深层的组件时，不用Context就得在每个中间组件都加props传递，代码冗余且难维护。Context可以让深层组件直接消费，跳过中间层。"

**第三步：举例说明**
> "典型场景是主题切换。App顶层切换主题，深层的Button组件需要知道当前主题。用Context的话，App用ThemeProvider包裹，Button用useContext直接获取，中间的Layout、Sidebar等组件不需要参与传递。"

```jsx
const ThemeContext = createContext();

function App() {
  const [theme, setTheme] = useState('light');
  return (
    <ThemeContext.Provider value={theme}>
      <Layout>
        <Button />  {/* 深层组件 */}
      </Layout>
    </ThemeContext.Provider>
  );
}

function Button() {
  const theme = useContext(ThemeContext);  // 直接获取
  return <button className={theme}>按钮</button>;
}
```

**第四步：补充使用场景**
> "适合半全局、变化不频繁的数据，如主题、语言、用户信息、权限配置等。如果是复杂全局状态或高频变化，建议用Redux。"

### 问题：Context有什么性能问题？如何优化？

**回答结构（2分钟）**：

**第一步：说明问题**
> "Context的主要性能问题是：Provider的value变化时，所有使用useContext的组件都会重新渲染，即使它们只用到value的一部分。"

**第二步：举例说明**
> "比如Provider的value是对象`{theme, user}`，某个组件只用theme，但user变化时这个组件也会重新渲染，造成不必要的性能开销。"

```jsx
// 问题代码
function App() {
  const [theme, setTheme] = useState('light');
  const [user, setUser] = useState({name: '张三'});
  
  // 每次渲染都是新对象，即使theme和user都没变
  const value = { theme, user };
  
  return <MyContext.Provider value={value}>...</MyContext.Provider>;
}
```

**第三步：提供优化方案**

**方案1：useMemo稳定引用**
> "用useMemo包裹value对象，只在依赖变化时才创建新对象。"

```jsx
const value = useMemo(() => ({ theme, user }), [theme, user]);
```

**方案2：拆分Context**
> "把theme和user拆成两个独立的Context，只用theme的组件不会受user变化影响。"

```jsx
<ThemeContext.Provider value={theme}>
  <UserContext.Provider value={user}>
    {children}
  </UserContext.Provider>
</ThemeContext.Provider>
```

**方案3：状态和更新函数分离**
> "把state和setState拆成两个Context，只修改不读取的组件不会因state变化而渲染。"

```jsx
<StateContext.Provider value={state}>
  <UpdateContext.Provider value={setState}>
    {children}
  </UpdateContext.Provider>
</StateContext.Provider>
```

**第四步：给出选择建议**
> "一般情况用useMemo就够了。如果Context很大、Consumer很多，考虑拆分。极端情况可以用第三方库use-context-selector实现精确订阅。"

### 问题：Context和Redux有什么区别？

**回答结构（2分钟）**：

**第一步：定位不同**
> "Context是React内置的跨层级数据共享机制，Redux是独立的状态管理库。两者定位不同：Context适合半全局、简单数据，Redux适合全局、复杂状态。"

**第二步：核心区别对比**

**更新粒度**
> "Context的Provider value变化，所有Consumer都更新；Redux通过connect或selector，组件只在用到的数据变化时更新，粒度更细。"

**调试能力**
> "Redux有强大的DevTools，支持时间旅行、action追踪；Context只能通过React DevTools看当前值。"

**中间件生态**
> "Redux有丰富的中间件处理异步、日志、持久化；Context需要自己实现。"

**样板代码**
> "Context代码少，createContext + Provider + useContext就行；Redux需要定义actions、reducers、store，代码多但结构清晰。"

**第三步：举例说明适用场景**

**Context适合：**
> "主题切换、语言设置、用户基本信息这类全局但变化不频繁的数据。"

```jsx
<ThemeContext.Provider value={theme}>
  <App />
</ThemeContext.Provider>
```

**Redux适合：**
> "购物车、表单状态、实时数据、需要历史回溯的复杂状态。"

```jsx
// Redux的优势：时间旅行、action追踪
dispatch({ type: 'ADD_TO_CART', item: {...} });
```

**第四步：给出选择建议**
> "实际项目中常混合使用：Context管理主题、语言这类配置，Redux管理业务状态。不要一上来就用Redux，先用Context，遇到性能问题或复杂逻辑再引入Redux。"

### 问题：Context的defaultValue什么时候生效？

**回答结构（90秒）**：

**第一步：说明生效条件**
> "defaultValue只在组件树中找不到对应的Provider时生效。如果有Provider，即使value是undefined，也会用Provider的value而不是defaultValue。"

**第二步：举例说明**

```jsx
const ThemeContext = createContext('light');  // 默认值

// 场景1：没有Provider，使用默认值
function App() {
  return <Button />;  // theme = 'light'
}

// 场景2：有Provider，使用Provider的value（即使是undefined）
function App() {
  return (
    <ThemeContext.Provider value={undefined}>
      <Button />  {/* theme = undefined，不是 'light' */}
    </ThemeContext.Provider>
  );
}
```

**第三步：说明实际用途**
> "defaultValue主要用于：1) 单元测试时不想包Provider；2) 组件库默认值，用户可以选择性提供Provider覆盖；3) 类型提示，告诉开发者Context的结构。"

**第四步：最佳实践**
> "生产代码通常会确保有Provider，defaultValue主要起类型提示作用。可以设置为null或抛出错误，强制开发者提供Provider。"

```jsx
const ThemeContext = createContext(null);

function useTheme() {
  const theme = useContext(ThemeContext);
  if (theme === null) {
    throw new Error('useTheme必须在ThemeProvider内部使用');
  }
  return theme;
}
```
