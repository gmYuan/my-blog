# React组件通信

## 1. 层级关系图

```
       React组件树
           |
    通信需求分类
      /    |    \
  父→子  子→父  跨层级
    |      |      |
  props  callback Context
    |      |      |
children 状态提升  ref
```

## 2. 各自的职责

### props（父→子）
- **职责**: 父组件向子组件传递数据和函数
- **输入**: 任意类型数据
- **输出**: 子组件接收到props
- **类比**: 快递从总部发到门店，单向配送

### children（特殊props）
- **职责**: 传递组件标签内部的内容，实现组件组合
- **输入**: JSX、字符串、数组、函数等
- **输出**: 父组件通过props.children获取
- **类比**: 礼盒包装，盒子不用知道里面装什么，由使用者决定

### callback（子→父）
- **职责**: 子组件通过调用父组件传来的函数，向上传递数据
- **输入**: 父组件的setState或自定义函数
- **输出**: 触发父组件状态更新
- **类比**: 门店上报销售数据给总部

### 状态提升
- **职责**: 多个子组件共享状态时，将state提升到共同父组件
- **输入**: 子组件的状态需求
- **输出**: 父组件统一管理，通过props分发
- **类比**: 多个门店的库存由总仓统一管理

## 3. 调用链路

**场景：父组件有计数器，子组件有按钮**

```
1. 父组件定义 count 和 setCount
   ↓
2. 父组件通过 props 传递 count 和 handleIncrement 给子组件
   ↓
3. 子组件渲染，显示 count，绑定 onClick 到 handleIncrement
   ↓
4. 用户点击子组件按钮
   ↓
5. 子组件调用 handleIncrement()
   ↓
6. 父组件的 setCount 被执行，count 更新
   ↓
7. 父组件重新渲染，新 count 通过 props 传给子组件
   ↓
8. 子组件重新渲染，显示新 count
```

## 4. 为什么这样设计？

**原因1**: 单向数据流 - 数据从上往下流，便于追踪和调试
- 例子：bug出现时，只需检查父组件的state修改逻辑，不用担心子组件意外改了数据

**原因2**: 组合优于继承 - children机制让组件像容器一样灵活复用
- 例子：`<Card>` 组件不需要知道内部放什么，可以放文字、图片、表单等任意内容

## 5. 类比锚定

想象一个公司组织架构：
- **props** 是总部下发通知给部门（单向，从上到下）
- **callback** 是部门向总部汇报（通过总部提供的汇报渠道）
- **children** 是文件夹，文件夹本身不关心装什么文件，由使用者决定
- **状态提升** 是多个部门需要共享数据时，数据统一存在总部，各部门从总部获取

## 6. 代码示例

```jsx
import { useState } from 'react';

// 父组件
function Parent() {
  const [count, setCount] = useState(0);
  
  // 传给子组件的回调函数
  const handleIncrement = () => {
    setCount(count + 1);
  };
  
  return (
    <div>
      <h1>父组件 count: {count}</h1>
      
      {/* 父→子: 通过props传数据和函数 */}
      <Child count={count} onIncrement={handleIncrement}>
        {/* children: 标签内部的内容 */}
        <p>这是通过children传递的内容</p>
      </Child>
    </div>
  );
}

// 子组件
function Child({ count, onIncrement, children }) {
  return (
    <div>
      <p>子组件收到的 count: {count}</p>
      
      {/* 子→父: 调用父组件传来的回调 */}
      <button onClick={onIncrement}>点击 +1</button>
      
      {/* 渲染 children */}
      <div>{children}</div>
    </div>
  );
}
```

### children 高级用法

```jsx
// Card 容器组件
function Card({ children, title }) {
  return (
    <div className="card">
      <h2>{title}</h2>
      <div className="card-body">
        {children}  {/* 灵活放置任意内容 */}
      </div>
    </div>
  );
}

// 使用
<Card title="用户信息">
  <p>姓名: 张三</p>
  <p>年龄: 25</p>
  <button>编辑</button>
</Card>
```

### 状态提升示例

```jsx
// 两个输入框需要共享同步的值
function Parent() {
  const [text, setText] = useState('');  // 状态提升到父组件
  
  return (
    <div>
      <InputA value={text} onChange={setText} />
      <InputB value={text} onChange={setText} />
    </div>
  );
}
```

## 7. 深入理解

### children的本质与设计哲学

**children不是语法糖，是React组合模式的核心**

```jsx
// 这两种写法完全等价
<Card>内容</Card>
<Card children="内容" />

// children可以是任意类型
<Layout>
  {isLoggedIn ? <Dashboard /> : <Login />}
</Layout>

// React.Children工具函数
function Parent({ children }) {
  // 遍历children
  const childrenArray = React.Children.toArray(children);
  
  // 克隆并添加props
  const enhancedChildren = React.Children.map(children, (child) => 
    React.cloneElement(child, { extraProp: 'value' })
  );
  
  return <div>{enhancedChildren}</div>;
}
```

**为什么组合优于继承？**
- 继承是"is-a"关系（Button是Component），但UI组件更多是"has-a"关系（Card有内容）
- 继承链深了会导致"脆弱基类问题"：修改父类影响所有子类
- 组合通过props和children可以灵活搭配，而不是被继承链束缚

### 状态提升的深层原因

**单一数据源（Single Source of Truth）原则**

```jsx
// ❌ 错误：两个组件各自维护state，容易不同步
function InputA() {
  const [value, setValue] = useState('');
  return <input value={value} onChange={e => setValue(e.target.value)} />;
}

function InputB() {
  const [value, setValue] = useState('');
  return <input value={value} onChange={e => setValue(e.target.value)} />;
}

// ✅ 正确：状态提升到父组件，保证单一数据源
function Parent() {
  const [value, setValue] = useState('');
  return (
    <>
      <InputA value={value} onChange={setValue} />
      <InputB value={value} onChange={setValue} />
    </>
  );
}
```

**状态提升的判断规则**：
1. 两个组件需要**共享**同一份数据 → 提升到最近的共同父组件
2. 数据被多个组件**读取** → 提升到能覆盖所有读取者的父组件
3. 数据被某个组件**修改**，其他组件需要**响应** → 提升并通过props分发

### props的单向数据流与不可变性

**为什么props是只读的？**

```jsx
// ❌ 错误：直接修改props（React会警告）
function Child({ user }) {
  user.name = '李四';  // 违反不可变性原则
  return <div>{user.name}</div>;
}

// ✅ 正确：通过回调通知父组件修改
function Child({ user, onUserChange }) {
  const handleClick = () => {
    onUserChange({ ...user, name: '李四' });  // 创建新对象
  };
  return <button onClick={handleClick}>修改</button>;
}
```

**单向数据流的优势**：
- **可预测性**：数据只能从父到子，出问题时排查路径清晰
- **可调试性**：React DevTools可以追踪props变化路径
- **可测试性**：组件的输出只依赖输入（props），容易写单元测试

### 性能优化：避免不必要的回调重新创建

```jsx
// ❌ 问题：每次渲染创建新函数，子组件会不必要地重新渲染
function Parent() {
  const [count, setCount] = useState(0);
  
  const handleIncrement = () => {  // 每次渲染都是新函数
    setCount(count + 1);
  };
  
  return <Child onIncrement={handleIncrement} />;
}

// ✅ 优化1：useCallback缓存函数
function Parent() {
  const [count, setCount] = useState(0);
  
  const handleIncrement = useCallback(() => {
    setCount(c => c + 1);  // 使用函数式更新
  }, []);  // 依赖为空，函数永不变
  
  return <MemoizedChild onIncrement={handleIncrement} />;
}

// ✅ 优化2：如果子组件用memo包裹
const Child = memo(function Child({ onIncrement }) {
  console.log('Child渲染');
  return <button onClick={onIncrement}>+1</button>;
});
```

### 其他通信方式对比

| 方式 | 适用场景 | 优点 | 缺点 |
|------|---------|------|------|
| props/callback | 父子、祖孙（浅层） | 简单直接、类型安全 | 层级深时代码冗余 |
| Context | 跨多层、半全局数据 | 避免props drilling | 过度使用影响性能 |
| Redux/Zustand | 全局复杂状态 | 集中管理、时间旅行 | 样板代码多 |
| EventEmitter | 非父子关系、兄弟组件 | 解耦 | 难追踪、容易内存泄漏 |
| ref | 访问DOM、组件实例 | 命令式操作 | 破坏声明式、难测试 |

## 8. 一句话总结

React通过props向下传数据、callback向上通知、children实现组合、状态提升保证单一数据源，保持可预测的单向数据流。

### 问题：父子组件如何通信？

**回答结构（2分钟）**：

**第一步：直接答核心方式**
> "父到子用props传递，子到父通过callback回调。具体来说..."

**第二步：解释单向数据流**
> "这体现了React的单向数据流设计，父组件通过props向下传数据和函数，子组件通过调用这些函数向上通知父组件，数据始终是单向流动的。"

**第三步：举一个完整例子**
> "举个例子，表单组件：父组件维护表单数据state，通过value传给子输入框，子输入框onChange时调用父组件传来的回调函数更新state，这样保证了单一数据源。"

```jsx
// 准备好这个例子
function Form() {
  const [name, setName] = useState('');
  return <Input value={name} onChange={setName} />;
}

function Input({ value, onChange }) {
  return <input value={value} onChange={e => onChange(e.target.value)} />;
}
```

**第四步：补充children（如果时间够）**
> "另外，React还提供了children机制，让组件可以像容器一样接收任意内容，这是组合优于继承的体现。"

**第五步：提及其他方式（显示知识广度）**
> "跨层级通信可以用Context，复杂全局状态用Redux，这些都是基于props/callback这个基础通信模式的扩展。"

### 问题：什么是状态提升？为什么需要？

**回答结构（90秒）**：

**第一步：定义**
> "状态提升是指当多个组件需要共享同一份数据时，将state提升到它们最近的共同父组件中管理。"

**第二步：说明原因（单一数据源）**
> "这遵循了单一数据源原则，避免多个组件各自维护副本导致数据不一致。"

**第三步：举例说明**
> "比如温度转换器，摄氏度和华氏度输入框需要同步，如果各自维护state就会不一致。正确做法是把温度state提升到父组件，父组件统一管理，两个输入框通过props接收值，通过callback通知更新。"

```jsx
function TemperatureConverter() {
  const [celsius, setCelsius] = useState(0);
  
  return (
    <>
      <CelsiusInput value={celsius} onChange={setCelsius} />
      <FahrenheitInput value={celsius * 9/5 + 32} onChange={f => setCelsius((f-32)*5/9)} />
    </>
  );
}
```

**第四步：总结规则**
> "判断是否需要状态提升的规则：如果两个组件需要共享、同步同一份数据，就提升到共同父组件。"

### 问题：children是什么？为什么需要？

**回答结构（90秒）**：

**第一步：定义**
> "children是一个特殊的props，代表组件标签内部的内容。当你写`<Card>内容</Card>`时，'内容'就会被传递给Card的props.children。"

**第二步：说明设计理念**
> "这体现了React'组合优于继承'的设计哲学。children让组件像容器一样，不需要预知内部会放什么内容，由使用者决定。"

**第三步：举例说明灵活性**
> "比如Modal组件，不同场景需要展示不同内容：有时是表单、有时是确认框、有时是图片。通过children，Modal只负责弹窗逻辑，内容由调用方传入，实现了高度复用。"

```jsx
<Modal isOpen={true}>
  <h2>删除确认</h2>
  <p>确定要删除吗？</p>
  <Button onClick={handleDelete}>确定</Button>
</Modal>

// Modal组件只关心弹窗行为，不关心内容
function Modal({ isOpen, children }) {
  if (!isOpen) return null;
  return <div className="modal">{children}</div>;
}
```

**第四步：对比继承方式（显示深度）**
> "如果用继承实现，需要为每种场景创建子类：ConfirmModal、FormModal、ImageModal，扩展性差。用children组合，一个Modal组件就够了。"

### 问题：如何避免props drilling（层层传递）？

**回答结构（2分钟）**：

**第一步：承认问题**
> "props drilling确实是React开发中的常见问题，当组件树很深时，中间层组件不需要某个props，却要一层层传递，代码冗余且难维护。"

**第二步：提供多种解决方案**

**方案1：Context（推荐）**
> "对于跨多层的半全局数据，如主题、语言、用户信息，用Context最合适。上层Provider提供，下层useContext消费，中间层不需要参与。"

**方案2：组件组合**
> "有时重新设计组件结构就能避免drilling。比如Page组件直接把完整的Header组件通过children传给Layout，而不是传数据让Layout去构建Header。"

```jsx
// ❌ props drilling
<Layout user={user}>
  <Content user={user}>
    <Profile user={user} />
  </Content>
</Layout>

// ✅ 组件组合
<Layout>
  <Content>
    <Profile user={user} />  {/* 直接在顶层传入 */}
  </Content>
</Layout>
```

**方案3：状态管理库**
> "如果是全局复杂状态，用Redux、Zustand等状态管理库，任何组件都可以直接connect或useStore获取，不需要传递。"

**第三步：给出选择建议**
> "选择原则：能用组件组合解决就不用Context，需要跨层但不复杂用Context，全局复杂状态用状态管理库。"
