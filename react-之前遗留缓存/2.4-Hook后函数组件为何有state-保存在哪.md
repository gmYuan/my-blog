# Hook后函数组件为何有state-保存在哪

## 1. 层级关系图

```
     React渲染
          |
    ┌─────┴─────┐
   组件       Fiber树
    |          |
  函数执行   Fiber节点
    |          |
  调用Hooks  memoizedState
              |
           Hooks链表
        (状态持久化)
```

## 2. 各自的职责

### 函数组件
- **职责**: 每次渲染重新执行
- **特点**: 本身无状态、无实例
- **类比**: 一张流程图，每次照着执行

### Fiber节点
- **职责**: React为每个组件创建的内部对象
- **包含**: 组件类型、props、state、DOM引用等
- **类比**: 组件的身份证，记录所有信息

### memoizedState
- **职责**: Fiber节点上的属性，存储Hooks链表
- **结构**: 链表，每个节点对应一个Hook调用
- **类比**: 身份证上的"家庭成员"栏，按顺序记录

### Hook对象
- **职责**: 链表中的一个节点，存储单个Hook的状态
- **包含**: memoizedState（Hook的值）、queue（更新队列）、next（下一个Hook）
- **类比**: 家庭成员中的一个人的信息卡

## 3. 调用链路

**场景：组件有两个useState**

```
首次渲染:
1. React 创建 Fiber 节点
   ↓
2. 组件函数执行
   ↓
3. 第1个 useState(0) 调用
   ↓
4. React 创建 Hook1 对象 { memoizedState: 0, next: null }
   ↓
5. Fiber.memoizedState = Hook1
   ↓
6. 第2个 useState('') 调用
   ↓
7. React 创建 Hook2 对象 { memoizedState: '', next: null }
   ↓
8. Hook1.next = Hook2
   ↓
9. 函数执行完毕，返回JSX，Fiber保留状态

第二次渲染（调用了setCount）:
1. React 找到对应的 Fiber 节点
   ↓
2. 组件函数重新执行
   ↓
3. 第1个 useState 调用
   ↓
4. React 从 Fiber.memoizedState 读取 Hook1
   ↓
5. 返回 Hook1.memoizedState 的值（已更新为新值）
   ↓
6. 第2个 useState 调用
   ↓
7. React 从 Hook1.next 读取 Hook2
   ↓
8. 返回 Hook2.memoizedState 的值
```

## 4. 为什么这样设计？

**原因1**: 函数组件性能更好 - 无需创建类实例，降低内存开销
- 函数执行完就可以销毁，状态存在Fiber上复用

**原因2**: 保持函数纯净 - 函数本身无状态，状态由React管理
- 便于测试、调试、时间旅行（React DevTools回溯状态）

## 5. 类比锚定

想象你去图书馆借书：
- **函数组件** 是图书管理员，每次你来都是重新工作（函数执行）
- **Fiber节点** 是你的借书证，记录你是谁、借了什么书
- **memoizedState** 是借书证上的"当前借阅清单"
- **Hooks链表** 是清单上的书目：第1本、第2本、第3本...
- 管理员换班了（函数重新执行），但借书证还在（Fiber保留），清单不会丢

## 6. 代码示例

### Hook的内部结构（简化版）

```js
// Fiber节点结构（简化）
const fiber = {
  type: MyComponent,        // 组件函数
  memoizedState: null,      // Hooks链表的头节点
  // ... 其他属性
};

// Hook对象结构（简化）
const hook = {
  memoizedState: 0,         // Hook的当前值
  queue: null,              // 更新队列
  next: null,               // 下一个Hook
};
```

### useState 的简化实现原理

```js
// React内部实现（极度简化）
let currentFiber = null;     // 当前正在渲染的Fiber
let currentHook = null;      // 当前正在处理的Hook

function useState(initialValue) {
  // 首次渲染：创建新Hook
  if (!currentFiber.memoizedState) {
    const hook = {
      memoizedState: initialValue,
      queue: [],
      next: null
    };
    currentFiber.memoizedState = hook;
    currentHook = hook;
  } 
  // 重新渲染：读取已有Hook
  else {
    if (!currentHook) {
      currentHook = currentFiber.memoizedState;  // 从头开始
    } else {
      currentHook = currentHook.next;  // 移到下一个
    }
  }
  
  const setState = (newValue) => {
    currentHook.memoizedState = newValue;
    // 触发重新渲染...
  };
  
  return [currentHook.memoizedState, setState];
}
```

### 实际使用示例

```jsx
import { useState } from 'react';

function Counter() {
  // 第1个Hook：存在 Fiber.memoizedState
  const [count, setCount] = useState(0);
  
  // 第2个Hook：存在 Hook1.next
  const [name, setName] = useState('张三');
  
  console.log('函数重新执行了');  // 每次渲染都会执行
  
  return (
    <div>
      <p>Count: {count}</p>
      <p>Name: {name}</p>
      <button onClick={() => setCount(count + 1)}>+1</button>
    </div>
  );
}

// Fiber结构示意：
// fiber.memoizedState → Hook1 → Hook2 → null
//                       (count)  (name)
```

### 可视化Fiber和Hooks的关系

```
Counter组件的Fiber节点:
{
  type: Counter,
  memoizedState: {              ← Hook1 (count)
    memoizedState: 0,
    queue: [...],
    next: {                     ← Hook2 (name)
      memoizedState: '张三',
      queue: [...],
      next: null
    }
  }
}

渲染过程:
1. 函数执行 Counter()
2. useState(0) → 读取/创建 Hook1
3. useState('张三') → 读取/创建 Hook2
4. 函数执行完毕，返回JSX
5. Hooks链表保留在Fiber上
```

## 7. 深入理解

### Fiber架构的设计目的

**为什么需要Fiber？**

在Fiber出现之前，React使用Stack Reconciler（栈调和器），递归遍历虚拟DOM树进行diff和更新，一旦开始就无法中断，导致：
- 大型应用渲染卡顿
- 动画掉帧
- 用户输入响应延迟

Fiber将递归变成链表遍历，可以中断和恢复，实现**时间切片**和**优先级调度**。

**Fiber是什么？**

Fiber既是一种数据结构，也是一种工作单元：
- **数据结构**：每个组件对应一个Fiber节点，包含组件信息和Hooks链表
- **工作单元**：可中断、可恢复的最小执行单元

### Fiber节点的完整结构

```js
const fiber = {
  // 组件相关
  type: Counter,           // 组件类型（函数/类/DOM标签）
  key: null,               // key
  elementType: Counter,    // 元素类型
  
  // 父子兄弟指针（链表结构）
  return: parentFiber,     // 父节点
  child: childFiber,       // 第一个子节点
  sibling: siblingFiber,   // 兄弟节点
  
  // 状态相关
  memoizedState: hooksList, // Hooks链表
  memoizedProps: {},        // 上次渲染的props
  pendingProps: {},         // 本次渲染的props
  
  // 副作用相关
  flags: Update | Placement, // 副作用标记
  subtreeFlags: 0,           // 子树副作用
  
  // 双缓冲
  alternate: workInProgressFiber, // 指向另一棵树的对应节点
  
  // DOM相关
  stateNode: domNode,       // 对应的DOM节点或组件实例
  
  // 调度相关
  lanes: 0,                 // 优先级
  childLanes: 0,            // 子树优先级
};
```

### 双缓冲机制（Double Buffering）

**React维护两棵Fiber树**

```
当前树(current)         工作树(workInProgress)
     App                      App'
      |                        |
    Counter      更新时 →    Counter'
      |          复制+修改      |
    Button                   Button'
```

**工作流程**：
1. 首次渲染创建current树
2. 更新时复制current树为workInProgress树
3. 在workInProgress树上修改
4. 渲染完成后，workInProgress变成新的current

**优势**：
- 可中断：工作树修改一半被中断，不影响当前树，用户看到的是旧UI
- 可回滚：工作树出错，丢弃即可，不影响当前树
- 性能：复用Fiber节点，不每次都创建

### useState的完整实现流程

**mount阶段（首次渲染）**

```js
function mountState(initialState) {
  // 1. 创建Hook对象
  const hook = {
    memoizedState: initialState,
    baseState: initialState,
    baseQueue: null,
    queue: {
      pending: null,
      lanes: 0,
      dispatch: null,
      lastRenderedReducer: basicStateReducer,
      lastRenderedState: initialState
    },
    next: null
  };
  
  // 2. 挂载到Fiber的memoizedState链表
  if (workInProgressHook === null) {
    currentlyRenderingFiber.memoizedState = workInProgressHook = hook;
  } else {
    workInProgressHook = workInProgressHook.next = hook;
  }
  
  // 3. 创建dispatch函数（绑定了当前Fiber和queue）
  const dispatch = dispatchSetState.bind(null, currentlyRenderingFiber, queue);
  queue.dispatch = dispatch;
  
  return [hook.memoizedState, dispatch];
}
```

**update阶段（重新渲染）**

```js
function updateState() {
  // 1. 从current Fiber克隆Hook到workInProgress Fiber
  const hook = updateWorkInProgressHook();
  
  // 2. 处理更新队列
  const queue = hook.queue;
  const pending = queue.pending;
  
  if (pending !== null) {
    // 3. 计算新状态
    let newState = hook.memoizedState;
    let update = pending.next;
    
    do {
      const action = update.action;
      newState = typeof action === 'function' ? action(newState) : action;
      update = update.next;
    } while (update !== pending.next);
    
    // 4. 保存新状态
    hook.memoizedState = newState;
    queue.pending = null;
  }
  
  return [hook.memoizedState, queue.dispatch];
}
```

**setState的异步批量更新**

```jsx
function Counter() {
  const [count, setCount] = useState(0);
  
  const handleClick = () => {
    setCount(count + 1);  // 1
    setCount(count + 1);  // 2
    setCount(count + 1);  // 3
    console.log(count);   // 仍然是0，因为闭包
  };
  // 点击后count只会+1，因为三次setCount都基于旧的count=0
}

// 解决方案：函数式更新
const handleClick = () => {
  setCount(c => c + 1);  // 基于最新值
  setCount(c => c + 1);
  setCount(c => c + 1);
  // count会+3
};
```

### 多个组件实例的状态隔离

```jsx
function Counter() {
  const [count, setCount] = useState(0);
  return <button onClick={() => setCount(count + 1)}>{count}</button>;
}

function App() {
  return (
    <>
      <Counter />  {/* Fiber1，有自己的memoizedState链表 */}
      <Counter />  {/* Fiber2，有自己的memoizedState链表 */}
      <Counter />  {/* Fiber3，有自己的memoizedState链表 */}
    </>
  );
}
```

**每个组件实例都有独立的Fiber节点**，因此状态互不影响。

### 组件卸载与Fiber清理

```jsx
function Parent() {
  const [show, setShow] = useState(true);
  
  return (
    <div>
      <button onClick={() => setShow(!show)}>Toggle</button>
      {show && <Counter />}
    </div>
  );
}

// Counter卸载时
// 1. 执行useEffect的清理函数
// 2. Fiber节点标记为deletion
// 3. commit阶段从DOM移除
// 4. Fiber节点和Hooks链表会被垃圾回收
// 5. 下次show=true时创建新的Fiber节点，state重新初始化
```

## 8. 一句话总结

函数组件每次渲染重新执行，useState的状态存在Fiber节点的memoizedState链表上，通过Fiber的持久化和双缓冲机制保证状态不丢失，每个组件实例有独立Fiber因此状态隔离。

## 9. 面试回答路径

### 问题：函数组件每次渲染都重新执行，为什么useState的值不会重置？

**回答结构（2分钟）**：

**第一步：说明现象**
> "虽然函数组件每次渲染都重新执行，但useState返回的值是上次的state，不会重置为初始值。这是React Hooks的核心机制之一。"

**第二步：解释存储位置**
> "useState的状态不是存在函数内部，而是存在React的Fiber节点上。每个组件对应一个Fiber节点，Fiber节点有个memoizedState属性，指向Hooks链表，链表上存储了所有Hook的状态。"

```js
// Fiber节点结构
const fiber = {
  type: Counter,          // 组件函数
  memoizedState: {        // Hooks链表头
    memoizedState: 0,     // useState的值
    next: {               // 下一个Hook
      memoizedState: '',
      next: null
    }
  }
};
```

**第三步：说明工作流程**
> "首次渲染时，useState会创建Hook对象加入Fiber的链表。重新渲染时，虽然函数重新执行，但React知道这是同一个组件（通过Fiber节点），会从Fiber的链表上读取之前保存的状态，而不是使用初始值。"

```jsx
// 首次渲染
function Counter() {
  const [count, setCount] = useState(0);  // 创建Hook，保存到Fiber
  return <div>{count}</div>;              // 显示0
}

// setCount(1)后重新渲染
function Counter() {
  const [count, setCount] = useState(0);  // 从Fiber读取，返回1（不是0）
  return <div>{count}</div>;              // 显示1
}
```

**第四步：类比说明**
> "可以把Fiber节点理解成组件的'身份证'，函数执行完就销毁了，但身份证一直保存着，上面记录了这个组件的所有状态。每次函数执行就是'照着身份证读取信息'，而不是'重新登记'。"

**第五步：补充双缓冲**
> "React还用了双缓冲机制，维护两棵Fiber树。更新时在备用树上修改，完成后交换，这样可以中断和回滚，保证状态安全。"

### 问题：多个组件使用同一个函数组件，它们的state会互相影响吗？

**回答结构（90秒）**：

**第一步：直接答案**
> "不会互相影响。虽然用的是同一个函数组件，但每个组件实例都有独立的Fiber节点，因此状态完全隔离。"

**第二步：解释原因**
> "每次在JSX中写`<Counter />`，React都会创建一个新的Fiber节点，每个Fiber节点有自己的memoizedState链表。虽然执行的是同一个Counter函数，但读写的是各自Fiber节点上的状态。"

```jsx
function App() {
  return (
    <>
      <Counter />  {/* Fiber节点1，count独立 */}
      <Counter />  {/* Fiber节点2，count独立 */}
      <Counter />  {/* Fiber节点3，count独立 */}
    </>
  );
}
```

**第三步：可视化说明**

```
Fiber树:
App (Fiber0)
 ├─ Counter (Fiber1)
 │   └─ memoizedState: { memoizedState: 5 }  ← 独立状态
 ├─ Counter (Fiber2)
 │   └─ memoizedState: { memoizedState: 3 }  ← 独立状态
 └─ Counter (Fiber3)
     └─ memoizedState: { memoizedState: 7 }  ← 独立状态
```

**第四步：对比Class组件**
> "这和Class组件类似，虽然用的同一个class定义，但每个`<MyComponent />`都会`new MyComponent()`创建新实例，实例之间this.state互不影响。Fiber节点就相当于函数组件的'实例'。"

### 问题：Hook的状态存在哪里？为什么组件卸载后状态会丢失？

**回答结构（2分钟）**：

**第一步：明确存储位置**
> "Hook的状态存在Fiber节点的memoizedState链表上。Fiber是React的内部数据结构，每个组件对应一个Fiber节点。"

**第二步：说明完整链路**
> "完整的存储链路是：React维护一棵Fiber树 → 每个节点是一个Fiber对象 → Fiber对象有memoizedState属性 → memoizedState指向Hooks链表 → 链表的每个节点存储一个Hook的状态。"

```js
FiberRootNode (React根节点)
 └─ rootFiber
     └─ App Fiber
         ├─ memoizedState: null  // 函数组件如果没用Hook就是null
         └─ child → Counter Fiber
             └─ memoizedState → Hook1 → Hook2 → null
```

**第三步：解释卸载过程**
> "组件卸载时，React会从Fiber树中移除对应的Fiber节点。具体流程是：1) 执行useEffect的清理函数；2) 标记Fiber节点为deletion；3) commit阶段从DOM移除；4) Fiber节点失去引用，被JavaScript垃圾回收，链表上的Hooks状态也随之回收。"

```jsx
function Parent() {
  const [show, setShow] = useState(true);
  return (
    <div>
      {show && <Counter />}  {/* show变false时Counter卸载 */}
      <button onClick={() => setShow(!show)}>Toggle</button>
    </div>
  );
}

// show: true→false
// Counter的Fiber节点被移除
// memoizedState链表被垃圾回收
// 状态丢失

// show: false→true
// 创建新的Counter Fiber节点
// useState重新初始化
// 状态从0开始
```

**第四步：对比持久化方案**
> "如果需要状态在卸载后保留，有三种方案：1) 状态提升到父组件；2) 用Context或Redux等外部状态管理；3) 用localStorage等浏览器存储。"

**第五步：补充Fiber的生命周期**
> "Fiber节点在组件从创建到卸载的整个过程中都存在，只要组件在，状态就在。这也是为什么组件重新渲染（props变化、父组件渲染）时state不会丢失，因为Fiber节点还在。"
