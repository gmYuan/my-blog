# 为什么以Hooks为主

## 1. 层级关系图

```
React组件写法演变
         |
    ┌────┴────┐
  过去        现在
Class组件   函数组件+Hooks
    |           |
 复杂样板    简洁直观
    |           |
this/super  闭包捕获
    |           |
生命周期    useEffect
    |           |
难复用      自定义Hooks
```

## 2. 各自的职责

### 函数组件 + Hooks（当前主流）
- **职责**: 用函数 + Hooks管理状态和副作用
- **优势**: 无this、无super、逻辑复用简单
- **特点**: useState管理状态、useEffect处理副作用
- **类比**: 轻装上阵的快递员，只带必需工具

### Class组件（遗留代码）
- **职责**: 用类和生命周期方法管理组件
- **劣势**: this绑定、super样板、逻辑分散
- **特点**: constructor、componentDidMount等生命周期
- **类比**: 背着沉重装备的快递员，流程繁琐

## 3. 调用链路对比

### Hooks写法
```
1. 组件函数执行
   ↓
2. useState 返回 [state, setState]，通过闭包捕获
   ↓
3. useEffect 注册副作用，返回清理函数
   ↓
4. 事件触发，直接调用 handleClick（无需bind）
   ↓
5. setState 触发重新渲染
   ↓
6. 组件函数重新执行，useState 返回新state
```

### Class写法
```
1. new 组件实例
   ↓
2. constructor 里 super(props)，初始化 this.state
   ↓
3. 绑定 this: this.handleClick = this.handleClick.bind(this)
   ↓
4. componentDidMount 执行副作用
   ↓
5. 事件触发，调用绑定了this的 handleClick
   ↓
6. this.setState 触发重新渲染
   ↓
7. componentDidUpdate 执行
```

## 4. 为什么Hooks成为主流？

**原因1**: 降低心智负担 - 无需理解this、super、bind等ES6类的复杂概念
- 例子：Class组件事件要bind，或用箭头函数，新手容易出错；Hooks直接用闭包

**原因2**: 逻辑复用更简单 - 自定义Hooks比HOC、mixin更清晰
- 例子：多个组件需要监听窗口大小，用useWindowSize一行搞定，Class需要HOC嵌套

## 5. 类比锚定

想象两种快递员工作方式：
- **Class组件** 是传统快递员：需要打卡上班（constructor）、背工牌（super）、每次送货要确认身份（bind this）、按固定流程走（生命周期）
- **Hooks** 是现代灵活快递员：接单就干（函数执行）、按需选工具（useState/useEffect），逻辑打包复用（自定义Hooks）

## 6. 代码示例对比

### Hooks - 简洁直观
```jsx
import { useState, useEffect } from 'react';

function Counter() {
  const [count, setCount] = useState(0);
  
  useEffect(() => {
    document.title = `点击了 ${count} 次`;
    
    return () => {
      console.log('清理');
    };
  }, [count]);
  
  // 无需bind，直接用闭包
  const handleClick = () => {
    setCount(count + 1);
  };
  
  return <button onClick={handleClick}>{count}</button>;
}
```

### Class - 样板代码多
```jsx
import React, { Component } from 'react';

class Counter extends Component {
  constructor(props) {
    super(props);  // 必须调用，否则this.props是undefined
    this.state = { count: 0 };
    
    // 必须bind this，否则handleClick里的this是undefined
    this.handleClick = this.handleClick.bind(this);
  }
  
  componentDidMount() {
    document.title = `点击了 ${this.state.count} 次`;
  }
  
  componentDidUpdate() {
    document.title = `点击了 ${this.state.count} 次`;
  }
  
  componentWillUnmount() {
    console.log('清理');
  }
  
  handleClick() {
    this.setState({ count: this.state.count + 1 });
  }
  
  render() {
    return <button onClick={this.handleClick}>{this.state.count}</button>;
  }
}
```

### 为什么Class有this/super问题？

**super问题**：
- Class继承React.Component，构造函数必须先调用`super(props)`
- 否则父类的`this.props`赋值失败，导致`this.props === undefined`

**bind问题**：
- React事件是直接调用`callback(event)`，不是`obj.callback()`
- Class方法直接传给onClick会丢失this
- 解决：constructor里bind，或用箭头函数`handleClick = () => {}`

**Hooks为什么没这些问题**：
- 函数组件没有this，状态通过闭包捕获
- 事件直接`onClick={handleClick}`即可

### 自定义Hooks复用逻辑

```jsx
// 抽取窗口宽度逻辑
function useWindowWidth() {
  const [width, setWidth] = useState(window.innerWidth);
  
  useEffect(() => {
    const handleResize = () => setWidth(window.innerWidth);
    window.addEventListener('resize', handleResize);
    return () => window.removeEventListener('resize', handleResize);
  }, []);
  
  return width;
}

// 多个组件直接复用
function ComponentA() {
  const width = useWindowWidth();
  return <div>宽度: {width}</div>;
}

function ComponentB() {
  const width = useWindowWidth();
  return width > 768 ? <div>桌面端</div> : <div>移动端</div>;
}
```

## 7. 深入理解

### this绑定的底层原因

**JavaScript的this机制（非箭头函数）**

```jsx
class Counter {
  constructor() {
    this.count = 0;
  }
  
  increment() {
    console.log(this);  // this取决于调用方式
  }
}

const counter = new Counter();

// 方式1：作为对象方法调用，this指向对象
counter.increment();  // this = counter对象 ✓

// 方式2：单独调用函数，this丢失
const fn = counter.increment;
fn();  // this = undefined (严格模式) 或 window ✗

// React事件就是方式2
<button onClick={this.increment} />
// 等价于：onClick = this.increment; onClick();
// this丢失！
```

**Class组件的三种绑定方案对比**

```jsx
class Counter extends Component {
  constructor(props) {
    super(props);
    this.state = { count: 0 };
    
    // 方案1：constructor里bind（推荐）
    this.handleClick1 = this.handleClick1.bind(this);
  }
  
  // 方案1
  handleClick1() {
    this.setState({ count: this.state.count + 1 });
  }
  
  // 方案2：箭头函数class field（推荐）
  handleClick2 = () => {
    this.setState({ count: this.state.count + 1 });
  }
  
  // 方案3：render里bind（不推荐，每次渲染创建新函数）
  render() {
    return (
      <>
        <button onClick={this.handleClick1}>方式1</button>
        <button onClick={this.handleClick2}>方式2</button>
        <button onClick={this.handleClick1.bind(this)}>方式3</button>
        <button onClick={() => this.handleClick1()}>方式4</button>
      </>
    );
  }
}
```

**性能对比**：
- 方案1（constructor bind）：只创建一次，性能最好
- 方案2（箭头函数field）：每个实例一个函数，性能好
- 方案3/4（render里创建）：每次渲染创建新函数，子组件会不必要渲染

### super(props)的深层原理

**ES6 class继承的要求**

```jsx
class Parent {
  constructor(name) {
    this.name = name;
  }
}

class Child extends Parent {
  constructor(name, age) {
    // ❌ 错误：super之前不能使用this
    // this.age = age;  // ReferenceError
    
    // ✓ 必须先调用super
    super(name);
    
    // ✓ super之后才能使用this
    this.age = age;
  }
}
```

**React.Component的constructor做了什么**

```jsx
// React源码简化版
class Component {
  constructor(props) {
    this.props = props;  // 关键：给this赋值props
    this.state = null;
  }
}

// 你的组件
class MyComponent extends Component {
  constructor(props) {
    // 如果不调用super(props)，父类构造函数不执行
    // 父类的 this.props = props 就不会执行
    // 导致 constructor 里 this.props 是 undefined
    
    super(props);  // 执行父类构造，给this.props赋值
    
    console.log(this.props);  // 现在可以访问了
  }
}
```

**super()和super(props)的区别**

```jsx
class MyComponent extends Component {
  constructor(props) {
    super();  // 只调用super()，不传props
    
    // ❌ constructor里this.props是undefined
    console.log(this.props);  // undefined
    console.log(props);       // {name: '张三'} ✓
  }
  
  componentDidMount() {
    // ✓ 生命周期里this.props正常（React内部会赋值）
    console.log(this.props);  // {name: '张三'} ✓
  }
  
  render() {
    // ✓ render里this.props正常
    return <div>{this.props.name}</div>;  // ✓
  }
}
```

**结论**：不传props只影响constructor里的this.props，其他地方React会自动赋值

### Hooks的闭包机制

**为什么Hooks不需要this？**

```jsx
function Counter() {
  const [count, setCount] = useState(0);
  
  // handleClick是普通函数，通过闭包捕获count
  const handleClick = () => {
    // count来自闭包，不是this.count
    setCount(count + 1);
  };
  
  // 每次渲染，整个函数重新执行
  // count是新的，handleClick也是新的（捕获新count）
  return <button onClick={handleClick}>{count}</button>;
}
```

**闭包陷阱：异步中的旧状态**

```jsx
function Counter() {
  const [count, setCount] = useState(0);
  
  const handleClick = () => {
    setCount(count + 1);
    
    // ❌ 1秒后打印的是旧的count
    setTimeout(() => {
      console.log(count);  // 闭包捕获的是当时的count
    }, 1000);
  };
  
  return <button onClick={handleClick}>{count}</button>;
}

// 解决方案1：用ref
function Counter() {
  const [count, setCount] = useState(0);
  const countRef = useRef(count);
  
  useEffect(() => {
    countRef.current = count;  // 每次更新ref
  }, [count]);
  
  const handleClick = () => {
    setCount(count + 1);
    setTimeout(() => {
      console.log(countRef.current);  // ✓ 总是最新值
    }, 1000);
  };
  
  return <button onClick={handleClick}>{count}</button>;
}

// 解决方案2：函数式更新
function Counter() {
  const [count, setCount] = useState(0);
  
  const handleClick = () => {
    setCount(c => c + 1);  // c总是最新值
    
    setCount(c => {
      setTimeout(() => {
        console.log(c);  // ✓ 获取当时的最新值
      }, 1000);
      return c;
    });
  };
  
  return <button onClick={handleClick}>{count}</button>;
}
```

### 生命周期与useEffect的对应关系

| Class生命周期 | Hooks等价 | 说明 |
|-------------|-----------|------|
| constructor | useState/useRef | 初始化状态 |
| componentDidMount | useEffect(fn, []) | 依赖为空，只执行一次 |
| componentDidUpdate | useEffect(fn, [deps]) | 依赖变化时执行 |
| componentWillUnmount | useEffect返回函数 | 清理函数 |
| shouldComponentUpdate | React.memo | 优化渲染 |
| getDerivedStateFromProps | 渲染时计算 | 无需单独生命周期 |

**复杂生命周期的Hooks实现**

```jsx
// Class组件：逻辑分散在多个生命周期
class Example extends Component {
  componentDidMount() {
    // 订阅1
    ChatAPI.subscribeToFriend(this.props.friendID, this.handleStatusChange);
    // 订阅2
    window.addEventListener('resize', this.handleResize);
  }
  
  componentDidUpdate(prevProps) {
    // 取消旧订阅
    if (prevProps.friendID !== this.props.friendID) {
      ChatAPI.unsubscribeFromFriend(prevProps.friendID, this.handleStatusChange);
      ChatAPI.subscribeToFriend(this.props.friendID, this.handleStatusChange);
    }
  }
  
  componentWillUnmount() {
    // 清理1
    ChatAPI.unsubscribeFromFriend(this.props.friendID, this.handleStatusChange);
    // 清理2
    window.removeEventListener('resize', this.handleResize);
  }
}

// Hooks：相关逻辑聚在一起
function Example({ friendID }) {
  // 逻辑1：订阅好友状态
  useEffect(() => {
    ChatAPI.subscribeToFriend(friendID, handleStatusChange);
    return () => ChatAPI.unsubscribeFromFriend(friendID, handleStatusChange);
  }, [friendID]);
  
  // 逻辑2：监听窗口变化
  useEffect(() => {
    window.addEventListener('resize', handleResize);
    return () => window.removeEventListener('resize', handleResize);
  }, []);
}
```

### 为什么React团队推崇Hooks？

**官方动机（React Conf 2018）**：

1. **难以复用状态逻辑**
   - HOC和render props导致"嵌套地狱"
   - 自定义Hooks可以扁平化复用

2. **复杂组件难以理解**
   - 生命周期分散相关逻辑
   - useEffect按功能聚合代码

3. **Class让人和机器都困惑**
   - this绑定让新手困惑
   - 压缩工具难以优化class（方法名不能压缩）

4. **未来优化方向**
   - Hooks更容易做编译优化（React Forget）
   - 函数组件更适合并发模式（Concurrent Mode）

## 8. 一句话总结

Hooks用闭包替代this和super，用useEffect替代生命周期，用自定义Hook扁平化复用逻辑，降低心智负担且更易优化。

## 9. 面试回答路径

### 问题：为什么React推荐用Hooks而不是Class？

**回答结构（2分钟）**：

**第一步：概括优势**
> "主要有三个原因：降低心智负担、便于逻辑复用、代码更简洁。"

**第二步：逐个解释**

**降低心智负担**
> "Class组件需要理解this绑定、super调用等ES6 class概念，新手容易出错。比如事件回调要bind this，否则this会丢失。Hooks用闭包捕获状态，不需要this。"

```jsx
// Class：要bind
constructor() {
  this.handleClick = this.handleClick.bind(this);
}

// Hooks：直接用
const handleClick = () => {...};
```

**便于逻辑复用**
> "Class组件复用逻辑要用HOC或render props，容易形成嵌套地狱。Hooks可以抽成自定义Hook，一行代码搞定。"

```jsx
// Class：嵌套
withRouter(withAuth(withTheme(MyComponent)))

// Hooks：扁平
const user = useAuth();
const theme = useTheme();
```

**代码更简洁**
> "Class组件需要constructor、super、生命周期方法，样板代码多。Hooks用useState、useEffect几行代码就够了。"

**第三步：补充深层原因（显示深度）**
> "从React团队的角度，Hooks还有两个优势：一是更容易做编译优化，函数组件可以被压缩得更小；二是更适合未来的并发模式，函数组件可以被中断和恢复。"

**第四步：承认Class的位置**
> "当然Class组件还在维护，遗留项目可以继续用，但新项目官方推荐Hooks。"

### 问题：为什么Class组件的事件回调要bind this？

**回答结构（90秒）**：

**第一步：说明现象**
> "如果不bind，事件回调执行时this会是undefined，导致报错无法读取this.state或this.props。"

**第二步：解释原因**
> "这是JavaScript的this机制决定的。React事件调用函数的方式是`callback(event)`，不是`obj.callback(event)`，所以this会丢失。"

```jsx
// React内部类似这样调用
const callback = this.handleClick;  // 赋值给变量
callback(event);  // 单独调用，this丢失
```

**第三步：对比Hooks**
> "函数组件没有this，事件回调通过闭包捕获外层的变量，不存在this丢失问题。"

```jsx
function Component() {
  const [count, setCount] = useState(0);
  
  // 通过闭包捕获count和setCount
  const handleClick = () => {
    setCount(count + 1);
  };
  
  return <button onClick={handleClick}>+1</button>;
}
```

**第四步：说明解决方案**
> "Class组件有三种方案：constructor里bind、箭头函数class field、render里箭头函数。前两种性能好，第三种每次渲染创建新函数不推荐。"

---

### 问题：为什么必须调用super(props)？

**回答结构（90秒）**：

**第一步：说明ES6要求**
> "这是ES6 class继承的语法要求，子类constructor必须先调用super()执行父类构造函数，才能使用this。"

**第二步：解释为什么要传props**
> "React.Component的constructor会执行`this.props = props`，如果不传props，父类拿不到props就无法赋值，导致constructor里this.props是undefined。"

```jsx
class Component {
  constructor(props) {
    this.props = props;  // 父类要赋值
  }
}

class MyComponent extends Component {
  constructor(props) {
    super();  // 没传props，父类的this.props赋值失败
    console.log(this.props);  // undefined ✗
    console.log(props);       // ✓ 参数props正常
  }
}
```

**第三步：说明影响范围**
> "只影响constructor里的this.props，其他生命周期和render里React会自动赋值，所以this.props正常。但为了规范，一般都传props。"

**第四步：对比Hooks**
> "函数组件props直接作为参数传入，不需要super，这也是Hooks更简单的原因之一。"

```jsx
function MyComponent(props) {
  // 直接使用props，没有this和super的概念
  return <div>{props.name}</div>;
}
```
