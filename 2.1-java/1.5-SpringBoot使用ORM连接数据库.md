# 实战：Spring Boot 使用 ORM / MyBatis 连接数据库

> 面向后端新手。这节课不是上一份"从零创建项目"的重复，而是新的一条主线：**应用跑起来之后，怎么接上 MySQL 数据库，把原来写死的假数据，改成真正从数据库里查出来。** 核心路径：Spring Boot + Docker MySQL + MyBatis Mapper + `application.properties` 数据源配置。

---

### 请求到数据库的完整链路

```
浏览器
  │  GET /
  ▼
HelloController
  │  userService.getUserById(1)
  ▼
UserService
  │  userMapper.findUserById(id)
  ▼
UserMapper（MyBatis 接口）
  │  执行 SELECT * FROM user WHERE id = #{id}
  ▼
MySQL（Docker 容器内 3306）
  │  返回一行数据
  ▼
MyBatis 自动映射成 User 对象
  ▼
返回给浏览器（JSON）
```

### 技术栈关键词速查

| 关键词 | 一句话 |
|--------|--------|
| **JDBC** | Java 连接数据库的底层标准接口，所有 ORM 框架的底层 |
| **ORM** | 对象关系映射，让 Java 对象和数据库表互相对应 |
| **MyBatis** | 你写 SQL，框架帮你执行并把结果映射成 Java 对象 |
| **JPA** | 另一种 ORM 风格，更面向对象，几乎不手写 SQL |
| **`mybatis-spring-boot-starter`** | 让 Spring Boot 整合 MyBatis |
| **`mysql-connector-java`** | MySQL 的 JDBC 驱动，Java 连接 MySQL 必须有它 |
| **`application.properties`** | 告诉 Spring Boot 数据库地址、账号、密码 |
| **`@Mapper`** | 声明这是一个 MyBatis Mapper 接口 |
| **`@Select`** | 在 Mapper 接口方法上直接写 SQL |
| **`@Param`** | 把方法参数和 SQL 里的 `#{}` 占位符对应起来 |
| **`-p 3306:3306`** | Docker 把宿主机 3306 端口流量转发到容器内 3306 |

### 各层分工一览

```
Spring Boot / Java 应用
         │
    MyBatis（ORM 层）
         │  执行 SQL，结果 → Java 对象
    MySQL JDBC Driver
         │  底层网络通信
    MySQL（在 Docker 容器里运行）
```

## 第一部分：实战步骤（可直接照做）

### 步骤 1：用 Docker 把 MySQL 跑起来

```bash
# 1. 拉取 MySQL 镜像
docker pull mysql

# 2. 运行 MySQL 容器
docker run --name my-mysql \
  -e MYSQL_ROOT_PASSWORD=my-secret-pw \
  -e MYSQL_DATABASE=xdml \
  -p 3306:3306 \
  -d mysql
```

这条命令做了 4 件事：

| 参数 | 作用 |
|------|------|
| `--name my-mysql` | 给容器起名字（方便后续管理） |
| `-e MYSQL_ROOT_PASSWORD=my-secret-pw` | 设置 root 密码 |
| `-e MYSQL_DATABASE=xdml` | 启动时自动创建名为 `xdml` 的数据库 |
| `-p 3306:3306` | 把宿主机 3306 映射到容器内 3306 |
| `-d` | 后台运行（detached 模式） |

> **如果起错了容器，怎么清理：**
> ```bash
> docker ps                    # 查看正在运行的容器
> docker rm -f my-mysql        # 强制删除并重来
> ```

---

### 步骤 2：在 IDEA 里先验证数据库是否通了

打开 IDEA → Database / DataSource 面板，新增 MySQL 数据源：

| 配置项 | 值 |
|--------|---|
| Host | `localhost` |
| Port | `3306` |
| User | `root` |
| Password | `my-secret-pw` |
| Database | `xdml` |
| URL | `jdbc:mysql://localhost:3306/xdml` |

点击 `Test Connection`，看到绿色勾 ✓ 才算通了。

> **为什么要先在 IDEA 里验证？** 这是实用排错习惯：先确认"数据库本身能连接"，再写 Java 代码。不然后续代码报错，你不知道是数据库问题还是代码问题。

### 步骤 3：建表 + 插测试数据

在 IDEA 的数据库控制台（或任何 MySQL 客户端）执行：

```sql
-- 建表
CREATE TABLE user (
    id   INT,
    name VARCHAR(10)
);

-- 插测试数据
INSERT INTO user (id, name) VALUES (1, '张三');
```

### 步骤 4：给 `pom.xml` 加数据库相关依赖

在原有依赖基础上，新增：

```xml
<dependencies>
    <dependency>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-starter-web</artifactId>
    </dependency>

    <dependency>
        <groupId>javax.inject</groupId>
        <artifactId>javax.inject</artifactId>
        <version>1</version>
    </dependency>

    <!-- MyBatis + Spring Boot 整合 -->
    <dependency>
        <groupId>org.mybatis.spring.boot</groupId>
        <artifactId>mybatis-spring-boot-starter</artifactId>
        <version>2.0.1</version>
    </dependency>

    <!-- MySQL JDBC 驱动 -->
    <dependency>
        <groupId>mysql</groupId>
        <artifactId>mysql-connector-java</artifactId>
        <version>8.0.15</version>
    </dependency>
</dependencies>
```

> **两个依赖的分工：**
> - `mybatis-spring-boot-starter`：让 Spring Boot 能整合 MyBatis，自动扫描 `@Mapper` 接口
> - `mysql-connector-java`：真正的 MySQL JDBC 驱动，没它 Java 根本连不上 MySQL

---

### 步骤 5：写 `application.properties`

文件路径：`src/main/resources/application.properties`

```properties
spring.datasource.url=jdbc:mysql://localhost:3306/xdml
spring.datasource.username=root
spring.datasource.password=my-secret-pw
spring.datasource.driver-class-name=com.mysql.cj.jdbc.Driver
```

> **注意点：**
> - `localhost:3306`：对应 `docker run -p 3306:3306`（宿主机端口被 Docker 转发到容器内 MySQL）
> - `/xdml`：对应 `docker run -e MYSQL_DATABASE=xdml`（创建的数据库名）
> - `driver-class-name`：告诉 Spring Boot 用 MySQL 的驱动类来连接

---

### 步骤 6：写 `UserMapper`（数据访问层）

文件路径：`src/main/java/hello/mapper/UserMapper.java`

```java
package hello.mapper;

import hello.service.User;
import org.apache.ibatis.annotations.Mapper;
import org.apache.ibatis.annotations.Param;
import org.apache.ibatis.annotations.Select;

@Mapper
public interface UserMapper {

    @Select("SELECT * FROM user WHERE id = #{id}")
    User findUserById(@Param("id") Integer id);
}
```

| 代码 | 作用 |
|------|------|
| `@Mapper` | 告诉 Spring / MyBatis：这是一个 Mapper 接口（自动注册为 Bean） |
| `@Select(...)` | 这里写真正要执行的 SQL |
| `#{id}` | SQL 里的占位符，对应方法参数 `id` |
| `@Param("id")` | 把方法参数 `id` 绑定到 SQL 里的 `#{id}` |
| 返回 `User` | 查询结果自动映射成 `User` 对象（字段名要对应） |

> **为什么 `User` 能接住 SQL 结果？**
> 数据库表有 `id`、`name` 两列，`User` 类也有 `id`、`name` 两个字段，字段名一致时 MyBatis 自动映射。

---

### 步骤 7：改 `UserService`（从假数据改成真查库）

**改之前（假数据版）：**

```java
package hello.service;

public class UserService {
    public User getUserById(Integer id) {
        return new User(id, "name1");  // 写死的假数据
    }
}
```

**改之后（数据库版）：**

```java
package hello.service;

import hello.mapper.UserMapper;
import javax.inject.Inject;

public class UserService {
    private UserMapper userMapper;

    @Inject
    public UserService(UserMapper userMapper) {
        this.userMapper = userMapper;  // 依赖 UserMapper
    }

    public User getUserById(Integer id) {
        return userMapper.findUserById(id);  // 真正去数据库查
    }
}
```

**分工变化：**
- `UserService`：业务逻辑层，不自己造数据，把"查库"这件事委托给 `UserMapper`
- `UserMapper`：数据访问层，专门负责数据库查询

---

### 步骤 8：同步更新 `JavaConfiguration`（因为 UserService 依赖变了）

`UserService` 现在需要 `UserMapper`，如果你用的是 Java Config 方式，配置类也要更新：

**改之前：**

```java
@Bean
public UserService userService() {
    return new UserService();  // 编译报错：UserService 没有无参构造器了
}
```

**改之后：**

```java
package hello.configuration;

import hello.mapper.UserMapper;
import hello.service.OrderService;
import hello.service.UserService;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;

@Configuration
public class JavaConfiguration {

    @Bean
    public UserService userService(UserMapper userMapper) {
        // Spring 自动把容器里的 UserMapper Bean 传进来
        return new UserService(userMapper);
    }

    @Bean
    public OrderService orderService(UserService userService) {
        return new OrderService(userService);
    }
}
```

> **注意：** `UserMapper` 是由 `@Mapper` 注解 + MyBatis 自动注册到容器里的 Bean，所以可以直接在 `@Bean` 方法参数里用。

---

### 步骤 9：改 `HelloController`（返回真实用户对象）

```java
package hello;

import hello.service.User;
import hello.service.UserService;
import org.springframework.web.bind.annotation.RequestMapping;
import org.springframework.web.bind.annotation.RestController;
import javax.inject.Inject;

@RestController
public class HelloController {
    private UserService userService;

    @Inject
    public HelloController(UserService userService) {
        this.userService = userService;
    }

    @RequestMapping("/")
    public User index() {
        return this.userService.getUserById(1);  // 从数据库里取真实数据
    }
}
```

运行后，浏览器访问 `http://localhost:8080/`，看到 JSON：

```json
{"id": 1, "name": "张三"}
```

## 第二部分：核心知识点（面试 / 工作常问）

### 知识点 1：ORM、JPA、MyBatis 的关系

| 名词 | 含义 |
|------|------|
| **ORM（Object-Relational Mapping）** | 概念/思想：让 Java 对象和数据库表互相对应，省去手动写 ResultSet 转换 |
| **JPA** | ORM 的一套规范（接口），常见实现是 Hibernate；特点是面向对象，几乎不写 SQL |
| **MyBatis** | ORM 框架之一；特点是**你写 SQL**，框架帮你执行并映射成 Java 对象 |

**这节课落地的是 MyBatis，记住一句话：**
> MyBatis = 我写 SQL，框架帮我执行并把结果变成 Java 对象

---

### 知识点 2：Spring Boot、MySQL、JDBC、MyBatis 四者分工

```
Spring Boot（Web 框架 + 自动配置）
     ↓ 通过 application.properties 配置数据源
MyBatis（ORM 层）
     ↓ 解析 @Select 注解，生成 PreparedStatement
MySQL JDBC Driver（mysql-connector-java）
     ↓ 底层网络通信协议
MySQL 数据库（Docker 容器里运行）
```

不是 Spring Boot 直接"魔法连数据库"，而是：
- Spring Boot 帮你**配数据源**
- MyBatis 帮你**写查询映射**
- 底层最终还是 **JDBC 在和 MySQL 通信**

---

### 知识点 3：Docker MySQL vs 本机装 MySQL —— 为什么推荐 Docker

| 对比 | 本机装 MySQL | Docker 跑 MySQL |
|------|------------|----------------|
| 环境隔离 | 污染本机环境 | 完全隔离在容器里 |
| 版本管理 | 装了难换版本 | 换版本 = 换镜像，秒级完成 |
| 团队一致性 | 各自本机可能版本不一 | 同一个镜像，版本完全一致 |
| 清理 | 卸载麻烦 | `docker rm -f` 秒删 |

**一句话：** Docker 让"数据库环境"变成一个可随时创建、删除的独立盒子。

---

### 知识点 4：MyBatis `@Select` 注解 vs XML Mapper 文件

本课使用的是**注解版 MyBatis**（SQL 直接写在接口方法上）：

```java
@Select("SELECT * FROM user WHERE id = #{id}")
User findUserById(@Param("id") Integer id);
```

还有一种**XML Mapper 文件版**（SQL 写在 XML 里）：

```xml
<mapper namespace="hello.mapper.UserMapper">
    <select id="findUserById" resultType="hello.service.User">
        SELECT * FROM user WHERE id = #{id}
    </select>
</mapper>
```

| | 注解版 | XML 版 |
|--|-------|--------|
| SQL 写在哪 | 接口方法上 | 单独的 .xml 文件 |
| 适合场景 | SQL 简单，快速开发 | 复杂 SQL，动态 SQL |
| 工作中常见度 | ★★ | ★★★（复杂项目更常见） |

---

### 知识点 5：`application.properties` vs `application.yml`

两种格式都是 Spring Boot 的配置文件，功能完全一样，格式不同：

**`application.properties`（本课使用）：**
```properties
spring.datasource.url=jdbc:mysql://localhost:3306/xdml
spring.datasource.username=root
```

**`application.yml`（更常见于现代项目）：**
```yaml
spring:
  datasource:
    url: jdbc:mysql://localhost:3306/xdml
    username: root
```

YAML 格式通过缩进表达层级，更清晰，现代项目更多用 `.yml`。两者放在 `src/main/resources/` 下都会被 Spring Boot 自动读取。

## 第三部分：学习 Q&A

### Q1：`@Mapper` 和 `@Repository` 有什么区别？

**A：**

| 注解 | 来源 | 作用 |
|------|------|------|
| `@Mapper` | MyBatis 提供 | 声明 MyBatis Mapper 接口，让 MyBatis 自动扫描并生成代理实现 |
| `@Repository` | Spring 提供 | 声明数据访问层 Bean，同时启用 Spring 的异常转换（将数据库异常转换为 Spring 的 `DataAccessException`） |

两者有时会一起用：

```java
@Mapper
@Repository
public interface UserMapper { ... }
```

只用 `@Mapper` 也能正常工作，`@Repository` 带来的额外好处是异常处理更统一。

---

### Q2：`#{id}` 和 `${id}` 有什么区别？

**A：** 这是 MyBatis 里很重要的安全问题：

| | `#{id}` | `${id}` |
|--|---------|---------|
| 处理方式 | 预编译（PreparedStatement），**防 SQL 注入** | 字符串拼接，**不安全** |
| 适用场景 | 参数值（WHERE 条件、INSERT 值） | 表名、列名等不能预编译的情况 |

**工作中原则：** 永远优先用 `#{}`，只有在必须动态拼表名/列名时才用 `${}`，且要自己做好校验。

---

### Q3：为什么浏览器访问控制器，`User` 对象会自动变成 JSON？

**A：** 因为 `spring-boot-starter-web` 里包含了 **Jackson** 库，它会：

1. 看到方法返回值是 Java 对象
2. 调用对象的所有 `get` 方法（如 `getId()`、`getName()`）
3. 把属性名和值组成 JSON 对象

所以：
- `User` 类里必须有 `getter` 方法，Jackson 才能读到字段值
- 字段即使是 `private`，只要有 `getter`，JSON 里就有这个字段

---

### Q4：数据库里是 `name` 列，Java 里是 `username` 字段，MyBatis 会报错吗？

**A：** 会导致映射失败（该字段为 `null`）。

解决方案：

**方案一：用 SQL 别名**
```java
@Select("SELECT id, name AS username FROM user WHERE id = #{id}")
```

**方案二：用 `@Results` 手动映射**
```java
@Results({
    @Result(column = "name", property = "username")
})
@Select("SELECT * FROM user WHERE id = #{id}")
User findUserById(@Param("id") Integer id);
```

**方案三：在 `application.properties` 开启下划线转驼峰（仅适用于 `user_name` → `userName` 这种规律）**
```properties
mybatis.configuration.map-underscore-to-camel-case=true
```

---

### Q5：`java.sql.SQLException: Access denied` 报错，是什么问题？

**A：** 这是数据库连接权限问题，常见原因：

1. **用户名或密码写错** → 检查 `application.properties` 里的 `username` 和 `password`
2. **数据库没起来** → `docker ps` 确认容器在运行
3. **端口映射不对** → 确认 `docker run -p 3306:3306`（不是其他端口）
4. **MySQL root 用户不允许远程连接** → Docker 镜像通常已经处理好了，如果是本机装的 MySQL 可能需要额外配置

**排查顺序：** 先用 IDEA DataSource 连接测试（步骤 2），把"数据库本身是否可以连接"单独验证掉。
