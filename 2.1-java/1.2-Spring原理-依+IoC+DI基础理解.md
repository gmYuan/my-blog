# Spring 原理：依赖、Bean、容器、IoC、DI — 初学者导读

> 面向后端新手。这节课没有具体代码，全是白板概念图。核心只有 5 个词，但这 5 个词是后面所有 Spring 学习的地基：**依赖、Bean、容器、IoC、DI**。

---

## 开篇：一图看懂全局

### 这 5 个词怎么串成一条线

```
现实里"依赖"一定存在
（OrderService 要用 UserService 查用户）
           │
           │ 对象一多，不能每个类里都手写 new
           ▼
    把对象交给 Spring 登记成 Bean
           │
           ▼
      由"容器"统一创建 + 保管 + 接线
           │
           ▼
   IoC：控制权交给框架，不再由业务类到处 new
           │
           ▼
   DI：容器把依赖"注入"进组件，组件不自己 new
```

### 5 个词速查表

| 概念 | 它在解决什么问题 | 一句话 |
|------|-----------------|--------|
| **依赖** | A 干活离不开 B | A 要用 B，就说 A 依赖 B |
| **Bean** | 让 Spring 认识这个对象 | 交给 Spring 创建和管理的对象 |
| **容器** | 统一管理所有 Bean | 造对象 + 存对象 + 按依赖关系把对象接好 |
| **IoC** | 谁来负责创建和组装 | 控制权从业务代码交给框架 |
| **DI** | 依赖怎么给到组件 | 容器把依赖"打进来"，不是组件自己 `new` |

### 截图各自讨论什么

| 截图顺序 | 讨论主题 |
|---------|---------|
| 1 | **依赖**：什么叫「A 依赖 B」 |
| 2 | **Bean + 容器**：多个 Bean 的依赖关系，容器容纳这一切 |
| 3 | **Spring IoC 容器**：框里的都是 Bean，箭头是依赖关系 |
| 4 | **IoC**：没有容器时依赖乱成一团；有容器时由配置 + 容器负责装配 |
| 5 | **DI**：依赖不是 A 自己造 B，而是容器把 B 注入给 A |

## 第二部分：核心知识点

### 知识点 1：依赖 —— 先认清「关系」，不谈框架

**是什么：**
「**当 A 必须使用 B，才能完成自己的工作**」—— 就说 **A 依赖 B**。箭头写成 **A → B**（A 需要 B）。

**和继承不一样：**
依赖是"我要用你提供的能力"（组合、协作），不是"我是你的一种"。

**生活比喻：**
你做 PPT（A），必须用同事整理的表格（B）。**能不能做 PPT，取决于你能不能拿到那份表** —— 这就是依赖：A 的工作链条里硬性地需要 B。

**代码直觉（还不涉及 Spring）：**
`OrderService` 里要调用 `UserService` 查用户，才能完成下单校验 —— 订单服务依赖用户服务。这种关系客观存在，Spring 只是帮你换了一种满足这种关系的方式。

---

### 知识点 2：Bean —— Spring 管的对象，有「户口」的那种

**是什么：**
在 Spring 里，**由容器创建、配置、并纳入生命周期管理** 的对象，叫 **Bean**。

**和「随便 `new` 出来的对象」差在哪：**

| 方式 | Spring 知道它的存在吗 | 能自动注入其他 Bean 吗 |
|------|--------------------|--------------------|
| 自己 `new` | 不知道 | 不能 |
| 声明成 Bean | 知道 | 能（由容器接线） |

**生活比喻：**
Bean 像**公司通讯录里的正式员工**：有档案、有工位、人事知道谁和谁要配合。
自己 `new` 的像**临时叫来的外援**：公司系统里没登记，别的部门不知道怎么对接。

---

### 知识点 3：Spring 容器 —— 造对象 + 保管 + 接线

**是什么：**
一个**运行时的大管家**（实现上对应 `ApplicationContext` 这类接口）：
- **造**：按配置/注解创建 Bean
- **存**：保管这些实例（常见单例策略）
- **接**：知道谁依赖谁，创建时把依赖装配好

**生活比喻：**
像**中央厨房 + 出餐口**：菜谱和分工定好以后，厨房按单配菜、装盘；你到窗口领一份已经搭配好的套餐，而不是自己在座位上手抓每种原料现组合。

**容器怎么知道「谁依赖谁」？**
是你告诉它的 —— 通过**元数据**（配置或注解在类上留下的信息）：

| 方式 | 你在表达什么 |
|------|------------|
| 注解（`@Component`、`@Autowired` 等） | 在类上声明"这是 Bean"，在注入点声明"这里要注入" |
| Java Config（`@Configuration` + `@Bean`） | 用 Java 方法手动创建 Bean 并表达依赖 |
| XML（`<bean>`、`<constructor-arg>`） | 在配置文件里手写 Bean 和依赖关系（老项目常见） |

**容器启动时大致做什么（直觉版）：**
收集所有 Bean 定义 → 看每个 Bean 的构造参数/注入点 → 和容器里已有的 Bean 做类型或名称匹配 → 按依赖顺序创建（先创建被依赖方，再注入到依赖方）。

---

### 知识点 4：IoC（控制反转）—— 谁说了算

**是什么：**
**Inversion of Control**：控制对象创建与组装的权力，从业务代码手里，**反转**到框架/容器手里。

| | 传统写法 | Spring IoC |
|--|---------|-----------|
| **谁创建对象** | 你的业务类里 `new` | 容器 |
| **谁组装依赖** | 你的业务类里手动 set | 容器 |
| **控制权在哪** | 业务代码 | 框架 |

**为什么叫「反转」：**
传统写法里，在你的类内部你决定 `new` 谁、怎么连 —— 控制权在"我"。用 Spring 后，创建和组装由容器驱动 —— 控制权在"容器"，所以叫反转。

**生活比喻：**
- **没有 IoC**：自己种菜、洗菜、炒菜，每一步控制权都在自己手上
- **有 IoC**：你只点单（配置/注解），厨房按单出餐，炒菜流程的控制权在厨房

---

### 知识点 5：DI（依赖注入）—— 「塞进来」比「自己造」更直观

**是什么：**
**Dependency Injection**：组件**不自己负责创建依赖对象**，而是由**外部（容器）** 把已经准备好的依赖**传入**组件内部（构造方法、Setter、字段等途径）。

**和 IoC 的关系（初学者记法）：**
- **IoC** 偏**思想/分工**：控制权交给容器
- **DI** 偏**手段**：容器通过注入把 B 交给 A，实现这种分工

常说：**DI 是实现 IoC 的一种典型方式**。

**课程里注射器图在表达什么：**
A 需要 B 这种依赖关系不变；变的是 B 从哪来 —— 不是 A 肚子里 `new B()`，而是容器像打针一样把 B 打进 A。

**代码直觉（后面学注解时会见到）：**
构造器参数、`@Autowired` 等，本质都是：**留出"口子"让容器把依赖灌进来**，而不是在方法里写死 `new 具体实现类()`。

---

### 知识点 6：要理解 Spring 装配 / 容器原理，要先掌握什么

可以分两层，不必一开始全部啃透：

**第一层 —— 先把 Spring「用明白」（优先学）：**
- Java 基础：类与对象、接口与实现、构造方法；知道依赖在代码里长什么样
- 注解（会用即可）：知道 `@Component`、`@Service`、`@Autowired`、`@SpringBootApplication` 等，理解「声明了 Bean」和「声明了注入点」的区别

到这一层，你已经能回答：谁依赖谁是靠配置/注解告诉容器的，原理上不必先深入反射。

**第二层 —— 读源码、抠「容器到底怎么干的」（按需深入）：**
- 反射：容器大量通过反射查看构造器、参数类型、字段、方法，并创建实例、设字段
- 设计层面：工厂/策略的直觉（`BeanFactory`）、代理（AOP、事务）、单例与作用域

**正确学习顺序：** 先会用注解把 Bean 和依赖写清楚 → 再按需学反射与容器扩展点，用来解释"为什么这样标一下就能注入"。

## 第三部分：学习 Q&A

### Q1：依赖、组合、继承，有什么区别？

**A：** 三种完全不同的关系：

| 关系 | 含义 | 代码形式 | 举例 |
|------|------|---------|------|
| **继承** | A 是 B 的一种 | `class A extends B` | `Cat extends Animal` |
| **组合** | A 内部持有 B | `class A { B b; }` | `Car` 里有 `Engine` |
| **依赖** | A 用到 B 但不持有 | 方法参数里有 B | `OrderService.placeOrder(UserService us)` |

Spring 里说的"依赖注入"，通常是指**组合**形式的依赖（A 内部持有 B），由容器来创建 B 并传给 A。

---

### Q2：Bean 一定是单例吗？

**A：** 默认是单例（Singleton）—— 容器里只有一个实例，所有地方注入的都是同一个对象。

Spring 支持多种作用域（Scope）：

| Scope | 含义 |
|-------|------|
| `singleton`（默认） | 全局只有一个实例 |
| `prototype` | 每次注入都创建新实例 |
| `request` | Web 场景，每个 HTTP 请求一个实例 |
| `session` | Web 场景，每个 Session 一个实例 |

初学阶段记住默认是**单例**就够了。

---

### Q3：`@Autowired` 和 `@Inject` 有什么区别？

**A：** 功能基本相同，来源不同：

| 注解 | 来源 | 说明 |
|------|------|------|
| `@Autowired` | Spring 自带 | Spring 最常用的注入注解，按**类型**匹配 |
| `@Inject` | JSR-330 标准（`javax.inject`） | Java 标准，需要额外引入依赖，与 Spring 无关的标准注解 |
| `@Resource` | JSR-250 标准（`javax.annotation`） | 按**名称**匹配（默认），也可按类型 |

对初学者来说，三者共同点更重要：**都是"告诉容器帮我注入依赖"**。工作中最常见的是 `@Autowired`。

---

### Q4：容器是什么时候启动的？Bean 是什么时候创建的？

**A：**

- **容器启动时机：** Spring Boot 应用执行 `SpringApplication.run(...)` 时，容器随之启动
- **Bean 创建时机（默认）：** 容器启动时就把所有单例 Bean 都创建好（称为"饿汉式"），不是等到第一次用的时候才创建

这就是为什么 Spring Boot 启动时日志里能看到很多 Bean 初始化的信息。

---

### Q5：一句话总结 IoC 和 DI 的区别

**A：**
- **IoC（控制反转）**：是一种**设计思想**，描述"谁来负责对象的创建和组装"这件事 → 答案是框架/容器
- **DI（依赖注入）**：是 IoC 的一种**具体实现方式**，描述容器如何把依赖传给组件 → 方式是"注入"（构造器/字段/Setter）

**比喻：** IoC 是"点外卖"这种思想（不自己做饭）；DI 是"外卖员把饭送到门口"这个具体动作。
