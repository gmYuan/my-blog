## 第一部分：Spring 依赖注入（DI）

### Q1：容器里有多个同类型 Bean，Spring 如何选择？

#### 问题场景

**真实工作场景：多数据源**

```java
// 你的项目需要连接 2 个数据库
@Configuration
public class DataSourceConfig {
    
    @Bean
    public DataSource mysqlDataSource() {
        // 连接 MySQL
    }
    
    @Bean
    public DataSource postgresDataSource() {
        // 连接 PostgreSQL
    }
}

// 在 Service 里注入时：
@Service
public class UserService {
    
    @Inject
    private DataSource dataSource;  // ← Spring 该注入哪个？💥
}
```

**Spring 会报错：**
```
No qualifying bean of type 'DataSource' available: 
expected single matching bean but found 2: mysqlDataSource, postgresDataSource
```

**翻译：** 我找到 2 个 `DataSource`，你要哪个？我不知道，报错！

---

#### Spring 的选择规则（重要！）

**规则 1：先按类型匹配**
```
Spring 容器：找所有 DataSource 类型的 Bean
结果：找到 2 个 ["mysqlDataSource", "postgresDataSource"]
```

**规则 2：如果有多个，按参数名匹配 Bean 名字**
```java
@Inject
private DataSource mysqlDataSource;  // ← 参数名
```

Spring 容器：
```
① 按类型找：找到 2 个
② 按参数名匹配：
   └─ 参数名 "mysqlDataSource" == Bean 名字 "mysqlDataSource" ✓
   └─ 注入 mysqlDataSource
```

**规则 3：如果参数名也不匹配，看有没有 `@Primary`**
```java
@Bean
@Primary  // ← 标记为"默认的"
public DataSource mysqlDataSource() { ... }

// 注入时，自动选 mysqlDataSource
@Inject
private DataSource dataSource;  // ✓ 注入 mysqlDataSource
```

**规则 4：如果都不满足，用 `@Qualifier` 明确指定**
```java
@Inject
@Qualifier("postgresDataSource")  // ← 明确指定
private DataSource dataSource;  // ✓ 注入 postgresDataSource
```

---

#### 完整的匹配流程图

```
Spring 容器注入流程：

1. 按类型匹配
   └─ 找到 1 个 → 直接注入 ✓
   └─ 找到 0 个 → 报错："No qualifying bean"
   └─ 找到多个 → 继续下一步

2. 按参数名匹配 Bean 名字
   └─ 匹配成功 → 注入 ✓
   └─ 不匹配 → 继续下一步

3. 看有没有 @Primary
   └─ 有 → 注入标记 @Primary 的 Bean ✓
   └─ 没有 → 继续下一步

4. 看有没有 @Qualifier
   └─ 有 → 注入指定的 Bean ✓
   └─ 没有 → 报错："expected single matching bean but found X"
```

---

#### 类比锚定：餐厅点菜

**场景：** 你去餐厅，菜单上有 2 道"鱼"：清蒸鱼、红烧鱼。

**情况 1：按名字点（参数名匹配）**
```
你：来一份"清蒸鱼"
服务员：好的 ✓
```

**情况 2：只说"鱼"，但有"招牌菜"（@Primary）**
```
你：来一份鱼
服务员：我们有 2 种鱼，红烧鱼是招牌，给您上这个
你：好 ✓
```

**情况 3：只说"鱼"，又没有招牌，服务员懵了**
```
你：来一份鱼
服务员：我们有 2 种鱼，您要哪个？
你：...（没说清楚）
服务员：那我没法上菜 ✗
```

**情况 4：用"编号"明确指定（@Qualifier）**
```
你：来一份鱼，要菜单上的"2 号鱼"
服务员：好的，红烧鱼 ✓
```

#### 代码示例（完整）

```java
@Configuration
public class DataSourceConfig {
    
    @Bean
    @Primary  // ← 方式 1：标记默认的
    public DataSource mysqlDataSource() {
        HikariDataSource ds = new HikariDataSource();
        ds.setJdbcUrl("jdbc:mysql://localhost:3306/db1");
        return ds;
    }
    
    @Bean
    public DataSource postgresDataSource() {
        HikariDataSource ds = new HikariDataSource();
        ds.setJdbcUrl("jdbc:postgresql://localhost:5432/db2");
        return ds;
    }
}

@Service
public class UserService {
    
    // 方式 1：参数名匹配
    @Inject
    private DataSource mysqlDataSource;  // ✓ 注入 MySQL
    
    // 方式 2：@Primary（已标记，直接注入）
    @Inject
    private DataSource dataSource;  // ✓ 注入 MySQL（因为有 @Primary）
    
    // 方式 3：@Qualifier 明确指定
    @Inject
    @Qualifier("postgresDataSource")
    private DataSource pgDataSource;  // ✓ 注入 PostgreSQL
}
```

---

#### 一句话总结

**Spring 选择 Bean 的顺序：类型 → 参数名 → @Primary → @Qualifier → 报错。**

### Q2：`@Primary` 和 `@Qualifier` 的区别？

#### 核心区别

| 特性 | @Primary | @Qualifier |
|------|---------|-----------|
| **标记位置** | 在 `@Bean` 方法上 | 在注入点（`@Inject`）上 |
| **作用** | 标记"默认的" Bean | 明确指定要哪个 Bean |
| **优先级** | 低（可以被 @Qualifier 覆盖） | 高（明确指定） |
| **使用场景** | 大多数地方用这个 | 少数地方需要用另一个 |
| **灵活性** | 全局生效 | 局部控制 |

---

#### 类比锚定：默认浏览器 vs 明确指定

**@Primary = 设置默认浏览器**
```
你的电脑上有：Chrome、Safari、Firefox
你设置 Chrome 为"默认浏览器"

双击网页链接 → 自动用 Chrome 打开 ✓
不用每次都选
```

**@Qualifier = 右键选择"打开方式"**
```
某个特殊文件，你想用 Firefox 打开
右键 → 打开方式 → 选择 Firefox ✓
只有这一次用 Firefox，不影响默认设置
```

#### 代码对比

**场景：** 项目有 2 个缓存：Redis（快，贵）、本地缓存（慢，免费）

```java
@Configuration
public class CacheConfig {
    
    @Bean
    @Primary  // ← 大部分地方用 Redis
    public CacheService redisCache() {
        return new RedisCache();
    }
    
    @Bean
    public CacheService localCache() {
        return new LocalCache();
    }
}

@Service
public class UserService {
    
    // 场景 1：默认用 Redis（不写 @Qualifier）
    @Inject
    private CacheService cache;  // ✓ 注入 redisCache（因为有 @Primary）
    
    public User getUser(Long id) {
        return cache.get("user:" + id);  // 从 Redis 取
    }
}

@Service
public class ReportService {
    
    // 场景 2：报表数据不重要，用本地缓存省钱
    @Inject
    @Qualifier("localCache")  // ← 明确指定
    private CacheService cache;  // ✓ 注入 localCache
    
    public Report getReport(Long id) {
        return cache.get("report:" + id);  // 从本地缓存取
    }
}
```

---

#### 什么时候用哪个？

**用 `@Primary`：**
- 有一个"主要的"、"默认的" Bean
- 大部分地方都用这个
- 例如：主数据库、默认缓存、默认消息队列

**用 `@Qualifier`：**
- 少数地方需要用"另一个"
- 例如：某个特殊功能要用从库、某个功能要用本地缓存

**两者结合（推荐）：**
```java
// 配置端：标记默认的
@Bean
@Primary
public DataSource masterDataSource() { ... }

@Bean
public DataSource slaveDataSource() { ... }

// 使用端：大部分地方不用写 @Qualifier
@Inject
private DataSource dataSource;  // ✓ 自动用 masterDataSource

// 少数地方明确指定
@Inject
@Qualifier("slaveDataSource")
private DataSource slaveDs;  // ✓ 用从库
```

---

#### 一句话总结

**`@Primary` = 设置默认，`@Qualifier` = 明确指定，两者结合最灵活。**

### Q3：什么时候用注解，什么时候用 Java Config？

#### 三种配置 Bean 的方式

**方式 1：注解（`@Component`/`@Service`/`@Repository`）**
```java
@Service  // ← 注解方式
public class UserService {
    // Spring 自动扫描，注册成 Bean
}
```

**方式 2：Java Config（`@Configuration` + `@Bean`）**
```java
@Configuration
public class AppConfig {
    
    @Bean  // ← Java Config 方式
    public UserService userService() {
        return new UserService();
    }
}
```

**方式 3：XML 配置（已过时，不推荐）**
```xml
<bean id="userService" class="com.example.UserService"/>
```

---

#### 什么时候用注解？

**场景 1：你自己写的类**
```java
// ✓ 推荐用注解
@Service
public class UserService { ... }

@Controller
public class UserController { ... }
```

**优点：**
- 简单，一个注解搞定
- 类在哪，配置在哪，不用跳来跳去

**适用：** 你项目里自己写的业务类

---

#### 什么时候用 Java Config？

**场景 1：第三方库的类（你改不了源码）**
```java
// ✗ 不能改第三方库的源码加 @Service
// HikariDataSource 是第三方库的类

// ✓ 用 Java Config
@Configuration
public class DataSourceConfig {
    
    @Bean
    public DataSource dataSource() {
        HikariDataSource ds = new HikariDataSource();
        ds.setJdbcUrl("jdbc:mysql://...");
        ds.setUsername("root");
        ds.setPassword("123456");
        return ds;
    }
}
```

**场景 2：需要复杂的初始化逻辑**
```java
@Configuration
public class CacheConfig {
    
    @Bean
    public RedisTemplate<String, Object> redisTemplate() {
        RedisTemplate<String, Object> template = new RedisTemplate<>();
        
        // 复杂的配置
        template.setConnectionFactory(connectionFactory());
        template.setKeySerializer(new StringRedisSerializer());
        template.setValueSerializer(new GenericJackson2JsonSerializer());
        template.afterPropertiesSet();
        
        return template;
    }
}
```

**场景 3：需要根据条件创建不同的 Bean**
```java
@Configuration
public class DataSourceConfig {
    
    @Value("${spring.profiles.active}")
    private String profile;
    
    @Bean
    public DataSource dataSource() {
        if ("dev".equals(profile)) {
            // 开发环境：用 H2 内存数据库
            return new EmbeddedDatabaseBuilder().build();
        } else {
            // 生产环境：用 MySQL
            HikariDataSource ds = new HikariDataSource();
            ds.setJdbcUrl("jdbc:mysql://...");
            return ds;
        }
    }
}
```

**场景 4：一个类需要创建多个 Bean**
```java
@Configuration
public class DataSourceConfig {
    
    @Bean
    public DataSource masterDataSource() {
        // 主库
    }
    
    @Bean
    public DataSource slaveDataSource() {
        // 从库
    }
    
    // 同一个类 HikariDataSource，创建 2 个不同的 Bean
}
```

---

#### 类比锚定：装修房子

**注解 = 精装修房**
```
开发商已经装修好了（类上加 @Service）
你拎包入住就行 ✓
```

**Java Config = 毛坯房自己装修**
```
开发商只给你毛坯房（第三方库的类）
你自己买家具、刷墙、装修（@Bean 方法里配置）
```

---

#### 选择建议

| 场景 | 推荐方式 | 原因 |
|------|---------|------|
| 你自己的业务类 | **注解** | 简单，配置在类上 |
| 第三方库的类 | **Java Config** | 你改不了源码 |
| 需要复杂初始化 | **Java Config** | @Bean 方法里可以写逻辑 |
| 需要条件判断 | **Java Config** | 可以用 if/else |
| 一个类多个实例 | **Java Config** | 可以写多个 @Bean 方法 |

---

#### 代码示例（综合）

```java
// 你自己的类：用注解 ✓
@Service
public class UserService {
    
    @Inject
    private UserMapper userMapper;  // 自动注入
    
    public User getUser(Long id) {
        return userMapper.findById(id);
    }
}

// 第三方库的类：用 Java Config ✓
@Configuration
public class ThirdPartyConfig {
    
    @Bean
    public ObjectMapper objectMapper() {
        ObjectMapper mapper = new ObjectMapper();
        mapper.setSerializationInclusion(JsonInclude.Include.NON_NULL);
        mapper.configure(DeserializationFeature.FAIL_ON_UNKNOWN_PROPERTIES, false);
        return mapper;
    }
    
    @Bean
    public RestTemplate restTemplate() {
        RestTemplate template = new RestTemplate();
        template.setRequestFactory(new HttpComponentsClientHttpRequestFactory());
        return template;
    }
}
```

---

#### 一句话总结

**自己的类用注解，第三方库和复杂配置用 Java Config。**

## 第二部分：Spring Security 认证流程

### Q4：Spring Security 的认证流程是什么？

#### 问题背景

你在学 1.6 和 1.7 文档时，看到了这些类：
- `UserDetailsService`
- `AuthenticationManager`
- `Provider`
- `UsernamePasswordAuthenticationToken`

**你的困惑：** 它们之间是什么关系？整个登录流程到底是怎样的？

---

#### 完整的认证流程图（核心）

```
用户登录流程（从前端到后端）：

1. 前端
   └─ 用户输入：username="user", password="password"
   └─ 发送：POST /auth/login

2. Controller 层（AuthController）
   └─ 接收请求
   └─ 调用 userDetailsService.loadUserByUsername("user")
        ↓
   ┌─────────────────────────────────────┐
   │ 3. UserDetailsService（你的实现）    │
   │    └─ 从数据库/内存查用户             │
   │    └─ 返回 UserDetails 对象          │
   │       ├─ username: "user"           │
   │       ├─ password: "$2a$10$xxx..."  │
   │       └─ authorities: ["ROLE_USER"] │
   └─────────────────────────────────────┘
        ↓
   └─ 创建 UsernamePasswordAuthenticationToken
        ↓
   └─ 调用 authenticationManager.authenticate(token)
        ↓
   ┌─────────────────────────────────────┐
   │ 4. AuthenticationManager             │
   │    └─ 内部有多个 Provider            │
   │    └─ 遍历每个 Provider 尝试验证      │
   └─────────────────────────────────────┘
        ↓
   ┌─────────────────────────────────────┐
   │ 5. Provider（如 DaoAuthenticationProvider） │
   │    └─ 拿着 token 里的密码              │
   │    └─ 和 UserDetails 里的密码对比      │
   │    └─ 密码匹配 ✓                      │
   │    └─ 返回"已认证"的 Authentication     │
   └─────────────────────────────────────┘
        ↓
   └─ 验证成功，设置到 SecurityContext
   └─ 返回给前端：{"status": "ok", "user": {...}}
```

---

#### 各个组件的作用（一句话总结）

| 组件 | 作用 | 类比 |
|------|------|------|
| **UserDetailsService** | 从数据库/内存加载用户信息 | HR 部门（查员工档案） |
| **UserDetails** | 用户信息对象（用户名、密码、权限） | 员工档案卡 |
| **AuthenticationManager** | 认证总调度，管理多个 Provider | 安检总站（管多道关卡） |
| **Provider** | 具体的验证器（验证密码是否正确） | 某道安检关卡 |
| **UsernamePasswordAuthenticationToken** | 认证请求对象（装着用户名和密码） | 登机牌（待验证） |
| **Authentication** | 认证结果对象（验证成功后的身份信息） | 登机牌（已盖章） |
| **SecurityContext** | 存储当前用户的认证信息 | 随身携带的登机牌 |

---

#### 类比锚定：机场安检

**完整流程：**

```
你要坐飞机（登录）

1. 你拿着身份证和机票去安检口
   └─ UsernamePasswordAuthenticationToken（你的身份证 + 机票）

2. 安检总站（AuthenticationManager）接收你的材料
   └─ "让我找个关卡验证你"

3. 第一道关卡（Provider）验证
   ├─ 先查档案（UserDetailsService.loadUserByUsername）
   │   └─ 档案系统返回你的信息（UserDetails）
   │        ├─ 姓名：张三
   │        ├─ 身份证号：123456（加密后的）
   │        └─ 权限：可以坐经济舱
   │
   ├─ 对比你提供的身份证号和档案里的
   │   └─ 匹配 ✓
   │
   └─ 在你的机票上盖章（返回 Authentication）
        └─ 你现在是"已认证"状态

4. 把盖好章的机票放进你的口袋（SecurityContext）
   └─ 你随身携带，随时能证明你是谁

5. 你拿着盖章的机票去登机 ✓
```

---

#### 代码示例（完整流程）

```java
// 1. 实现 UserDetailsService（HR 部门）
@Service
public class UserService implements UserDetailsService {
    
    @Inject
    private UserMapper userMapper;  // 数据库查询
    
    @Override
    public UserDetails loadUserByUsername(String username) 
            throws UsernameNotFoundException {
        
        // 从数据库查用户
        User user = userMapper.findByUsername(username);
        if (user == null) {
            throw new UsernameNotFoundException("用户不存在");
        }
        
        // 返回 Spring Security 的 UserDetails 对象
        return new org.springframework.security.core.userdetails.User(
            user.getUsername(),
            user.getPassword(),  // 加密后的密码
            user.getAuthorities()  // 权限列表
        );
    }
}

// 2. 配置 AuthenticationManager（安检总站）
@Configuration
@EnableWebSecurity
public class WebSecurityConfig extends WebSecurityConfigurerAdapter {
    
    @Inject
    private UserService userService;
    
    @Override
    protected void configure(AuthenticationManagerBuilder auth) throws Exception {
        // 告诉 AuthenticationManager：用 userService 加载用户
        auth.userDetailsService(userService)
            .passwordEncoder(new BCryptPasswordEncoder());  // 密码加密器
    }
    
    @Bean
    @Override
    public AuthenticationManager authenticationManagerBean() throws Exception {
        return super.authenticationManagerBean();
    }
}

// 3. Controller 里使用（安检流程）
@Controller
public class AuthController {
    
    @Inject
    private UserDetailsService userDetailsService;
    
    @Inject
    private AuthenticationManager authenticationManager;
    
    @PostMapping("/auth/login")
    @ResponseBody
    public Result login(@RequestBody Map<String, String> params) {
        String username = params.get("username");
        String password = params.get("password");
        
        // 步骤 1：加载用户信息
        UserDetails userDetails;
        try {
            userDetails = userDetailsService.loadUserByUsername(username);
        } catch (UsernameNotFoundException e) {
            return new Result("fail", "用户不存在", false);
        }
        
        // 步骤 2：创建认证请求（未认证的 token）
        UsernamePasswordAuthenticationToken token = 
            new UsernamePasswordAuthenticationToken(username, password);
        
        // 步骤 3：验证密码
        try {
            Authentication auth = authenticationManager.authenticate(token);
            
            // 步骤 4：验证成功，设置到 SecurityContext
            SecurityContextHolder.getContext().setAuthentication(auth);
            
            return new Result("ok", "登录成功", true);
        } catch (BadCredentialsException e) {
            return new Result("fail", "密码错误", false);
        }
    }
}
```

---

#### 关键点详解

**1. 为什么需要 `UserDetailsService`？**

因为 Spring Security 不知道你的用户数据在哪（数据库？文件？内存？），你要告诉它"如何加载用户"。

**2. 为什么要创建 `UsernamePasswordAuthenticationToken`？**

这是 Spring Security 的标准格式，就像你去银行取钱，要填银行的标准表格一样。

**3. 为什么验证成功后要设置到 `SecurityContext`？**

因为后续的请求需要知道"当前登录的是谁"，`SecurityContext` 就像你的"随身钱包"，装着你的身份证明。

---

#### 一句话总结

**登录流程：Controller 调用 UserDetailsService 加载用户 → 创建 token → AuthenticationManager 验证 → Provider 对比密码 → 验证成功 → 存到 SecurityContext。**

### Q5：`UserDetailsService`、`AuthenticationManager`、`Provider` 的关系？

#### 三者的层级关系

```
┌─────────────────────────────────────────┐
│         AuthenticationManager           │  ← 最外层：总调度
│  （安检总站）                            │
│                                          │
│  ┌────────────────────────────────────┐ │
│  │        Provider 1                  │ │  ← 中间层：验证器
│  │  （第一道安检关卡）                  │ │
│  │                                    │ │
│  │  ┌──────────────────────────────┐ │ │
│  │  │   UserDetailsService         │ │ │  ← 最内层：数据加载
│  │  │   （HR 查档案）              │ │ │
│  │  └──────────────────────────────┘ │ │
│  └────────────────────────────────────┘ │
│                                          │
│  ┌────────────────────────────────────┐ │
│  │        Provider 2                  │ │
│  │  （第二道安检关卡）                  │ │
│  └────────────────────────────────────┘ │
└─────────────────────────────────────────┘
```

---

#### 各自的职责

**UserDetailsService（最内层）：**
- **职责：** 只负责"查数据"，不负责验证
- **输入：** 用户名
- **输出：** 用户信息（用户名、密码、权限）
- **类比：** HR 部门，你问"张三是谁"，HR 给你张三的档案

```java
public interface UserDetailsService {
    UserDetails loadUserByUsername(String username) 
        throws UsernameNotFoundException;
    // 只是查数据，不验证密码！
}
```

**Provider（中间层）：**
- **职责：** 负责"验证"，对比密码是否匹配
- **输入：** 认证请求（username + password）
- **输出：** 验证结果（成功/失败）
- **类比：** 安检关卡，拿着你的身份证和档案对比

```java
public interface AuthenticationProvider {
    Authentication authenticate(Authentication authentication) 
        throws AuthenticationException;
    // 验证密码是否正确
}
```

**AuthenticationManager（最外层）：**
- **职责：** 总调度，管理多个 Provider
- **输入：** 认证请求
- **输出：** 验证结果
- **类比：** 安检总站，管理多道安检关卡

```java
public interface AuthenticationManager {
    Authentication authenticate(Authentication authentication) 
        throws AuthenticationException;
    // 调度 Provider 进行验证
}
```

---

#### 调用链路

```
1. Controller 调用 AuthenticationManager
       ↓
2. AuthenticationManager 选择一个 Provider
       ↓
3. Provider 调用 UserDetailsService.loadUserByUsername()
       ↓
4. UserDetailsService 返回 UserDetails（用户信息）
       ↓
5. Provider 拿着 UserDetails 的密码和用户输入的密码对比
       ↓
6. Provider 返回验证结果给 AuthenticationManager
       ↓
7. AuthenticationManager 返回结果给 Controller
```

---

#### 为什么要分这么多层？

**原因 1：职责分离**
- `UserDetailsService`：只管查数据（可能从数据库、LDAP、文件等）
- `Provider`：只管验证逻辑（可能用不同的验证算法）
- `AuthenticationManager`：只管调度（可能有多种验证方式）

**原因 2：灵活扩展**

**场景：** 你的公司有 3 种登录方式

```
AuthenticationManager
├─ Provider 1：用户名密码登录
│   └─ 使用 DatabaseUserDetailsService（从数据库查）
├─ Provider 2：微信扫码登录
│   └─ 使用 WechatUserDetailsService（调微信 API）
└─ Provider 3：指纹登录
    └─ 使用 BiometricUserDetailsService（指纹识别）
```

**好处：** 添加新的登录方式，只需要加一个 Provider，不用改其他代码！

---

#### 类比锚定：医院看病

**UserDetailsService = 病历科**
```
你：我要查张三的病历
病历科：好的，给你病历（只是查资料，不诊断）
```

**Provider = 医生**
```
医生：拿着病历，看看你的症状
医生：对比检查（验证）
医生：诊断结果：你感冒了（验证成功）
```

**AuthenticationManager = 挂号台**
```
挂号台：你要看什么科？
你：内科
挂号台：好，给你分配内科医生（选择 Provider）
```

#### 代码示例（完整）

```java
// 1. UserDetailsService（查数据）
@Service
public class DatabaseUserDetailsService implements UserDetailsService {
    
    @Inject
    private UserMapper userMapper;
    
    @Override
    public UserDetails loadUserByUsername(String username) {
        User user = userMapper.findByUsername(username);
        if (user == null) {
            throw new UsernameNotFoundException("用户不存在");
        }
        return new org.springframework.security.core.userdetails.User(
            user.getUsername(),
            user.getPassword(),
            user.getAuthorities()
        );
    }
}

// 2. Provider（验证）
// Spring Security 默认提供了 DaoAuthenticationProvider
// 它内部会：
// ① 调用 UserDetailsService.loadUserByUsername()
// ② 拿到 UserDetails 后，对比密码
// ③ 返回验证结果

// 3. AuthenticationManager（调度）
@Configuration
@EnableWebSecurity
public class WebSecurityConfig extends WebSecurityConfigurerAdapter {
    
    @Inject
    private DatabaseUserDetailsService userDetailsService;
    
    @Override
    protected void configure(AuthenticationManagerBuilder auth) throws Exception {
        // 配置：AuthenticationManager 用哪个 UserDetailsService
        auth.userDetailsService(userDetailsService)
            .passwordEncoder(new BCryptPasswordEncoder());
        
        // Spring 会自动创建：
        // AuthenticationManager
        //   └─ DaoAuthenticationProvider
        //       └─ userDetailsService
    }
    
    @Bean
    @Override
    public AuthenticationManager authenticationManagerBean() throws Exception {
        return super.authenticationManagerBean();
    }
}

// 4. Controller（使用）
@Controller
public class AuthController {
    
    @Inject
    private AuthenticationManager authenticationManager;  // ← 只用最外层
    
    @PostMapping("/auth/login")
    @ResponseBody
    public Result login(@RequestBody Map<String, String> params) {
        String username = params.get("username");
        String password = params.get("password");
        
        // 创建认证请求
        UsernamePasswordAuthenticationToken token = 
            new UsernamePasswordAuthenticationToken(username, password);
        
        // 调用 AuthenticationManager
        // 它内部会：
        // ① 找到 DaoAuthenticationProvider
        // ② Provider 调用 UserDetailsService
        // ③ Provider 验证密码
        // ④ 返回结果
        try {
            Authentication auth = authenticationManager.authenticate(token);
            SecurityContextHolder.getContext().setAuthentication(auth);
            return new Result("ok", "登录成功", true);
        } catch (BadCredentialsException e) {
            return new Result("fail", "密码错误", false);
        }
    }
}
```

**UserDetailsService 查数据，Provider 验证密码，AuthenticationManager 调度 Provider。**

## 第三部分：Spring AOP 原理

### Q6：AOP 是如何实现的？（JDK 代理 vs CGLIB）

#### 问题背景

你在 Q5（StackOverflowError）里看到了 `$Proxy69` 这样的日志，这是什么？

**答：** 这是 Spring 的 AOP 代理对象。

---

#### 什么是 AOP？

**AOP = Aspect Oriented Programming（面向切面编程）**

**通俗理解：** 在方法调用前后"偷偷加料"（记日志、开事务、检查权限等），不用改原来的代码。

**类比：** 快递代收点（菜鸟驿站）

```
正常流程：快递员 → 你
AOP 流程：快递员 → 【菜鸟驿站】 → 你
             ↑
         代理（偷偷加料）
         ├─ 记录：XX 时间收到快递
         ├─ 验货：检查包装是否完好
         └─ 转发：最终还是给你
```

---

#### AOP 的两种实现方式

**方式 1：JDK 动态代理（基于接口）**
- **要求：** 目标类必须实现接口
- **原理：** 创建一个接口的代理对象
- **优点：** JDK 自带，不需要额外依赖
- **缺点：** 必须有接口

**方式 2：CGLIB 代理（基于继承）**
- **要求：** 目标类不能是 `final`
- **原理：** 创建目标类的子类
- **优点：** 不需要接口
- **缺点：** 需要额外的库（Spring 已自带）

---

#### JDK 动态代理示例

```java
// 1. 定义接口
public interface UserService {
    void save(User user);
}

// 2. 实现类
@Service
public class UserServiceImpl implements UserService {
    @Override
    public void save(User user) {
        System.out.println("保存用户：" + user.getName());
    }
}

// 3. Spring 创建代理
// 实际上容器里的 Bean 是：
UserService userService = (UserService) Proxy.newProxyInstance(
    UserServiceImpl.class.getClassLoader(),
    new Class[]{UserService.class},  // ← 基于接口
    new InvocationHandler() {
        @Override
        public Object invoke(Object proxy, Method method, Object[] args) {
            // 前置：记录日志
            System.out.println("【AOP】开始调用：" + method.getName());
            
            // 调用真实方法
            Object result = method.invoke(target, args);
            
            // 后置：记录日志
            System.out.println("【AOP】调用结束：" + method.getName());
            
            return result;
        }
    }
);

// 4. 调用时
userService.save(user);
// 输出：
// 【AOP】开始调用：save
// 保存用户：张三
// 【AOP】调用结束：save

// 5. 查看代理对象的类名
System.out.println(userService.getClass().getName());
// 输出：com.sun.proxy.$Proxy69  ← 这就是日志里看到的！
```

---

#### CGLIB 代理示例

```java
// 1. 没有接口的类
@Service
public class OrderService {  // ← 没实现接口
    
    public void createOrder(Order order) {
        System.out.println("创建订单：" + order.getId());
    }
}

// 2. Spring 创建代理
// 实际上容器里的 Bean 是：
OrderService orderService = (OrderService) Enhancer.create(
    OrderService.class,  // ← 基于类（不是接口）
    new MethodInterceptor() {
        @Override
        public Object intercept(Object obj, Method method, Object[] args, 
                                MethodProxy proxy) {
            // 前置
            System.out.println("【AOP】开始调用：" + method.getName());
            
            // 调用真实方法
            Object result = proxy.invokeSuper(obj, args);
            
            // 后置
            System.out.println("【AOP】调用结束：" + method.getName());
            
            return result;
        }
    }
);

// 3. 调用时
orderService.createOrder(order);
// 输出：
// 【AOP】开始调用：createOrder
// 创建订单：12345
// 【AOP】调用结束：createOrder

// 4. 查看代理对象的类名
System.out.println(orderService.getClass().getName());
// 输出：com.example.OrderService$$EnhancerBySpringCGLIB$$12345678
```

---

#### Spring 如何选择代理方式？

**默认规则：**

```
目标类有接口？
├─ 是 → 用 JDK 动态代理
└─ 否 → 用 CGLIB 代理
```

**手动指定（强制用 CGLIB）：**

```java
@Configuration
@EnableAspectJAutoProxy(proxyTargetClass = true)  // ← 强制用 CGLIB
public class AppConfig {
    // ...
}
```

---

#### 类比锚定：快递代收

**JDK 动态代理 = 快递柜（必须有接口）**
```
快递员 → 【快递柜】 → 你
         ↑
     标准接口（柜子编号）
```

**CGLIB 代理 = 菜鸟驿站（不需要接口）**
```
快递员 → 【菜鸟驿站】 → 你
         ↑
     不需要标准接口，直接存
```

#### 常见的 AOP 应用

**1. 事务管理（`@Transactional`）**
```java
@Service
public class UserService {
    
    @Transactional  // ← AOP 实现
    public void transferMoney(Long fromId, Long toId, BigDecimal amount) {
        // AOP 前置：开启事务
        accountMapper.deduct(fromId, amount);
        accountMapper.add(toId, amount);
        // AOP 后置：提交事务
    }
}
```

**2. 日志记录**
```java
@Aspect
@Component
public class LogAspect {
    
    @Around("execution(* com.example.service.*.*(..))")
    public Object log(ProceedingJoinPoint pjp) throws Throwable {
        System.out.println("【日志】调用方法：" + pjp.getSignature().getName());
        Object result = pjp.proceed();  // 调用真实方法
        System.out.println("【日志】方法返回：" + result);
        return result;
    }
}
```

**3. 权限检查**
```java
@Aspect

    @Before("@annotation(RequiresAdmin)")

        User user = SecurityContextHolder.getContext().getAuthentication();
        if (!user.hasRole("ADMIN")) {
            throw new AccessDeniedException("需要管理员权限");
        }
    }
}
```

**AOP = 方法调用的"拦截器"，JDK 代理基于接口，CGLIB 代理基于继承。**

### Q7：为什么 `@Transactional` 有时不生效？

#### 问题场景

```java
@Service
public class UserService {
    
    public void updateUser(User user) {
        this.saveUser(user);  // ← 调用自己的方法
    }
    
    @Transactional  // ← 事务不生效！
    public void saveUser(User user) {
        userMapper.update(user);
        if (user.getAge() < 0) {
            throw new RuntimeException("年龄不能为负数");
        }
    }
}
```

**问题：** `saveUser()` 上有 `@Transactional`，为什么事务不生效？

---

#### 根本原因：绕过了 AOP 代理

**正常情况（事务生效）：**

```
外部类 → 【AOP 代理】 → UserService.saveUser()
                ↑
            事务拦截器
            ├─ 前置：开启事务
            ├─ 调用真实方法
            └─ 后置：提交/回滚事务
```

**有问题的情况（事务不生效）：**

```
UserService.updateUser() 
    ↓
this.saveUser()  ← 直接调用，绕过了代理！
    ↓
UserService.saveUser()
    ↓
没有经过 AOP 代理，事务拦截器没执行 ✗
```

---

#### 类比锚定：过安检

**正常流程（走安检）：**
```
你 → 【安检口】 → 登机口
        ↑
    检查身份证、行李
```

**有问题的流程（绕过安检）：**
```
你在机场里面 → 直接走后门去登机口
              ↓
          没过安检 ✗
```

---

#### 详细原理

**Spring 事务是通过 AOP 实现的：**

```java
// Spring 创建的代理对象
UserService userServiceProxy = (UserService) Proxy.newProxyInstance(...);

// 代理对象的 saveUser() 方法：
public void saveUser(User user) {
    // 前置：开启事务
    TransactionStatus status = transactionManager.getTransaction(...);
    
    try {
        // 调用真实对象的方法
        target.saveUser(user);
        
        // 后置：提交事务
        transactionManager.commit(status);
    } catch (Exception e) {
        // 异常：回滚事务
        transactionManager.rollback(status);
        throw e;
    }
}
```

**当你用 `this.saveUser()` 时：**

```java
public void updateUser(User user) {
    this.saveUser(user);
    // this = 真实对象（不是代理）
    // 直接调用真实对象的方法，没走代理！
}
```

---

#### 解决方案

**方案 1：注入自己（推荐）**

```java
@Service
public class UserService {
    
    @Inject
    private UserService self;  // ← 注入的是代理对象
    
    public void updateUser(User user) {
        self.saveUser(user);  // ✓ 走代理
    }
    
    @Transactional
    public void saveUser(User user) {
        userMapper.update(user);
    }
}
```

**方案 2：拆成 2 个类（最推荐）**

```java
@Service
public class UserService {
    
    @Inject
    private UserRepository userRepository;  // ← 另一个类
    
    public void updateUser(User user) {
        userRepository.saveUser(user);  // ✓ 走代理
    }
}

@Service
public class UserRepository {
    
    @Transactional
    public void saveUser(User user) {
        userMapper.update(user);
    }
}
```

**方案 3：用 `AopContext`（不推荐）**

```java
@Service
@EnableAspectJAutoProxy(exposeProxy = true)  // ← 暴露代理
public class UserService {
    
    public void updateUser(User user) {
        UserService proxy = (UserService) AopContext.currentProxy();
        proxy.saveUser(user);  // ✓ 走代理
    }
    
    @Transactional
    public void saveUser(User user) {
        userMapper.update(user);
    }
}
```

---

#### 其他常见的事务失效场景

**场景 1：方法不是 `public`**

```java
@Transactional
private void saveUser(User user) {  // ✗ private 方法，事务不生效
    userMapper.update(user);
}
```

**原因：** Spring 的 AOP 代理只能拦截 `public` 方法。

---

**场景 2：异常被捕获了**

```java
@Transactional
public void saveUser(User user) {
    try {
        userMapper.update(user);
        throw new RuntimeException("出错了");
    } catch (Exception e) {
        // 异常被吃掉了，事务不会回滚 ✗
    }
}
```

**解决：** 重新抛出异常

```java
@Transactional
public void saveUser(User user) {
    try {
        userMapper.update(user);
    } catch (Exception e) {
        throw e;  // ✓ 重新抛出
    }
}
```

---

**场景 3：异常类型不对**

```java
@Transactional
public void saveUser(User user) throws Exception {
    userMapper.update(user);
    throw new Exception("出错了");  // ✗ checked 异常，默认不回滚
}
```

**原因：** Spring 默认只对 `RuntimeException` 和 `Error` 回滚。

**解决：** 指定回滚的异常类型

```java
@Transactional(rollbackFor = Exception.class)  // ✓ 所有异常都回滚
public void saveUser(User user) throws Exception {
    userMapper.update(user);
    throw new Exception("出错了");
}
```

---

#### 一句话总结

**`@Transactional` 是 AOP 实现的，用 `this` 调用自己的方法会绕过代理，事务不生效。**

### Q8：如何避免 AOP 相关的坑？

#### 常见的 AOP 坑（总结）

| 坑 | 现象 | 原因 | 解决 |
|----|------|------|------|
| **坑 1** | `@Transactional` 不生效 | `this` 调用，绕过代理 | 注入自己或拆类 |
| **坑 2** | `private` 方法事务不生效 | AOP 只拦截 `public` | 改成 `public` |
| **坑 3** | 异常被捕获，事务不回滚 | 异常没抛出 | 重新抛出异常 |
| **坑 4** | checked 异常不回滚 | 默认只回滚 `RuntimeException` | 加 `rollbackFor` |
| **坑 5** | 死循环（Q5 的 bug） | AOP 代理 + 多 Provider | 删掉多余的 Bean |

---

#### 避坑指南

**1. 事务方法的 5 个原则**

```java

    rollbackFor = Exception.class,  // ① 指定回滚异常
    propagation = Propagation.REQUIRED,  // ② 指定传播行为
    isolation = Isolation.DEFAULT  // ③ 指定隔离级别
)
public void saveUser(User user) throws Exception {  // ④ public 方法
    userMapper.update(user);
    // ⑤ 不要捕获异常
}
```

**2. 同类调用的 3 个方案**

```java
// 方案 1：注入自己
@Inject
private UserService self;

public void method1() {
    self.method2();  // ✓
}

// 方案 2：拆类
@Inject
private AnotherService anotherService;

public void method1() {
    anotherService.method2();  // ✓
}

// 方案 3：用 AopContext（不推荐）
UserService proxy = (UserService) AopContext.currentProxy();
proxy.method2();  // ✓
```

**3. 调试 AOP 的 3 个技巧**

    // 可以用 AopContext.currentProxy()

---

#### 一句话总结

**记住：AOP 是代理，`this` 调用绕过代理，事务要 `public`，异常要抛出。**

## 第四部分：常见面试题速查

### 面试题 1：Spring 依赖注入

**Q1：如果容器里有多个同类型的 Bean，Spring 如何选择？**

**答：** 类型 → 参数名 → @Primary → @Qualifier → 报错。

**Q2：`@Primary` 和 `@Qualifier` 的区别？**

**答：** `@Primary` 标记默认的（全局），`@Qualifier` 明确指定（局部）。

**Q3：`@Autowired` 和 `@Inject` 的区别？**

**答：** 
- `@Autowired`：Spring 提供，有 `required` 属性
- `@Inject`：JSR-330 标准，没有 `required` 属性
- 推荐用 `@Inject`（标准，可移植）

---

### 面试题 2：Spring Security

**Q1：Spring Security 的认证流程？**

**答：** Controller → UserDetailsService（查用户）→ AuthenticationManager → Provider（验证密码）→ SecurityContext。

**Q2：`UserDetailsService` 的作用？**

**答：** 从数据库/内存加载用户信息，不负责验证密码。

**Q3：如何实现多种登录方式？**

**答：** 配置多个 Provider，每个 Provider 对应一种登录方式（如账号密码、微信、指纹）。

---

### 面试题 3：Spring AOP

**Q1：Spring AOP 是如何实现的？**

**答：** JDK 动态代理（基于接口）或 CGLIB 代理（基于继承）。

**Q2：为什么 `@Transactional` 有时不生效？**

**答：** 
1. `this` 调用，绕过代理
2. 方法不是 `public`
3. 异常被捕获
4. 异常类型不对（checked 异常）

**Q3：JDK 代理和 CGLIB 代理的区别？**

**答：** 
- JDK 代理：基于接口，必须有接口，生成 `$Proxy` 类
- CGLIB 代理：基于继承，不需要接口，生成子类

## 总结

### 核心知识点回顾

| 主题 | 核心问题 | 一句话总结 |
|------|---------|-----------|
| **依赖注入** | 多个 Bean 如何选择？ | 类型 → 参数名 → @Primary → @Qualifier |
| **Spring Security** | 认证流程是什么？ | UserDetailsService 查数据，Provider 验证，AuthenticationManager 调度 |
| **AOP** | 事务为什么不生效？ | `this` 调用绕过代理，要注入自己或拆类 |

### 学习建议

1. **理解原理，不要死记硬背**
   - 记住"快递代收点"的类比，就能理解 AOP
   - 记住"安检总站"的类比，就能理解 Spring Security

2. **多做实验**
   - 自己写代码测试"多个 Bean 的注入"
   - 测试 `@Transactional` 什么时候生效

3. **遇到问题先看日志**
   - `$Proxy69` → AOP 代理
   - `StackOverflowError` → 死循环
   - `No qualifying bean` → Bean 冲突

4. **面试前复习"一句话总结"**
   - 每个问题都有"一句话总结"
   - 面试时先说总结，面试官感兴趣再展开
