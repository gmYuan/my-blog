# 实战：Spring Security 用户管理从内存改成自定义实现

> 面向后端新手。这节课解决的核心问题是：**之前 `InMemoryUserDetailsManager` 把用户写死在代码里，现在要把用户查询的逻辑接到自己的 `UserService` 上，并且加上密码加密（BCrypt）。** 本质是把"用户从哪来"的控制权，从 Spring Security 内置的内存管理器，换成你自己的业务层。

---

## 第一部分：自顶向下的顶层鸟瞰

### 背景：我们在解决什么问题

你在之前的 1.6 文档里已经实现了登录功能，但有一个硬伤：

```java
// WebSecurityConfig.java（之前的写法）
@Bean
public UserDetailsService userDetailsService() {
    UserDetails user = User.withDefaultPasswordEncoder()
        .username("user")
        .password("password")
        .roles("USER")
        .build();
    return new InMemoryUserDetailsManager(user);
}
```

**问题出在哪：**

| 问题 | 后果 |
|------|------|
| 用户名/密码**写死在代码里** | 每次加用户都要改代码、重启项目，根本没法用 |
| 用户信息存在**内存**里 | 重启项目，用户全丢 |
| 不能和你自己的业务逻辑接上 | `UserService` 里明明已经可以查数据库了，Spring Security 却还在用内存里的假数据 |

**这节课要达成的目标：**

```
之前：Spring Security 用 InMemoryUserDetailsManager（写死的假数据）
           ↓
现在：Spring Security 用你自己的 UserService（可以查数据库的真数据）
           ↓
同时：密码不再明文存储，而是用 BCrypt 加密
```

---

### 核心概念：类比锚定

#### 概念 1：`UserDetailsService` 是什么

**类比：** 像公司的**人事档案系统接口**。

Spring Security 说："给我一个用户名，你告诉我这个人的档案（密码、角色、是否启用等）。"

不管你的档案存在 Excel 里、数据库里、还是内存里，Spring Security 不关心，它只要求你提供一个统一接口：

```java
public interface UserDetailsService {
    UserDetails loadUserByUsername(String username) throws UsernameNotFoundException;
}
```

**之前：** 你用 Spring Security 提供的 `InMemoryUserDetailsManager`（内存档案库）。

**现在：** 你让自己的 `UserService` 实现这个接口，变成"自己的档案库"。

---

#### 概念 2：`BCryptPasswordEncoder` 是什么

**类比：** 像银行卡密码的**不可逆加密机器**。

你输入明文密码 `password`，它吐出一串加密后的字符串：

```
$2a$10$abc...xyz（60个字符）
```

**特点：**
- **单向加密**：从 `password` 可以算出 `$2a$10$abc...`，但从 `$2a$10$abc...` **推不回去**原文
- **每次加密结果不同**：同样是 `password`，第一次加密出来是 `aaa...`，第二次是 `bbb...`（因为盐值随机），但验证时都能匹配
- **验证时不解密**：用户输入 `password`，你把它再加密一次，用特殊算法比对加密结果，看是否匹配

**为什么要用它：** 密码不能明文存数据库，被人拖库（黑客偷走数据库备份）后全泄露。加密后即使被偷，也推不回原密码。

### 全局地图：这 20 张截图在干什么

```
原来的架构（1.6 文档实现的）：
WebSecurityConfig.userDetailsService()
    │
    └─ return new InMemoryUserDetailsManager(user)
         └─ 用户写死：username="user", password="password"
              │
              └─ AuthController.login() 调用
                   └─ userDetailsService.loadUserByUsername()
                        └─ 从内存里找用户

现在的改造（这 20 张截图要实现的）：
① 让 UserService 实现 UserDetailsService 接口
    └─ 把"查用户"的逻辑接到自己的 Map 或数据库上

② 加入密码加密器（BCryptPasswordEncoder）
    └─ save() 存密码时加密
    └─ 验证密码时也用加密比对

③ 改 WebSecurityConfig
    └─ 不再 return new InMemoryUserDetailsManager
    └─ 改成两种方式之一：
         方式1（JavaConfiguration）：在 JavaConfiguration 里 @Bean 返回 new UserService()
         方式2（注解+字段注入）：给 UserService 加 @Service，在 WebSecurityConfig 里 @Autowired 注入

④ 遇到报错（PasswordEncoder 没配置）
    └─ 在 WebSecurityConfig 里新增 @Bean BCryptPasswordEncoder

⑤ 手绘图梳理：sign up 注册流程
    └─ 用户输入明文密码 → 加密存储 → 登录时比对加密后的密码
```

### 改造前后对比

| | 之前（内存版） | 现在（自定义版） |
|--|-------------|----------------|
| **用户从哪来** | `InMemoryUserDetailsManager`（写死在代码里） | 你自己的 `UserService`（可以查 Map 或数据库） |
| **密码怎么存** | 明文 `"password"` | 加密 `bCryptPasswordEncoder.encode("password")` |
| **怎么让 Spring Security 用上** | `@Bean userDetailsService() { return new InMemoryUserDetailsManager(...) }` | 让 `UserService` 实现 `UserDetailsService` 接口，注册成 Bean |
| **扩展性** | 每次加用户要改代码重启 | 调 `save()` 就能动态增加用户 |

---

### 技术栈关键词速查

| 关键词 | 一句话 |
|--------|--------|
| `UserDetailsService` | Spring Security 要求的"查用户"接口，只有一个方法 `loadUserByUsername()` |
| `UserDetails` | 描述一个用户的接口（用户名、密码、角色、是否启用等） |
| `InMemoryUserDetailsManager` | Spring Security 内置的内存用户管理器，用户信息存内存里 |
| `BCryptPasswordEncoder` | BCrypt 密码加密器，单向加密，每次结果不同但能验证 |
| `PasswordEncoder` | Spring Security 密码加密器的接口，`BCryptPasswordEncoder` 是实现之一 |
| `@Service` | 声明业务层 Bean（会被扫描注册） |
| `@Autowired` | 告诉容器：把某个 Bean 注入到这个字段里 |
| `Collections.emptyList()` | 返回一个空列表（不可变） |
| `.encode(password)` | 把明文密码加密 |

### 步骤 1：让 `UserService` 实现 `UserDetailsService` 接口

#### 改造前（只是普通业务类）

```java
package hello.service;

import hello.entity.User;
import hello.mapper.UserMapper;
import javax.inject.Inject;

public class UserService {
    private UserMapper userMapper;

    @Inject
    public UserService(UserMapper userMapper) {
        this.userMapper = userMapper;
    }

    public User getUserById(Integer id) {
        return userMapper.findUserById(id);
    }
}
```

这个版本的 `UserService` 只是普通业务类，Spring Security 不认识它。

---

#### 改造后（实现 `UserDetailsService` 接口）

**核心改动：**
1. 加 `implements UserDetailsService`
2. 新增 `Map<String, String> userPasswords` 存用户密码（演示用，生产环境应该查数据库）
3. 实现 `loadUserByUsername()` 方法（Spring Security 要求的）
4. 新增 `save()`、`getPassword()` 辅助方法

**完整代码（第 1-4 张截图的演进过程）：**

```java
package hello.service;

import hello.entity.User;
import org.springframework.security.core.userdetails.UserDetails;
import org.springframework.security.core.userdetails.UserDetailsService;
import org.springframework.security.core.userdetails.UsernameNotFoundException;

import java.util.Collections;
import java.util.Map;
import java.util.concurrent.ConcurrentHashMap;

public class UserService implements UserDetailsService {
    // 用 Map 临时存储用户名和密码（真实项目应该查数据库）
    private Map<String, String> userPasswords = new ConcurrentHashMap<>();

    // 保存用户（注册时调用）
    public void save(String username, String password) {
        userPasswords.put(username, password);
    }

    // 获取密码（仅用于演示，真实项目不暴露）
    public String getPassword(String username) {
        return userPasswords.get(username);
    }

    // 根据 ID 查用户（之前就有的业务方法，保持不变）
    public User getUserById(Integer id) {
        return null;  // 演示阶段先返回 null
    }

    // ↓↓↓ 实现 UserDetailsService 接口要求的方法（Spring Security 登录时会调这个）
    @Override
    public UserDetails loadUserByUsername(String username) throws UsernameNotFoundException {
        // 先看这个用户名存不存在
        if (!userPasswords.containsKey(username)) {
            throw new UsernameNotFoundException(username + " 不存在！");
        }

        // 找到用户后，从 Map 里取出存储的密码
        String password = userPasswords.get(username);

        // 构造并返回 Spring Security 的 User 对象（注意不是你自己的 entity.User）
        return new org.springframework.security.core.userdetails.User(
            username,                  // 用户名
            password,                  // 存储的密码（后面会改成加密版本）
            Collections.emptyList()    // 角色列表（暂时为空，后续可以加）
        );
    }
}
```

> **注意点（截图 1-6 的迭代过程）：**
> - **截图 1-2**：只有 `save()`、`getPassword()`、`getUserById()` 方法，还没实现接口
> - **截图 4-6**：加上 `implements UserDetailsService`，实现 `loadUserByUsername()`
> - **`Collections.emptyList()`**：返回一个空的角色列表（因为目前只关注用户名密码验证，角色权限暂时不管）
> - **`org.springframework.security.core.userdetails.User`**：这是 Spring Security 自带的 `User` 类，不是你的 `hello.entity.User`，注意区分包名

### 步骤 2：给 `UserService` 加上密码加密能力

现在 `save()` 方法存的还是明文密码，要改成加密存储。

#### 2.1 先在 `UserService` 里注入密码加密器

**改动：**
1. 新增字段 `BCryptPasswordEncoder bCryptPasswordEncoder`
2. 构造器注入这个加密器
3. `save()` 方法改成存**加密后的密码**

```java
package hello.service;

import hello.entity.User;
import org.springframework.security.core.userdetails.UserDetails;
import org.springframework.security.core.userdetails.UserDetailsService;
import org.springframework.security.core.userdetails.UsernameNotFoundException;
import org.springframework.security.crypto.bcrypt.BCryptPasswordEncoder;

import javax.inject.Inject;
import java.util.Collections;
import java.util.Map;
import java.util.concurrent.ConcurrentHashMap;

public class UserService implements UserDetailsService {
    private BCryptPasswordEncoder bCryptPasswordEncoder;  // ← 新增
    private Map<String, String> userPasswords = new ConcurrentHashMap<>();

    // 构造器注入密码加密器
    @Inject
    public UserService(BCryptPasswordEncoder bCryptPasswordEncoder) {
        this.bCryptPasswordEncoder = bCryptPasswordEncoder;
        // 演示：启动时存一个测试用户（用户名 gebilaowang，密码加密后存储）
        save("gebilaowang", "gebilaowang");
    }

    // 存密码时先加密
    public void save(String username, String password) {
        userPasswords.put(username, bCryptPasswordEncoder.encode(password));
    }

    public String getPassword(String username) {
        return userPasswords.get(username);
    }

    public User getUserById(Integer id) {
        return null;
    }

    @Override
    public UserDetails loadUserByUsername(String username) throws UsernameNotFoundException {
        if (!userPasswords.containsKey(username)) {
            throw new UsernameNotFoundException(username + " 不存在！");
        }

        String encodedPassword = userPasswords.get(username);

        return new org.springframework.security.core.userdetails.User(
            username,
            encodedPassword,  // ← 注意：这里返回的是加密后的密码
            Collections.emptyList()
        );
    }
}
```

> **注意点（截图 7-8 的演进）：**
> - **截图 7**：只有 `@Service` 注解，还没有构造器注入，也没有加密
> - **截图 11-16**：加入构造器 `@Inject`，注入 `BCryptPasswordEncoder`
> - **构造器里调 `save("gebilaowang", "gebilaowang")`**：演示在 Bean 创建时就存入一个测试用户
> - **`bCryptPasswordEncoder.encode(password)`**：把明文密码加密后再存进 Map

### 步骤 3：配置 `WebSecurityConfig`（两种方式选一种）

现在 `UserService` 已经实现了 `UserDetailsService` 接口，要让 Spring Security 用它。

#### 方式一：在 `JavaConfiguration` 里注册 `UserService`（截图 8）

```java
package hello.configuration;

import hello.service.UserService;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;

@Configuration
public class JavaConfiguration {

    @Bean
    public UserDetailsService userDetailsService() {
        return new UserService();  // ← 返回你自己的 UserService
    }
}
```

但这种方式有问题：`UserService` 需要 `BCryptPasswordEncoder`，而 `new UserService()` 没法传参数。所以截图里这个写法后面报错了。

---

#### 方式二：给 `UserService` 加 `@Service`，在 `WebSecurityConfig` 里注入（截图 11 + 20）

**第 1 步：给 `UserService` 加 `@Service` 注解**

```java
import org.springframework.stereotype.Service;

@Service  // ← 让容器自动扫描注册
public class UserService implements UserDetailsService {
    // ... 其余代码不变
}
```

**第 2 步：在 `WebSecurityConfig` 里用 `@Autowired` 注入 `UserService`**

```java
package hello.configuration;

import hello.service.UserService;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.security.authentication.AuthenticationManager;
import org.springframework.security.config.annotation.web.builders.HttpSecurity;
import org.springframework.security.config.annotation.web.configuration.EnableWebSecurity;
import org.springframework.security.config.annotation.web.configuration.WebSecurityConfigurerAdapter;
import org.springframework.security.core.userdetails.UserDetailsService;
import org.springframework.security.crypto.bcrypt.BCryptPasswordEncoder;

@Configuration
@EnableWebSecurity
public class WebSecurityConfig extends WebSecurityConfigurerAdapter {
    @Autowired
    private UserDetailsService userDetailsService;  // ← 注入你自己的 UserService

    @Override
    protected void configure(HttpSecurity http) throws Exception {
        http.csrf().disable();
        http.authorizeRequests()
            .antMatchers("/", "/auth/**").permitAll()
            .anyRequest().authenticated();
    }

    // ↓↓↓ 注册密码加密器（UserService 需要它）
    @Bean
    public BCryptPasswordEncoder bCryptPasswordEncoder() {
        return new BCryptPasswordEncoder();
    }

    // ↓↓↓ 不需要重写 userDetailsService() 了，因为容器里已经有 UserService Bean
    // Spring Security 会自动找到类型为 UserDetailsService 的 Bean 去用
    
    @Bean
    public AuthenticationManager customAuthenticationManager() throws Exception {
        return authenticationManager();
    }

    // ↓↓↓ 配置全局认证管理器，让它用你注入的 userDetailsService 和密码加密器
    @Autowired
    public void configureGlobal(AuthenticationManagerBuilder auth) throws Exception {
        auth.userDetailsService(userDetailsService)
            .passwordEncoder(bCryptPasswordEncoder());
    }
}
```

> **注意点（截图 14-20 的核心改动）：**
> - **删掉了之前的 `userDetailsService()` 方法**：不再手动 `return new InMemoryUserDetailsManager`
> - **新增 `@Autowired UserDetailsService userDetailsService`**：容器自动注入你的 `UserService`（因为它实现了 `UserDetailsService` 接口且有 `@Service`）
> - **新增 `bCryptPasswordEncoder()` Bean**：返回密码加密器实例
> - **新增 `configureGlobal()` 方法**：告诉 Spring Security："认证时用这个 `userDetailsService` 和这个 `passwordEncoder`"
> - **截图 18-19 的报错**：`There is no PasswordEncoder mapped for the id "null"`，原因是 Spring Security 默认不知道用哪个加密器，必须显式配置 `BCryptPasswordEncoder`

### 步骤 4：理解注册 + 登录的完整流程（手绘图 - 截图 12-13）

```
注册流程（sign up）
用户输入：username="zhangsan", password="123"
    │
    ▼
调用 userService.save("zhangsan", "123")
    │
    ▼
bCryptPasswordEncoder.encode("123")
    │
    └─ 生成加密字符串：$2a$10$abc...xyz（60个字符）
    ▼
存进 Map：userPasswords.put("zhangsan", "$2a$10$abc...xyz")

登录流程（之前 1.6 已实现，现在密码验证走加密比对）
用户输入：username="zhangsan", password="123"
    │
    ▼
AuthController.login() 调用：
    userDetailsService.loadUserByUsername("zhangsan")
    │
    └─ 从 Map 里取出加密后的密码：$2a$10$abc...xyz
    └─ 返回 UserDetails（包含用户名 + 加密密码）
    ▼
authenticationManager.authenticate(token)
    │
    └─ 内部调 passwordEncoder.matches("123", "$2a$10$abc...xyz")
    └─ BCrypt 算法验证：匹配 ✓ → 登录成功
```

**关键点：**
- 注册时：明文 → `encode()` → 加密后存储
- 登录时：用户输入明文 → Spring Security 用 `passwordEncoder.matches()` 比对（不是你手动比对）
- **你不需要在 `AuthController` 里手动调 `encode()`**，`authenticationManager.authenticate()` 内部已经帮你做了

### 易错点汇总（截图里踩过的坑）

| 易错点 | 报错 | 原因 | 解决 |
|-------|------|------|------|
| **没配置 `PasswordEncoder`** | `There is no PasswordEncoder mapped for the id "null"` | Spring Security 不知道用哪个加密器验密码 | 在 `WebSecurityConfig` 里加 `@Bean BCryptPasswordEncoder` |
| **`JavaConfiguration` 里 `new UserService()` 没传参数** | 构造器参数不匹配（如果 `UserService` 构造器需要 `BCryptPasswordEncoder`） | 手动 `new` 无法享受容器注入 | 改用 `@Service` + `@Autowired` 方式 |
| **忘记调 `configureGlobal()`** | 密码验证失败（即使密码对） | Spring Security 没用你配置的 `PasswordEncoder` | 显式调 `auth.userDetailsService(...).passwordEncoder(...)` |

## 第三部分：核心知识点（面试和实战高频）

### Q1：`@Mapper` 的含义作用，没有它会怎样

#### 类比锚定

`@Mapper` 就像给一份**合同盖公章**。

你写了一个接口 `UserMapper`，只有方法签名，没有实现类。MyBatis 说："我帮你实现，但你得在接口上盖个章（`@Mapper`），我才知道这是要我管的。"

#### 有它 vs 没有它

| | 有 `@Mapper` | 没有 `@Mapper` |
|--|-------------|---------------|
| MyBatis 扫描到吗 | 扫描到 | 扫不到 |
| 自动生成实现类吗 | 生成 | 不生成 |
| 能注册成 Bean 吗 | 能（自动注册） | 不能 |
| 能被 `@Inject` 注入吗 | 能 | 不能（容器里找不到这个 Bean） |

**没有它会怎样：** `UserService` 构造器里注入 `UserMapper` 时，容器找不到对应 Bean，启动就报错：

```
***************************
APPLICATION FAILED TO START
***************************

Field userMapper in miniBlog.service.UserService required a bean of type 'miniBlog.mapper.UserMapper' that could not be found.
```

**一句话记忆：** `@Mapper` = 告诉 MyBatis"这个接口归你管"，MyBatis 自动生成实现类并注册成 Bean。

#### 追问：MyBatis 怎么生成实现类，根据什么来生成

**类比：** MyBatis 像个**自动翻译机** — 输入你写的 `@Select("SQL")` 注解，输出包含 JDBC 代码的 Java 类。

**完整生成过程：**

```
第 1 步：Spring Boot 启动时，MyBatis 扫描所有 @Mapper 接口
  └─ 找到：UserMapper

第 2 步：MyBatis 分析接口里的方法和注解
  ├─ 方法名：findUserById
  ├─ 参数：Integer id（有 @Param("id")）
  ├─ 返回类型：User
  └─ SQL 语句：SELECT * FROM user WHERE id = #{id}

第 3 步：用 JDK 动态代理生成实现类（在内存里，不生成 .java 文件）
  └─ 生成的代码逻辑：
      ① 把 #{id} 替换成 ? → PreparedStatement
      ② 执行 SQL → ResultSet
      ③ 把 ResultSet 转成 User 对象（读取字段填充）

第 4 步：把生成的代理类注册成 Bean
  └─ Bean 名字：userMapper（接口名首字母小写）
```

**MyBatis 根据什么来生成：**

| MyBatis 读取的信息 | 用来生成什么代码 |
|-------------------|----------------|
| `@Select("SELECT ...")` | SQL 语句 → `PreparedStatement.execute()` |
| `@Param("id")` | 参数名 → `stmt.setInt(1, id)` |
| 返回类型 `User` | ResultSet 转换 → `new User()` 并填充字段 |
| 字段名（如 `username`） | `rs.getString("username")` → `user.setUsername()` |

**特殊情况：** 如果数据库列名是 `user_name`，Java 字段名是 `userName`，需要配置 `map-underscore-to-camel-case=true`。

### Q2：`@Configuration` 的含义作用，没有它会怎样

（之前在 1.6 文档的 Q8 已经详细解释过，这里简化版）

#### 类比锚定

`@Configuration` 就像**配菜间的规则牌**。

厨师看到这块牌子，就知道："这个房间是配菜的，不是炒菜的；这里的方法是告诉我怎么准备食材，不是给顾客上菜的。"

#### 有它 vs 没有它

| | 有 `@Configuration` | 没有（只有 `@Component`） |
|--|--------------------|-----------------------|
| `@Bean` 方法能注册 Bean 吗 | 能 | 能 |
| 同一个 `@Bean` 方法被调用多次时 | 返回**同一个单例**（Spring 代理拦截） | 每次真的 `new` 新对象（单例破了） |
| 推荐度 | ★★★ 配置类必加 | ★ 只在特殊场景用 |

**例子：**

```java
@Configuration
public class Config {
    @Bean
    public A beanA() { return new A(); }

    @Bean
    public B beanB() { return new B(beanA()); }  // ← 这里调了 beanA()
}
```

**有 `@Configuration`：** `beanB()` 里调 `beanA()` 时，Spring 拦截这次调用，直接从容器里取 A 的单例，不会真的 `new` 一个新 A。

**没有 `@Configuration`：** `beanA()` 真的执行，创建了新对象，单例语义破坏。

**一句话记忆：** `@Configuration` = 让配置类里的 `@Bean` 方法互相调用时走容器单例，而不是真执行方法。

### Q3：`return new org.springframework.security.core.userdetails.User(...)` 是什么意思

#### 类比锚定

Spring Security 说："我要一份**标准格式的用户档案**，必须包含用户名、密码、角色这几栏，你给我填好。"

`org.springframework.security.core.userdetails.User` 就是这个**标准表格模板**。

#### 逐参数拆解

```java
return new org.springframework.security.core.userdetails.User(
    username,              // 第 1 个参数：用户名（如 "gebilaowang"）
    password,              // 第 2 个参数：密码（加密后的，如 "$2a$10$abc..."）
    Collections.emptyList() // 第 3 个参数：角色列表（暂时为空列表）
);
```

| 参数位置 | 类型 | 含义 | 为什么这么写 |
|---------|------|------|------------|
| 第 1 个 | `String username` | 用户名 | 从 `loadUserByUsername(username)` 的参数直接用 |
| 第 2 个 | `String password` | 存储的密码（加密或明文） | 从 `userPasswords.get(username)` 取出来的 |
| 第 3 个 | `Collection<? extends GrantedAuthority>` | 这个用户拥有的角色/权限 | 目前没有角色体系，传空列表 |

#### 为什么包名这么长

因为有两个叫 `User` 的类：

| 类 | 包 | 用途 |
|----|---|------|
| `miniBlog.entity.User` | 你自己的 | 业务数据对象，和数据库表对应 |
| `org.springframework.security.core.userdetails.User` | Spring Security 的 | Spring Security 要求的标准用户对象 |

如果只写 `new User(...)`，编译器不知道你要哪个，所以要写全限定名 `org.springframework.security.core.userdetails.User`。

#### `Collections.emptyList()` 是什么

返回一个**空的、不可变的列表**。

因为 Spring Security 的 `User` 构造器要求传角色列表，但你目前没有角色体系（没有"管理员"、"普通用户"之分），所以传空列表占位。

等价写法（但不推荐）：
```java
new ArrayList<>()  // 可变空列表，占内存
Collections.emptyList()  // 不可变空列表，共享实例，省内存 ✓
```

**一句话记忆：** `new org.springframework.security...User(...)` = 把你的用户信息（用户名+密码+角色）填进 Spring Security 要求的标准表格里。

### Q4：同时存在 2 个 `loadUserByUsername`，会怎么处理

#### 第一层：你缺的前置知识

你缺的是：**Spring 容器里有多个同类型 Bean 时，按什么规则选择哪个来注入。**

#### 全局地图：你项目里实际有几个 `UserDetailsService` Bean

```
容器启动时扫描：

① WebSecurityConfig.userDetailsService() 方法
   └─ 有 @Bean 注解
   └─ 返回类型：UserDetailsService（InMemoryUserDetailsManager 的父接口）
   └─ 注册进容器：Bean 名字 = "userDetailsService"

② miniBlog.service.UserService 类
   └─ 有 @Service 注解
   └─ implements UserDetailsService
   └─ 注册进容器：Bean 名字 = "userService"（类名首字母小写）

结果：容器里有 2 个 UserDetailsService 类型的 Bean！
```

#### 谁会被优先使用

**规则：** 当 `AuthController` 构造器里注入 `UserDetailsService` 时：

```java
@Inject
public AuthController(UserDetailsService userDetailsService, ...) {
    ...
}
```

Spring 容器按以下顺序匹配：

```
① 按类型匹配：找所有 UserDetailsService 类型的 Bean
   └─ 找到 2 个：userDetailsService（内存版）、userService（你的版本）

② 有多个，看参数名是否和某个 Bean 名字一致
   └─ 参数名叫 "userDetailsService"
   └─ 正好有个 Bean 叫 "userDetailsService"（WebSecurityConfig 里的）
   └─ 优先注入这个 ✓

③ 如果参数名也不匹配，看有没有 @Primary
   └─ 都没有 @Primary → 报错（要求你明确指定）
```

**当前实际情况：** 因为你构造器参数名叫 `userDetailsService`，正好匹配 `WebSecurityConfig.userDetailsService()` 这个 Bean，所以用的还是**内存版本**，不是你自己的 `UserService`。

#### 怎么改成用你自己的 `UserService`

**方法一：改参数名（最简单）**

```java
@Inject
public AuthController(UserDetailsService userService, ...) {  // ← 参数名改成 userService
    this.userDetailsService = userService;
}
```

**方法二：删掉 `WebSecurityConfig.userDetailsService()` 方法（推荐）**

把 `WebSecurityConfig` 里第 25-35 行整个删掉，容器里就只剩一个 `UserDetailsService` Bean（你的 `UserService`），不会冲突。

**方法三：用 `@Qualifier` 明确指定**

```java
@Inject
public AuthController(@Qualifier("userService") UserDetailsService userDetailsService, ...) {
    ...
}
```

**一句话记忆：** 容器里有多个同类型 Bean 时，先按参数名匹配 Bean 名字；匹配不上就报错或用 `@Primary`/`@Qualifier` 明确指定。

#### 追问：为什么是按参数名匹配，内存版叫 `userDetailsService`，我的叫 `UserService`？

**纠正：你的理解有误！**

**Bean 名字生成规则：**

| 类/方法 | 注解 | 生成的 Bean 名字 | 规则 |
|--------|------|-----------------|------|
| `UserService` 类 | `@Service` | `userService`（小写 s） | **类名首字母小写** |
| `WebSecurityConfig.userDetailsService()` 方法 | `@Bean` | `userDetailsService` | **方法名** |

**关键：** 你的 `UserService` 类生成的 Bean 名字是 `userService`（小写 s），不是 `UserService`（大写 S）。

**完整匹配过程：**

```java
@Inject
public AuthController(UserDetailsService userDetailsService, ...) {
    // 参数名：userDetailsService
}
```

Spring 容器匹配步骤：

```
第 1 步：按类型找
  └─ 找到 2 个 Bean：["userDetailsService", "userService"]

第 2 步：按参数名匹配 Bean 名字
  ├─ 参数名是 "userDetailsService"
  ├─ "userDetailsService" == "userDetailsService" ✓ 匹配成功！
  └─ 注入内存版的 Bean

如果参数名改成 "userService"：
  ├─ "userService" == "userService" ✓ 匹配成功！
  └─ 注入你的 UserService
```

**验证方法：** 打印所有 Bean 名字：

```java
@SpringBootApplication
public class Application {
    public static void main(String[] args) {
        ConfigurableApplicationContext context = SpringApplication.run(Application.class, args);
        String[] beanNames = context.getBeanNamesForType(UserDetailsService.class);
        for (String name : beanNames) {
            System.out.println("Bean 名字: " + name);
        }
        // 输出：
        // Bean 名字: userDetailsService  ← @Bean 方法名
        // Bean 名字: userService          ← @Service 类名首字母小写
    }
}
```

**总结：** 匹配成功是因为参数名 `userDetailsService` 正好和 `@Bean` 方法名完全一致，而 `@Service` 类的 Bean 名字是类名首字母小写（`userService`），所以不匹配。

**重要提示：** 虽然 Spring 能正确注入，但同时存在 2 个 `UserDetailsService` 会导致 `AuthenticationManager` 配置变复杂，在某些情况下（如使用 3 参数构造 `UsernamePasswordAuthenticationToken`）会触发 `StackOverflowError`。详见下面的 Q5。

### Q5：为什么 `UserService implements UserDetailsService` 后会 `StackOverflowError`？

**问题场景：** 

在"正常版本"代码中：
- `WebSecurityConfig` 有一个 `@Bean UserDetailsService userDetailsService()`（内存版：user/password）
- `WebSecurityConfig` 有一个 `@Bean AuthenticationManager customAuthenticationManager()`
- `AuthController.login()` 使用 3 参数构造 `UsernamePasswordAuthenticationToken`
- **一切正常运行 ✅**

但当你给 `UserService` 加上 `implements UserDetailsService` 后：
- 容器中变成有 **2 个 `UserDetailsService` bean**
- 登录时报 `StackOverflowError`（栈溢出）
- 日志显示 `authenticationManager.authenticate()` 方法无限递归调用

**为什么会这样？** 

#### 第一层理解：2 个 UserDetailsService 的冲突

**正常版本（1 个 UserDetailsService）：**
```
Spring 容器：
├─ userDetailsService → 内存版（user/password）
└─ authenticationManager → 使用上面这个 userDetailsService
```

登录流程简单：
```
1. AuthController 调用 authenticationManager.authenticate(token)
2. AuthenticationManager 用唯一的 userDetailsService 验证
3. 验证完成，返回 ✅
```

**有问题版本（2 个 UserDetailsService）：**
```
Spring 容器：
├─ userDetailsService → 内存版（user/password，密码编码后）
├─ userService → 自定义版（gelilaowang/gelilaowang，明文）
└─ authenticationManager → ？？？它该用哪个？
```

#### 第二层理解：3 参数 token 是"已认证"状态

```java
// AuthController.login() 第 51 行
UsernamePasswordAuthenticationToken token = new UsernamePasswordAuthenticationToken(
    userDetails,                    // 用户信息
    password,                       // 密码
    userDetails.getAuthorities()    // 权限列表 ← 传了这个参数！
);
```

**关键：** 3 参数构造函数会把 token 标记为 `authenticated = true`（已认证）！

**2 参数 vs 3 参数的区别：**

| 构造函数 | authenticated 状态 | 语义 | 何时使用 |
|---------|------------------|------|---------|
| `new UsernamePasswordAuthenticationToken(username, password)` | `false`（未认证） | "我是谁，我的密码是啥，请验证我" | **认证前**，要交给 AuthenticationManager 验证 |
| `new UsernamePasswordAuthenticationToken(userDetails, password, authorities)` | `true`（已认证） | "我已经通过验证了，这是我的身份信息" | **认证后**，验证成功的结果 |

#### 第三层理解：为什么会死循环？

当你有 **2 个 UserDetailsService** + **3 参数 token（已认证）** 时：

```java
// 第 54 行
authenticationManager.authenticate(token);  // token.isAuthenticated() = true
```

`AuthenticationManager` 内部发现：
1. 你传进来的 token 已经是 `authenticated = true`
2. 但容器里有 2 个 `UserDetailsService`，配置变复杂了
3. Spring Security 尝试"重新验证"或"用另一个 Provider 验证"
4. 验证过程中又调用 `authenticate(token)`
5. token 仍然是 `authenticated = true`
6. 再次尝试验证...
7. **无限循环** 💥

**栈溢出日志证据：**
```
at com.sun.proxy.$Proxy69.authenticate(Unknown Source)
at org.springframework.aop.framework.JdkDynamicAopProxy.invoke(...)
at com.sun.proxy.$Proxy69.authenticate(Unknown Source)
at org.springframework.aop.framework.JdkDynamicAopProxy.invoke(...)
...（无限重复，直到栈满）
```

#### 第四层理解：为什么正常版本不会死循环？

**正常版本的关键：** 只有 1 个 `UserDetailsService`

- `AuthenticationManager` 配置简单，只有一条验证路径
- 即使 token 是 `authenticated = true`，也不会触发复杂的"多路径验证"逻辑
- 验证完成后直接返回，不会递归 ✅

**有问题版本的关键：** 2 个 `UserDetailsService` + 3 参数 token

- `AuthenticationManager` 配置变复杂，有多条验证路径
- token 是 `authenticated = true` 触发了特殊的验证逻辑
- 在多路径场景下，某些内部逻辑触发了递归调用
- **组合效果：死循环** 💥

#### 类比锚定：物业验证门禁卡

**正常版本（1 个物业管理员）：**
```
你：出示门禁卡（已激活状态）
物业：看了一眼，确认是本小区的，放行 ✅
```

**有问题版本（2 个物业管理员，规则不同）：**
```
你：出示门禁卡（已激活状态）
物业A：这卡已激活，但我们有2套系统，让我再确认一下...
         → 去问物业B
物业B：已激活的卡？让我查查是哪套系统的...
         → 又去问物业A
物业A：你刚才不是问过了吗，我再确认一下...
         → 又去问物业B
...（无限循环，你永远进不去）
```

#### 解决方案

**方案 1：只保留 1 个 UserDetailsService（推荐）**

移除 `WebSecurityConfig` 中的内存版：
```java
@Configuration
@EnableWebSecurity
public class WebSecurityConfig extends WebSecurityConfigurerAdapter {
    // 删除 userDetailsService() 方法
    
    // 只保留 UserService implements UserDetailsService
}
```

**方案 2：显式指定 AuthenticationManager 使用哪个（如果必须保留 2 个）**

```java
@Configuration
@EnableWebSecurity
public class WebSecurityConfig extends WebSecurityConfigurerAdapter {
    
    @Inject
    private UserService userService;  // 你的自定义版本
    
    @Override
    protected void configure(AuthenticationManagerBuilder auth) throws Exception {
        // 显式指定使用 userService，不是内存版
        auth.userDetailsService(userService);
    }
    
    @Bean
    @Override
    public AuthenticationManager authenticationManagerBean() throws Exception {
        return super.authenticationManagerBean();  // 注意改成 authenticationManagerBean()
    }
}
```

**方案 3：改用 2 参数构造 token（治标不治本）**

```java
// AuthController.login() 第 51 行
UsernamePasswordAuthenticationToken token = new UsernamePasswordAuthenticationToken(
    username, password  // 2 参数 = 未认证状态
);
```

但这只是让死循环变成"验证失败"，根本问题（2 个 UserDetailsService 冲突）仍然存在。

#### 一句话记忆

**多个 `UserDetailsService` + 3 参数 token（已认证）+ `AuthenticationManager` AOP 代理 = 无限递归死循环。**

#### 核心概念速查（通俗版）

下面会详细解释这两个概念：
- **多 Provider**：多个"验证方式"
- **AOP 代理**：Spring 给对象加的"监控外壳"

---

### 附录：核心概念通俗解释

#### 什么是"多 Provider"（多个验证方式）？

**类比：机场安检的多道关卡**

你去坐飞机，安检有多道关卡：
1. **身份证检查台**：核对你的身份证和机票是否匹配
2. **X 光机**：检查行李有没有违禁品
3. **人工抽查**：有时候还会额外人工检查

每一道关卡就是一个 **Provider**（验证提供者）。

**在 Spring Security 中：**

`AuthenticationManager` 就像"安检总调度室"，它手下管着多个 **Provider**：

```
AuthenticationManager（安检总调度）
├─ DaoAuthenticationProvider → 用数据库/内存的用户名密码验证
├─ LdapAuthenticationProvider → 用 LDAP 服务器验证（企业常用）
├─ OAuth2AuthenticationProvider → 用第三方登录（微信/GitHub）
└─ ...其他验证方式
```

**单 Provider vs 多 Provider：**

| 场景 | Provider 数量 | 类比 |
|------|--------------|------|
| 正常版本（1 个 UserDetailsService） | 1 个 Provider | 机场只有 1 道安检，过了就进 |
| 有问题版本（2 个 UserDetailsService） | 2 个 Provider（或更复杂） | 机场有 2 道身份证检查台，规则不同 |

**为什么多 Provider 容易出问题？**

```
你拿着"已激活"的门禁卡（authenticated=true）
         │
         ▼
AuthenticationManager：我有 2 个 Provider，都试试
         │
         ├─ Provider1：这卡已激活，但让我再验证一下密码...
         │    └─ 去查 userDetailsService（内存版）
         │         └─ 密码不匹配（因为内存版是编码后的）
         │              └─ 尝试重新认证？调用 authenticate()
         │                   └─ 又回到 AuthenticationManager...
         │                        └─ 死循环开始 💥
         │
         └─ Provider2：这卡已激活，我也验证一下...
              └─ 去查 userService（自定义版）
                   └─ 又触发某些验证逻辑...
                        └─ 也可能递归调用 authenticate()
```

**一句话记忆：** 多 Provider = 多条验证路径，当配置不清晰时，容易"互相询问"导致死循环。

---

#### 什么是"AOP 代理"（监控外壳）？

**类比：快递代收点**

你在网上买了东西，快递不直接送到你家门口，而是先送到小区门口的"菜鸟驿站"（代收点）。

```
快递员 → 【菜鸟驿站】 → 你
         ↑
      这是"代理"
```

菜鸟驿站能做什么？
- **记录**：记下"几点几分谁的快递到了"
- **增强功能**：帮你验货、拍照、发短信通知
- **转发**：最终还是把快递给你

**在 Spring 中，AOP 代理就是这个"驿站"：**

```java
@Bean
public AuthenticationManager customAuthenticationManager() { ... }
```

Spring 看到 `@Bean`，会给 `customAuthenticationManager` 包一层"代理外壳"：

```
AuthController 调用 authenticationManager.authenticate()
         │
         ▼
   【AOP 代理外壳 $Proxy69】
    （记录日志、事务管理、权限检查等）
         │
         ▼
   真正的 authenticationManager 对象
```

**代理的好处：**
- 你调用方法时，Spring 能"偷偷加料"（记日志、开事务、检查权限）
- 不用改原始代码，就能增强功能

**代理的问题（本次 bug 的罪魁祸首）：**

```java
@Bean
public AuthenticationManager customAuthenticationManager() throws Exception {
    return authenticationManager();  // 调用父类方法
}
```

这行代码的问题：
1. `authenticationManager()` 是父类的 protected 方法
2. 但因为 `customAuthenticationManager()` 有 `@Bean`，Spring 给它加了 AOP 代理
3. 当你调用 `authenticationManager.authenticate()` 时：
   ```
   调用 authenticate()
     ↓
   进入 AOP 代理 $Proxy69
     ↓
   代理尝试调用 authenticationManager()
     ↓
   但 authenticationManager() 又被拦截
     ↓
   又进入 AOP 代理 $Proxy69
     ↓
   又尝试调用 authenticationManager()
     ↓
   无限循环 💥
   ```

**为什么正常版本的代码也有这行，却不死循环？**

因为正常版本只有 **1 个 UserDetailsService**：
- 代理虽然存在，但验证逻辑简单
- `authenticate()` 很快就完成了，不会触发"重新验证"
- 所以不会进入递归

有问题版本有 **2 个 UserDetailsService**：
- 验证逻辑变复杂（多 Provider）
- `authenticate()` 内部尝试"多路验证"
- 某些路径触发了"重新认证"，又调用 `authenticate()`
- **代理 + 多 Provider + 已认证 token = 完美风暴** 💥

**日志里的证据：**
```
at com.sun.proxy.$Proxy69.authenticate(Unknown Source)
        ↑
    这就是 AOP 代理对象
```

**类比总结：**

| 概念 | 类比 | 作用 | 本次 bug 中的角色 |
|------|------|------|------------------|
| **Provider** | 机场的某道安检关卡 | 一种验证方式 | 2 个 UserDetailsService 产生了多个 Provider，验证逻辑混乱 |
| **AOP 代理** | 快递代收点（菜鸟驿站） | 给对象加"监控外壳" | 在多 Provider 场景下，代理的"转发"逻辑触发了递归 |
| **StackOverflowError** | 快递在 2 个驿站之间无限转发 | 递归调用太深，栈满了 | `authenticate()` 无限调用自己 |

**一句话记忆：**
- **多 Provider** = 多个验证方式，像多道安检关卡
- **AOP 代理** = Spring 给对象加的"监控外壳"，像快递代收点
- **组合起来** = 在代理外壳里，多个验证方式互相询问，导致无限转发（死循环）
