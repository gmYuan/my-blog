# 1.9 Cookie 与 Session：登录状态维持

## 第一部分：自顶向下的顶层鸟瞰

### 背景问题

**之前的问题（1.6-1.7）：**
1. `AuthController.auth()` 固定返回"用户没有登录"（无法检查真实登录状态）
2. `AuthController.login()` 返回硬编码数据（`new User(1, "张三")`，不管谁登录都返回张三）

**本节目标：**
1. 改造 `UserService` 数据结构（从只存密码，改为存完整 User 对象）
2. 新增 `getUserByUsername()` 方法（查询真实用户信息）
3. 修改 `AuthController`（返回真实的登录状态和用户信息）
4. 理解 Cookie 和 Session 的工作原理


### 本次改动核心总结

**一句话总结：从"假数据"改为"真数据"。**

**三个主要改动：**
1. **UserService 数据结构升级**：从 `Map<String, String>` 只存密码 → `Map<String, User>` 存完整对象
2. **新增查询方法**：新增 `getUserByUsername()` 方法，可通过用户名查询完整 User 对象
3. **AuthController 返回真实数据**：
   - `auth()` 从固定返回"未登录" → 从 SecurityContext 真实检查登录状态
   - `login()` 从硬编码（张三）→ 返回真实用户信息（gblw）

---

### 核心概念（类比锚定）

**统一类比：银行保险柜系统**

| 概念 | 类比 | 存储位置 | 作用 |
|------|------|---------|------|
| **Cookie** | 银行卡 | 浏览器 | 存储 Session ID，每次请求自动发送 |
| **Session** | 保险柜 | 服务器内存 | 存储 SecurityContext（登录信息） |
| **SecurityContext** | 保险柜里的档案 | Session 中 | 存放 Authentication 对象 |
| **SecurityContextHolder** | 保险柜管理员 | 静态工具类 | 获取当前线程的 SecurityContext |

**完整流程：**
```
登录成功 → Spring Security 创建 Session（保险柜）→ 返回 Cookie（银行卡）
       → 浏览器保存 Cookie → 下次请求自动发送 Cookie
       → 服务器根据 Session ID 查 Session → 恢复 SecurityContext
       → 应用代码通过 SecurityContextHolder 获取登录信息
```

### 步骤 1：修改 UserService - 改用 Map 存 User 对象

**这一步做了什么：** 把 `Map<String, String>` 改成 `Map<String, User>`，新增 `getUserByUsername()` 方法。

**文件：** `src/main/java/miniBlog/service/UserService.java`

**代码改动（diff 风格）：**

```java
@Service
public class UserService implements UserDetailsService {
    private BCryptPasswordEncoder bCryptPasswordEncoder;
-   private Map<String, String> userPasswords = new ConcurrentHashMap<>();  // 旧：只存密码
+   private Map<String, User> users = new ConcurrentHashMap<>();           // 新：存 User 对象

    @Inject

        this.bCryptPasswordEncoder = bCryptPasswordEncoder;
        this.save("gblw", "gblw");
    }

    public void save(String username, String password) {
-       userPasswords.put(username, bCryptPasswordEncoder.encode(password));  // 旧：只存密码
+       users.put(username, new User(1, username, bCryptPasswordEncoder.encode(password)));  // 新：存 User 对象
    }

    public String getPassword(String username) {
-       return userPasswords.get(username);  // 旧：直接从 Map 取密码
+       User user = users.get(username);     // 新：先取 User 对象
+       return user != null ? user.getEncryptedPassword() : null;  // 新：再取密码
    }

+   // 新增方法：根据用户名查询 User 对象
+   public User getUserByUsername(String username) {
+       return users.get(username);
+   }

    @Override
    public UserDetails loadUserByUsername(String username) throws UsernameNotFoundException {
-       if (!userPasswords.containsKey(username)) {  // 旧
+       if (!users.containsKey(username)) {          // 新
            throw new UsernameNotFoundException(username + " 不存在!");
        }
-       String encodedPassword = userPasswords.get(username);  // 旧
+       User user = users.get(username);                       // 新
+       String encodedPassword = user.getEncryptedPassword();  // 新

                encodedPassword,

    }

```

- ✏️ `Map<String, String>` → `Map<String, User>`

### 步骤 2：修改 AuthController - 注入 UserService 并返回真实用户信息

**这一步做了什么：** 
1. 注入 `UserService`（新增）
2. 修改 `auth()` 方法：从 SecurityContext 读取用户名，调用 `getUserByUsername()` 返回真实数据
3. 修改 `login()` 方法：返回真实用户信息（不再硬编码）

**文件：** `src/main/java/miniBlog/controller/AuthController.java`

**代码改动（diff 风格）：**

```java
@Controller
public class AuthController {
+   private UserService userService;  // 新增：注入 UserService
    private UserDetailsService userDetailsService;
    private AuthenticationManager authenticationManager;

    @Inject
-   public AuthController(UserDetailsService userDetailsService,  // 旧
-                         AuthenticationManager authenticationManager) {
+   public AuthController(UserService userService,                       // 新增参数
+                         UserDetailsService userDetailsService,
+                         AuthenticationManager authenticationManager) {
+       this.userService = userService;  // 新增赋值
        this.userDetailsService = userDetailsService;
        this.authenticationManager = authenticationManager;
    }

    @GetMapping("/auth")
    @ResponseBody
    public Object auth() {
-       return new Result("ok", "用户没有登录", false);  // 旧：固定返回未登录
+       
+       // 新增：从 SecurityContext 读取当前用户名
+       String userName = SecurityContextHolder.getContext().getAuthentication().getName();
+       
+       if (userName.contains("anonymous")) {
+           return new Result("ok", "用户没有登录", false);
+       } else {
+           User loggedInUser = userService.getUserByUsername(userName);  // 新增：查询真实用户
+           return new Result("ok", null, true, loggedInUser);
+       }
    }

    @PostMapping("/auth/login")
    @ResponseBody
    public Result login(@RequestBody Map<String, Object> usernameAndPassword) {
        String username = usernameAndPassword.get("username").toString();
        String password = usernameAndPassword.get("password").toString();

        UserDetails userDetails;
        try {
            userDetails = userDetailsService.loadUserByUsername(username);
        } catch (UsernameNotFoundException e) {
            return new Result("fail", "用户不存在", false);
        }

        UsernamePasswordAuthenticationToken token = new UsernamePasswordAuthenticationToken(
            userDetails, password, userDetails.getAuthorities()
        );

        try {
            authenticationManager.authenticate(token);
            SecurityContextHolder.getContext().setAuthentication(token);
            
-           User loggedInUser = new User(1, "张三");  // 旧：硬编码
+           User loggedInUser = userService.getUserByUsername(username);  // 新：查询真实用户
            return new Result("ok", "登录成功", true, loggedInUser);
        } catch (BadCredentialsException e) {
            return new Result("fail", "密码不正确", false);
        }
    }

    // Result 类不变（省略）
}
```

**关键变化：**
- ➕ 新增注入 `UserService`
- ✏️ `auth()` 从固定返回 → 读取 SecurityContext + 查询用户
- ✏️ `login()` 从硬编码 → 查询真实用户

---

**关键代码解释：**

```java
SecurityContextHolder.getContext().getAuthentication().getName()
        ↑                 ↑                 ↑              ↑
     管理员            拿出保险柜        拿出会员卡      看卡上的名字
  (静态工具类)      (返回 Context)   (返回 Auth)    (返回 String)
```

**未登录时：** `getName()` 返回 `"anonymousUser"`，所以用 `userName.contains("anonymous")` 判断。

### 步骤 3：理解 Cookie 和 Session 的完整配合流程

**这一步做了什么：** 理解登录状态持久化的完整链路。

**完整流程：**

```
【登录时】
  ↓
1. 用户：POST /auth/login（username: gblw, password: gblw）
  ↓
2. AuthController.login()：
   ├─ 验证密码 ✓
   └─ SecurityContextHolder.getContext().setAuthentication(token)
       └─ 把登录凭证放进保险柜
  ↓
3. Spring Security 自动：
   ├─ 创建 Session（保险柜编号：ABC123）
   ├─ 把 SecurityContext 存到 Session
   └─ 返回 Cookie: JSESSIONID=ABC123（银行卡）
  ↓
4. 浏览器：自动保存 Cookie
  ↓
─────────────────────────────────────────
【刷新页面时】
  ↓
1. 浏览器：GET /auth + Cookie: JSESSIONID=ABC123（自动发送银行卡）
  ↓
2. Spring Security 自动：
   ├─ 根据 Session ID 查找 Session（找到保险柜 ABC123）
   ├─ 从 Session 恢复 SecurityContext（打开保险柜）
   └─ 放回 SecurityContextHolder（你可以用了）
  ↓
3. AuthController.auth()：
   ├─ SecurityContextHolder.getContext().getAuthentication().getName()
   ├─ 返回 "gblw"（不是 "anonymous"）
   ├─ userService.getUserByUsername("gblw")
   └─ return {"status": "ok", "isLogin": true, "data": {...}}
  ↓
4. 前端：知道用户已登录 ✓
```

---

**一句话记忆：** Cookie 是银行卡（浏览器存），Session 是保险柜（服务器存），SecurityContext 是保险柜里的档案（存登录信息），Spring Security 自动同步。

## 第三部分：核心知识点（面试和实战高频）

### Q1：Cookie 是什么？它是如何工作的？

#### Cookie 的工作原理（银行卡类比）

**第一次办卡：**
```
你去银行开户：
  ↓
银行：给你一张银行卡（卡号：123456）
  └─ 服务器返回：Set-Cookie: JSESSIONID=123456
  ↓
你：把卡放进钱包
  └─ 浏览器自动保存 Cookie
  ↓
下次来银行：刷卡
  └─ 浏览器自动发送：Cookie: JSESSIONID=123456
  ↓
银行：看卡号 → 查系统 → "您是张三，余额 1000 元"
  └─ 服务器根据 Session ID 查 Session → 恢复登录信息
```

---

#### Cookie 的关键属性

| 属性 | 作用 | 示例 |
|------|------|------|
| **Name** | Cookie 的名字 | `JSESSIONID` |
| **Value** | Cookie 的值（Session ID） | `ABC123...` |
| **Path** | Cookie 生效的路径 | `/`（整个网站） |
| **HttpOnly** | 禁止 JavaScript 读取（防 XSS） | `HttpOnly` |
| **Secure** | 只在 HTTPS 下发送 | `Secure` |
| **Max-Age** | 过期时间（秒） | `3600`（1 小时） |

**Spring Boot 默认的 Cookie：**
```
Set-Cookie: JSESSIONID=ABC123; Path=/; HttpOnly
```

---

**面试回答（清晰逻辑线）：**

**A. Cookie 是什么** ==> 浏览器存储的键值对数据，包含 Session ID，用于维持登录状态

**B. 工作流程** ==> 登录成功时，服务器通过 `Set-Cookie` 响应头返回 Cookie ==> 浏览器自动保存 ==> 后续每次请求自动在请求头携带 Cookie ==> 服务器根据 Session ID 识别用户身份

**C. 安全属性** ==> `HttpOnly` 属性禁止 JavaScript 读取，防止 XSS 攻击 ==> `Secure` 属性限制仅在 HTTPS 下发送

### Q2：Session 是什么？它和 Cookie 的关系是什么？

#### Session 的工作原理（保险柜类比）

**为什么需要 Session：**

```
如果把所有信息都存在 Cookie：
  └─ Cookie: username=张三; balance=100000; vip=true; ...
      ↑
   【危险】用户可以篡改浏览器的 Cookie！
   
更安全的方案：
  ├─ Cookie 只存"卡号"（Session ID）
  └─ 敏感信息存在服务器的"保险柜"（Session）
```

---

#### Cookie 和 Session 的配合

```
登录成功：
  ↓
【服务器】创建保险柜
  ├─ 保险柜编号：ABC123
  ├─ 里面放着：SecurityContext
  │   └─ username: 张三, authorities: [USER]
  ↓
【服务器】给你一张银行卡
  └─ Set-Cookie: JSESSIONID=ABC123
  ↓
【浏览器】保存银行卡
  ↓
─────────────────────────────────────────
下次请求：
  ↓
【浏览器】出示银行卡
  └─ Cookie: JSESSIONID=ABC123
  ↓
【服务器】根据卡号找保险柜
  ├─ 查找 Session[ABC123]
  └─ 打开保险柜 → 拿出 SecurityContext → 知道是张三
```

---

#### 对比表格

| 对比维度 | Cookie | Session |
|---------|--------|---------|
| **存储位置** | 浏览器（客户端） | 服务器（内存/Redis） |
| **存储内容** | 保险柜编号（Session ID） | 用户档案（SecurityContext） |
| **大小限制** | 4KB | 无限制 |
| **安全性** | 低（用户可见可改） | 高（用户看不到） |
| **生命周期** | 可持久化（Max-Age） | 默认 30 分钟无活动失效 |

---

**面试回答（清晰逻辑线）：**

**A. Session 是什么** ==> 服务器端存储的用户登录状态数据，以 Session ID 作为唯一标识，存储在服务器内存中

**B. 创建和存储流程** ==> 用户登录成功后，Spring Security 自动创建 Session ==> 生成随机 Session ID（如 JSESSIONID=ABC123） ==> 将 SecurityContext 对象存入 Session ==> 通过 `Set-Cookie` 响应头将 Session ID 发送给浏览器

**C. 查询和恢复流程** ==> 后续请求中浏览器携带 Cookie 中的 Session ID ==> 服务器根据 ID 查找对应 Session ==> 恢复用户的 SecurityContext ==> 获取登录状态

**D. 生命周期** ==> 默认 30 分钟无活动失效 ==> 存储在服务器内存中，服务器重启会丢失

### Q3：Spring Security 是如何自动处理 Session 的？

#### 先理解问题：我们的代码里没有创建 Session 啊？

**回顾我们写的代码：**
```java
SecurityContextHolder.getContext().setAuthentication(token);
```

**疑问：**
- 只是把 token 放进了 `SecurityContext`
- 没有写"创建 Session""返回 Cookie"的代码
- 那 Session 和 Cookie 是怎么来的？

---

#### Spring Security 在背后自动做的事

**类比：你去银行存钱**

```
你的操作：
  └─ "我要把 100 元存进去"
  
银行自动做的事：
  ├─ 创建账户（如果没有）
  ├─ 生成银行卡号
  ├─ 给你一张银行卡
  ├─ 把钱存到保险柜
  └─ 关联卡号和保险柜
```

---

**Spring Security 的自动化流程：**

```
你的代码：
  ↓
SecurityContextHolder.getContext().setAuthentication(token);
  ↓
【Spring Security 自动做】
  ↓
步骤 1：检查是否有 Session
  ├─ 如果没有 → 创建新 Session（保险柜编号：ABC123）
  └─ 如果有 → 用现有 Session
  ↓
步骤 2：把 SecurityContext 存到 Session
  └─ Session[ABC123] = SecurityContext
      └─ Authentication(username: gblw, authorities: [])
  ↓
步骤 3：返回 Cookie（通过 HTTP 响应头）
  └─ Set-Cookie: JSESSIONID=ABC123; Path=/; HttpOnly
  ↓
步骤 4：下次请求自动恢复
  ├─ 浏览器发送：Cookie: JSESSIONID=ABC123
  ├─ Spring Security 拦截请求
  ├─ 根据 Session ID 查找 Session
  ├─ 从 Session 拿出 SecurityContext
  └─ 放回 SecurityContextHolder（你的代码才能用）
```

**关键：你不需要写任何代码，Spring Security 全自动！**

---

#### Session 的生命周期

**创建时机：**
- 用户登录成功（调用 `setAuthentication()` 后）
- 用户首次访问（Spring Boot 默认）

**销毁时机：**
- 用户主动退出（调用 `/logout`）
- Session 超时（默认 30 分钟无活动）
- 服务器重启（Session 存在内存，会丢失）

**配置超时时间：**
```properties
# application.properties
server.servlet.session.timeout=1h  # 1 小时
```

---

**面试回答（清晰逻辑线）：**

**A. Spring Security 自动化机制** ==> 调用 `SecurityContextHolder.getContext().setAuthentication(token)` 时 ==> Spring Security 自动检查当前请求是否有 Session ==> 如果没有则创建新 Session 并生成随机 Session ID ==> 将 SecurityContext 对象存入 Session ==> 通过 `Set-Cookie` 响应头返回 Session ID

**B. 自动恢复机制** ==> 后续请求中，Spring Security 的过滤器链自动拦截请求 ==> 从 Cookie 中读取 Session ID ==> 根据 ID 查找 Session ==> 取出 SecurityContext 并放回 SecurityContextHolder ==> 应用代码可以直接使用

**C. 关键特点** ==> 整个过程完全自动化，无需手动编写 Session 创建和恢复的代码

### Q4：CSRF 攻击是什么？我需要担心吗？

#### CSRF 攻击的场景（银行转账类比）

**你的正常操作：**
```
你在银行网站登录 → 手动点"转账"按钮 → 转账成功
  └─ 这是【你主动发起的请求】✓
```

---

**CSRF 攻击（你被利用了）：**

```
你登录了银行网站（已经有银行卡/Cookie）
  ↓
你在另一个网页看新闻（恶意网站）
  ↓
【恶意网站偷偷执行】：
  <form action="https://bank.com/transfer" method="POST">
    <input name="to" value="黑客账户">
    <input name="amount" value="10000">
  </form>
  <script>document.forms[0].submit()</script>
  ↓
【浏览器】自动发送请求到银行网站
  └─ 自动带上你的银行卡（Cookie）！
  ↓
【银行服务器】看到银行卡 → 以为是你发的 → 转账成功！
  ↓
你：？？？我没转账啊！
```

**关键问题：** 浏览器会自动带上 Cookie，服务器分不清是"你点的"还是"恶意网站伪造的"。

---

#### CSRF Token 防护（短信验证码类比）

**类比：银行的短信验证码**

```
你在手机银行转账：
  ↓
【银行】发短信给你：验证码 123456
  ↓
你输入验证码 → 转账成功
  ↓
─────────────────────────────────────────
黑客想伪造转账：
  ↓
黑客发请求：转 10000 元
  ↓
【银行】要求验证码
  ↓
黑客：没有！（短信发到你手机）
  ↓
银行拒绝 ✗
```

**CSRF Token 就是"短信验证码"：**

```
你访问页面：
  ↓
【服务器】生成随机 Token（XYZ789）
  ├─ 存到 Session
  └─ 返回给你的页面（藏在表单里）
  ↓
你提交表单：
  ↓
【浏览器】发送请求
  ├─ Cookie: JSESSIONID=ABC123（自动带上）
  └─ Body: {amount: 100, _csrf: "XYZ789"}（表单数据）
  ↓
【服务器】验证两样东西
  ├─ Cookie（JSESSIONID）→ 确认你登录了 ✓
  └─ CSRF Token（XYZ789）→ 确认是你的页面发的 ✓
  ↓
两者都对 → 请求通过 ✓
  ↓
─────────────────────────────────────────
恶意网站伪造请求：
  ↓
【浏览器】发送请求
  ├─ Cookie: JSESSIONID=ABC123（自动带上）✓
  └─ Body: {amount: 10000}
      └─ 没有 CSRF Token！✗
  ↓
【服务器】拒绝（403 Forbidden）
```

---

#### 为什么我们的代码要 `csrf().disable()`？

**我们的代码（WebSecurityConfig）：**
```java
http.csrf().disable();  // ← 关闭 CSRF 防护
```

**原因：我们用的是前后端分离（RESTful API）**

| 场景 | 是否需要 CSRF | 原因 |
|------|-------------|------|
| **传统 MVC（JSP/Thymeleaf）** | ✅ 需要 | 浏览器自动提交表单，容易被攻击 |
| **前后端分离（RESTful API）** | ⚠️ 看情况 | 如果用 Cookie 认证，理论上有风险 |
| **JWT Token 认证** | ❌ 不需要 | 不用 Cookie，CSRF 攻击无效 |

**我们暂时关闭的原因：**
1. 简化学习（不需要在前端处理 CSRF Token）
2. 前后端分离，风险相对较低

**实际项目建议：** 如果用 Cookie 认证，最好开启 CSRF 防护。

---

**面试回答（清晰逻辑线）：**

**A. CSRF 攻击原理** ==> 恶意网站利用用户浏览器自动发送 Cookie 的特性，伪造合法请求 ==> 例如：用户登录银行网站后访问恶意网站 ==> 恶意网站嵌入隐藏表单向银行发起转账请求 ==> 浏览器自动携带银行的 Cookie ==> 服务器误以为是合法请求

**B. 防护方案 - CSRF Token** ==> 服务器返回页面时生成随机 Token ==> 同时存入 Session 和页面表单中 ==> 提交时验证 Token 是否匹配 ==> 恶意网站无法获取 Token，请求被拒绝

**C. 我们为什么关闭** ==> 前后端分离架构下，前端通过 AJAX 发送 JSON 请求，不是传统表单提交 ==> 为简化学习暂时使用 `csrf().disable()` 关闭 ==> 实际生产环境中，如果使用 Cookie 认证，应该开启 CSRF 防护

---

### Q5：如果有多台服务器，Session 会丢失吗？怎么解决？

#### 先理解问题：单台 vs 多台服务器

**单台服务器（目前的情况）：**
```
服务器 A (localhost:8080)
  └─ Session[ABC123] = 张三的登录信息
  ↓
用户访问 → 服务器 A → 查到 Session → 登录状态正常 ✓
```

**没问题！**

---

**多台服务器（生产环境）：**

```
公司部署了 3 台服务器（负载均衡）：
  ↓
服务器 A：Session[ABC123] = 张三
服务器 B：没有 ABC123
服务器 C：没有 ABC123
  ↓
用户第 1 次请求 → 负载均衡 → 服务器 A → 登录成功 ✓
用户第 2 次请求 → 负载均衡 → 服务器 B → 提示未登录 ✗
```

**问题：** Session 只在一台服务器的内存里。

---

#### 解决方案：用 Redis 存 Session（共享保险柜）

**类比：银行的共享保险柜系统**

```
错误做法（各分行各存各的）：
  ├─ A 分行：张三的保险柜
  ├─ B 分行：李四的保险柜
  └─ C 分行：王五的保险柜
      ↓
  张三在 B 分行刷卡 → 查不到保险柜 ✗
  ↓
─────────────────────────────────────────
正确做法（统一存到中央金库）：
  ↓
所有分行连接同一个 Redis（中央金库）：
  ├─ Session[ABC123] = 张三
  ├─ Session[XYZ789] = 李四
  └─ Session[QWE456] = 王五
  ↓
张三在任何分行刷卡 → 都能查到 ✓
```

---

#### 配置 Redis Session（3 步）

**你现在不需要做，但了解一下：**

**步骤 1：添加依赖（`pom.xml`）**
```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-data-redis</artifactId>
</dependency>
<dependency>
    <groupId>org.springframework.session</groupId>
    <artifactId>spring-session-data-redis</artifactId>
</dependency>
```

**步骤 2：配置 Redis（`application.properties`）**
```properties
spring.redis.host=localhost
spring.redis.port=6379
spring.session.store-type=redis
```

**步骤 3：启动 Redis**
```bash
docker run -d -p 6379:6379 redis
```

---

**面试回答（清晰逻辑线）：**

**A. 单台服务器** ==> Session 存储在服务器内存中，无需额外配置

**B. 多台服务器问题** ==> 负载均衡将用户请求分配到不同服务器 ==> Session 只存在创建它的服务器内存中 ==> 用户可能在不同服务器间出现"已登录"和"未登录"状态交替

**C. 解决方案 - Redis 集中式存储** ==> 使用 Redis 作为集中式 Session 存储 ==> 所有服务器连接同一个 Redis 实例 ==> Spring Session 自动将 Session 数据序列化后存入 Redis

**D. 实现步骤** ==> 添加 `spring-boot-starter-data-redis` 和 `spring-session-data-redis` 依赖 ==> 配置 `spring.session.store-type=redis` 和 Redis 连接信息 ==> 启动 Redis 服务 ==> Session 的创建和读取逻辑完全不变，Spring Session 自动处理

## 第四部分：学习 Q&A

### Q1：项目全局是怎么运行的？Controller 和 Service 的关系是什么？

#### 全局运行流程（登录场景）

```
【1】用户在浏览器：
  └─ POST http://localhost:8080/auth/login
      └─ Body: {"username": "gblw", "password": "gblw"}
  ↓
【2】Tomcat 接收请求 → DispatcherServlet 路由
  └─ 找到 @PostMapping("/auth/login") → AuthController.login()
  ↓
【3】AuthController（接待员）：
  ├─ 解析参数：username, password
  ├─ 调用 userService.loadUserByUsername()（查用户）
  ├─ 调用 authenticationManager.authenticate()（验证密码）
  ├─ 调用 SecurityContextHolder.setContext()（存登录状态）
  └─ 调用 userService.getUserByUsername()（查完整用户信息）
  ↓
【4】UserService（厨师）：
  ├─ loadUserByUsername()：从 Map 查密码，返回 UserDetails
  └─ getUserByUsername()：从 Map 查用户，返回 User 对象
  ↓
【5】AuthController 返回响应：
  └─ return new Result("ok", "登录成功", true, user);
  ↓
【6】Spring Boot（Jackson）自动转 JSON：
  └─ {"status": "ok", "msg": "登录成功", "isLogin": true, "data": {...}}
  ↓
【7】返回给浏览器
```

---

#### Controller 和 Service 的职责

**类比：餐厅的服务员和厨师**

| 角色 | 职责 | 不应该做的事 |
|------|------|------------|
| **Controller（服务员）** | 接待客人、记录点单、上菜 | ❌ 不做菜（不写业务逻辑） |
| **Service（厨师）** | 准备食材、烹饪、做菜 | ❌ 不接待客人（不处理 HTTP） |
| **Mapper（仓库管理员）** | 取食材、管理库存 | ❌ 不做菜（不写业务逻辑） |

---

**什么写在 Controller：**
```java
@Controller
public class AuthController {
    ✅ 接收请求（@GetMapping、@PostMapping）
    ✅ 解析参数（@RequestBody、@PathVariable）
    ✅ 调用 Service（userService.getUserByUsername()）
    ✅ 返回响应（return new Result()）
    
    ❌ 不写数据库查询
    ❌ 不写复杂业务逻辑
    ❌ 不写密码加密
}
```

**什么写在 Service：**
```java
@Service
public class UserService {
    ✅ 业务逻辑（密码加密、数据校验）
    ✅ 数据查询（调用 Mapper 或操作 Map）
    ✅ 数据处理（格式转换、计算）
    
    ❌ 不处理 HTTP 请求
    ❌ 不返回 JSON
    ❌ 不写 @GetMapping/@PostMapping
}
```

---

**完整分层架构：**
```
用户请求
  ↓
【Controller 层】服务员（接待、上菜）
  ├─ AuthController（认证）
  ├─ UserController（用户管理）
  └─ BlogController（博客）
  ↓
【Service 层】厨师（做菜）
  ├─ UserService（用户业务）
  └─ BlogService（博客业务）
  ↓
【Mapper 层】仓库管理员（取食材）
  ├─ UserMapper（用户表）
  └─ BlogMapper（博客表）
  ↓
【数据库】仓库（MySQL / Redis）
```

---

**一句话记忆：** Controller = 服务员（接待、上菜），Service = 厨师（做菜），Mapper = 仓库管理员（取食材）。

### Q2：SecurityContext、SecurityContextHolder、getContext()、getAuthentication() 到底是什么？

#### 统一类比：银行保险柜系统

```
SecurityContextHolder（保险柜管理中心）
  └─ 全局唯一，静态工具类
      ↓
      .getContext()（找到你的保险柜）
        └─ 静态方法，返回 SecurityContext
            ↓
            SecurityContext（你的保险柜）
              └─ 接口，存放 Authentication
                  ↓
                  .getAuthentication()（从保险柜拿出会员卡）
                    └─ 实例方法，返回 Authentication
                        ↓
                        Authentication（会员卡）
                          └─ 接口，包含用户名、权限
                              ↓
                              .getName()（看会员卡上的名字）
                                └─ 实例方法，返回 String
```

---

#### 详细解释

**1. SecurityContextHolder（保险柜管理中心）**

- **是什么：** 静态工具类（全局唯一）
- **作用：** 管理所有 `SecurityContext`
- **类比：** 银行的"保险柜管理中心"
- **代码：** `SecurityContextHolder`（类名）

---

**2. SecurityContext（保险柜）**

- **是什么：** 接口，存放 `Authentication`
- **作用：** 容器，装着登录凭证
- **类比：** 你的"保险柜"
- **代码：** `SecurityContextHolder.getContext()`（返回保险柜对象）

---

**3. getContext()（找到保险柜）**

- **是什么：** `SecurityContextHolder` 的静态方法
- **作用：** 返回当前线程的 `SecurityContext`
- **类比：** 管理员帮你"打开保险柜"
- **代码：** `SecurityContextHolder.getContext()`

---

**4. Authentication（会员卡）**

- **是什么：** 接口，登录凭证
- **作用：** 包含用户名、权限、详细信息
- **类比：** 保险柜里的"会员卡"
- **代码：** `.getAuthentication()`（从保险柜拿出会员卡）

---

**5. getName()（看会员卡上的名字）**

- **是什么：** `Authentication` 的实例方法
- **作用：** 返回用户名（String）
- **类比：** 看会员卡上的"名字"
- **代码：** `.getName()`

---

#### 完整调用链拆解

```java
// 原始代码（链式调用）
String userName = SecurityContextHolder.getContext().getAuthentication().getName();

// 拆解成 4 步：
SecurityContextHolder              // 步骤 1：保险柜管理中心（静态类）
    .getContext()                   // 步骤 2：找到你的保险柜（返回 SecurityContext）
    .getAuthentication()            // 步骤 3：从保险柜拿出会员卡（返回 Authentication）
    .getName();                     // 步骤 4：看会员卡上的名字（返回 String）
```

---

#### 为什么要这么设计？（多层嵌套）

**类比：银行卡 → 保险柜 → 会员卡 → 名字**

```
为什么不直接把名字印在银行卡上：
  └─ 因为需要存更多信息（权限、登录时间等）
  
为什么不把所有信息都存在 Cookie：
  └─ 不安全，用户可以篡改
  
为什么分这么多层：
  └─ 灵活扩展（可以换成其他认证方式）
```

---

#### 实际使用场景

**场景 1：检查登录状态**
```java
String userName = SecurityContextHolder.getContext().getAuthentication().getName();
if (userName.contains("anonymous")) {
    // 未登录
} else {
    // 已登录
}
```

**场景 2：登录时存储**
```java
SecurityContextHolder.getContext().setAuthentication(token);
// Spring Security 自动把 SecurityContext 存到 Session
```

---

#### 为什么刷新页面，登录状态还在？

```
【登录时】
  ↓
SecurityContextHolder.getContext().setAuthentication(token)
  ↓
Spring Security 自动：
  ├─ 把 SecurityContext 存到 Session[ABC123]
  └─ 返回 Cookie: JSESSIONID=ABC123
  ↓
─────────────────────────────────────────
【刷新页面时】
  ↓
浏览器自动发送 Cookie: JSESSIONID=ABC123
  ↓
Spring Security 自动：
  ├─ 根据 Session ID 查 Session
  ├─ 从 Session 恢复 SecurityContext
  └─ 放回 SecurityContextHolder
  ↓
你的代码调用 SecurityContextHolder.getContext()
  └─ 拿到的是恢复后的 SecurityContext（有登录信息）
```

---

**一句话记忆：** 保险柜管理中心（Holder）→ 找到保险柜（getContext）→ 拿出会员卡（getAuthentication）→ 看名字（getName），Spring Security 自动同步到 Session。

### Q3：为什么 AuthController 要引入 UserService？为什么还需要 UserDetailsService？

#### 两个 Service 的作用对比

| Service | 作用 | 谁用它 | 返回类型 | 什么时候用 |
|---------|------|--------|---------|----------|
| **UserDetailsService** | 验证密码时查用户 | Spring Security | `UserDetails` | `authenticate()` 自动调用 |
| **UserService** | 查询完整用户信息 | 你的代码 | `User` | 需要返回用户信息时手动调用 |

---

#### UserDetailsService 的作用（不能删）

```java
@PostMapping("/auth/login")
public Result login(...) {
    // 验证密码时，Spring Security 需要 UserDetailsService
    UserDetails userDetails = userDetailsService.loadUserByUsername(username);
    UsernamePasswordAuthenticationToken token = new UsernamePasswordAuthenticationToken(...);
    authenticationManager.authenticate(token);  // ← 内部会调用 UserDetailsService
}
```

**如果删掉：** 无法验证密码，登录功能报错。

---

#### UserService 的作用（新增）

```java
@GetMapping("/auth")
public Object auth() {
    String userName = SecurityContextHolder.getContext().getAuthentication().getName();
    // 需要返回完整 User 对象（不只是密码）
    User loggedInUser = userService.getUserByUsername(userName);  // ← 需要这个方法
    return new Result("ok", null, true, loggedInUser);
}
```

**为什么 UserDetailsService 不够用：**
- `UserDetailsService` 只返回 `UserDetails`（用于验证密码）
- 我们需要返回 `User` 对象（业务实体，包含 id、username 等）

---

#### 能不能只注入一个 Service？

**可以！因为 `UserService implements UserDetailsService`：**

```java
@Service
public class UserService implements UserDetailsService {
    // UserDetailsService 的方法（给 Spring Security 用）
    @Override
    public UserDetails loadUserByUsername(String username) { ... }
    
    // 业务方法（给 Controller 用）
    public User getUserByUsername(String username) { ... }
}
```

**所以 AuthController 可以只注入 UserService：**
```java
@Inject
public AuthController(UserService userService, AuthenticationManager authenticationManager) {
    this.userService = userService;
    this.authenticationManager = authenticationManager;
}
```

**截图里为什么注入了两个：** 可能是演示用，实际只注入 `UserService` 就够了。
