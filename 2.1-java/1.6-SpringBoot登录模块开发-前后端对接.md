### 登录流程图

```
前端（Vue 静态页面）
   │
   │  POST /auth/login
   │  { "username": "user", "password": "password" }
   ▼
AuthController.login()
   │
   ├─① 从请求 JSON 取 username / password（@RequestBody）
   │
   ├─② userDetailsService.loadUserByUsername(username)
   │       ├─ 找不到用户 → return Result("fail", "用户不存在", false)
   │       └─ 找到 → 继续
   │
   ├─③ 构造 UsernamePasswordAuthenticationToken(userDetails, password, authorities)
   │
   ├─④ authenticationManager.authenticate(token)
   │       ├─ 密码错误 → return Result("fail", "密码不正确", false)
   │       └─ 验证通过 → 继续
   │
   ├─⑤ SecurityContextHolder.getContext().setAuthentication(token)  // 保存登录状态
   │
   └─⑥ return Result("ok", "登录成功", true, loggedInUser)

   ─────────────────────────────────────────────────────────────
   GET /auth → return Result("ok", "用户没有登录", false)
```

### 技术栈关键词速查

| 关键词 | 一句话 |
|--------|--------|
| `@Service` | 让 Spring 自动扫描注册该类为 Bean |
| `@RestController` | 等于 `@Controller` + 全局 `@ResponseBody`，返回 JSON |
| `@RequestBody` | 把前端发来的 JSON 转成 Java 对象 |
| `@ResponseBody` | 把 Java 对象转成 JSON 返回前端 |
| `spring-boot-starter-security` | Spring Security 依赖 |
| `WebSecurityConfig` | 安全规则配置类（放行路径、CSRF 等） |

| `UsernamePasswordAuthenticationToken` | 封装用户名/密码/权限的认证凭证 |
| `SecurityContextHolder` | 保存当前请求的登录状态 |
| Jackson | Spring Boot 内置的 JSON 序列化库，自动调用 getter |

## 第一部分：实战步骤（可直接照做）

### 步骤 1：切换 `UserService` 的注册方式

之前用 `JavaConfiguration` 的 `@Bean` 方法手动注册 `UserService`，现在改成**直接在类上加注解**，让 Spring 自动扫描。

#### `JavaConfiguration.java` — 注释掉 `@Bean`

```java
package hello.configuration;

import org.springframework.context.annotation.Configuration;

@Configuration
public class JavaConfiguration {
    // @Bean 方法已注释掉，UserService 改用 @Service 注解自动注册
    // @Bean
    // public UserService userService(UserMapper userMapper) {
    //     return new UserService(userMapper);
    // }
}
```

#### `UserService.java` — 加 `@Service`

```java
package hello.service;

import hello.mapper.UserMapper;
import org.springframework.stereotype.Component;
import org.springframework.stereotype.Service;

import javax.inject.Inject;

@Service
@Component
public class UserService {
    private UserMapper userMapper;

    @Inject
    public UserService(UserMapper userMapper) {
        this.userMapper = userMapper;
    }

    public User getUserById(Integer id) {
        return userMapper.findUserById(id);
    }
}
```

> **注意：** `@Service` 和 `@Component` 功能一样，加一个就够了，通常只用 `@Service`。截图里同时加了两个是演示对比，实际项目只保留 `@Service`。

### 步骤 4：创建安全配置类 `WebSecurityConfig.java`

```java
package hello.configuration;

import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.security.authentication.AuthenticationManager;
import org.springframework.security.config.annotation.web.builders.HttpSecurity;
import org.springframework.security.config.annotation.web.configuration.EnableWebSecurity;
import org.springframework.security.config.annotation.web.configuration.WebSecurityConfigurerAdapter;
import org.springframework.security.core.userdetails.User;
import org.springframework.security.core.userdetails.UserDetails;
import org.springframework.security.core.userdetails.UserDetailsService;
import org.springframework.security.provisioning.InMemoryUserDetailsManager;

@Configuration
@EnableWebSecurity
public class WebSecurityConfig extends WebSecurityConfigurerAdapter {

    @Override
    protected void configure(HttpSecurity http) throws Exception {
        http.csrf().disable();

        http.authorizeRequests()
            .antMatchers("/", "/auth/**").permitAll();
    }

    @Bean
    @Override
    public UserDetailsService userDetailsService() {
        UserDetails user =
            User.withDefaultPasswordEncoder()
                .username("user")
                .password("password")
                .roles("USER")
                .build();
        return new InMemoryUserDetailsManager(user);
    }

    @Bean
    public AuthenticationManager customAuthenticationManager() throws Exception {
        return authenticationManager();
    }
}
```

> **注意点：**
> - `http.csrf().disable()`：关掉 CSRF 保护，否则 POST 请求会被拦截（403 错误）
> - `.antMatchers("/", "/auth/**").permitAll()`：`/auth` 和 `/auth/login` 必须放行，不然登录接口自身也需要先登录才能访问
> - `userDetailsService()` 目前用内存里的测试用户（用户名 `user`，密码 `password`），后续接真实数据库后替换
> - `customAuthenticationManager()` 把 `AuthenticationManager` 暴露为 Bean，`AuthController` 需要注入它来验证密码

### 步骤 6：创建 `AuthController.java`（完整版）

```java
package hello.controller;

import hello.entity.User;
import org.springframework.security.authentication.AuthenticationManager;
import org.springframework.security.authentication.BadCredentialsException;
import org.springframework.security.authentication.UsernamePasswordAuthenticationToken;
import org.springframework.security.core.context.SecurityContextHolder;
import org.springframework.security.core.userdetails.UserDetails;
import org.springframework.security.core.userdetails.UserDetailsService;
import org.springframework.security.core.userdetails.UsernameNotFoundException;
import org.springframework.stereotype.Controller;
import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.PostMapping;
import org.springframework.web.bind.annotation.RequestBody;
import org.springframework.web.bind.annotation.ResponseBody;

import javax.inject.Inject;
import java.util.Map;

@Controller
public class AuthController {
    private UserDetailsService userDetailsService;
    private AuthenticationManager authenticationManager;

    @Inject
    public AuthController(UserDetailsService userDetailsService,
                          AuthenticationManager authenticationManager) {
        this.userDetailsService = userDetailsService;
        this.authenticationManager = authenticationManager;
    }

    // GET /auth：判断当前用户是否登录
    @GetMapping("/auth")
    @ResponseBody
    public Object auth() {
        return new Result("ok", "用户没有登录", false);
    }

    // POST /auth/login：登录接口
    @PostMapping("/auth/login")
    @ResponseBody
    public Result login(@RequestBody Map<String, String> usernameAndPassword) {
        String username = usernameAndPassword.get("username").toString();
        String password = usernameAndPassword.get("password").toString();

        // 第一步：查用户是否存在
        UserDetails userDetails = null;
        try {
            userDetails = userDetailsService.loadUserByUsername(username);
        } catch (UsernameNotFoundException e) {
            return new Result("fail", "用户不存在", false);
        }

        // 第二步：构造认证 Token
        UsernamePasswordAuthenticationToken token =
            new UsernamePasswordAuthenticationToken(
                userDetails, password, userDetails.getAuthorities()
            );

        // 第三步：验证密码
        try {
            authenticationManager.authenticate(token);
            // 验证通过：保存登录状态
            SecurityContextHolder.getContext().setAuthentication(token);

            User loggedInUser = new User(1, "张三");  // 目前写死，后续从数据库查
            return new Result("ok", "登录成功", true, loggedInUser);
        } catch (BadCredentialsException e) {
            return new Result("fail", "密码不正确", false);
        }
    }

    // 统一的返回结构（内部类）
    private static class Result {
        String status;
        String msg;
        boolean isLogin;
        Object data;

        public Result(String status, String msg, boolean isLogin) {
            this(status, msg, isLogin, null);
        }

        public Result(String status, String msg, boolean isLogin, Object data) {
            this.status = status;
            this.msg = msg;
            this.isLogin = isLogin;
            this.data = data;
        }

        public String getStatus() { return status; }
        public String getMsg() { return msg; }
        public boolean getIsLogin() { return isLogin; }
        public Object getData() { return data; }
    }
}
```

> **注意点：**
> - `@Controller` + `@ResponseBody` 的组合：`@ResponseBody` 必须加，否则 Spring 会把返回值当成视图名去找 HTML 模板；或者直接改用 `@RestController` 省掉每个方法加 `@ResponseBody`
> - `@RequestBody`：把前端发来的 JSON 自动转成 `Map<String, String>`
> - 登录逻辑分两层 try-catch：先查用户存不存在（`UsernameNotFoundException`），再验密码（`BadCredentialsException`），给出不同错误信息
> - `Result` 有两个构造器：不传 `data` 时默认 `null`，登录成功时传入用户对象

## 第二部分：核心知识点（面试 / 工作常问）

### 知识点 1：`@Controller` vs `@RestController`

| 注解 | 等价于 | 返回值默认行为 |
|------|--------|----------------|
| `@Controller` | 只标注为控制器 | 返回视图名（找 HTML 模板），想返回 JSON 需额外加 `@ResponseBody` |
| `@RestController` | `@Controller` + `@ResponseBody` | 直接返回 JSON，不找模板 |

**实际工作：** 写纯 API 接口时，直接用 `@RestController`，省掉每个方法都加 `@ResponseBody`。

---

### 知识点 2：`@RequestBody` vs `@ResponseBody`

| 注解 | 方向 | 作用 |
|------|------|------|
| `@RequestBody` | 前端 → 后端 | 把前端发来的 JSON 自动转成 Java 对象 |
| `@ResponseBody` | 后端 → 前端 | 把 Java 对象自动转成 JSON 返回 |

---

### 知识点 3：Authentication vs Authorization（面试高频）

| 英文 | 中文 | 含义 | 例子 |
|------|------|------|------|
| **Authentication** | 认证（身份验证） | 你是谁？你是不是你本人？ | 输入用户名密码登录 |
| **Authorization** | 授权（权限控制） | 你能干什么？你有权访问这个资源吗？ | 普通用户不能访问管理员后台 |

**记忆口诀：** 先认证（登录验身份），再授权（判断能干啥）。两者缺一不可。

---

### 知识点 4：Spring Security 四大核心组件

| 组件 | 作用 | 你负责什么 |
|------|------|------------|
| `UserDetailsService` | 定义"怎么查用户" | 实现 `loadUserByUsername()`，返回用户信息 |
| `AuthenticationManager` | 执行认证（比对密码） | 注入后调用 `authenticate(token)` 即可 |
| `SecurityContextHolder` | 保存当前请求的登录状态 | 认证成功后调用 `setAuthentication(token)` |
| `UsernamePasswordAuthenticationToken` | 封装认证信息的载体 | 把用户信息 + 密码 + 权限打包，传给 `AuthenticationManager` |

---

### 知识点 5：CSRF 是什么，为什么开发时关掉

**CSRF（跨站请求伪造）：** 攻击者利用你的登录状态，偷偷发恶意请求到后端。Spring Security 默认开启防护，要求每次请求都携带特殊的 token。

**开发阶段关掉的原因：**
- 用 curl / Postman 测试时，不方便携带 CSRF token
- 学习阶段先专注业务逻辑，上线前再开启 CSRF 保护

```java
http.csrf().disable();  // 关掉 CSRF 保护
```

### 知识点 6：Jackson 序列化规则

Jackson（Spring Boot 内置的 JSON 库）把 Java 对象转成 JSON 的规则：

1. **扫描所有 public 的 getter 方法**（`getXxx()` 或 `isXxx()`）
2. 去掉 `get` 前缀，首字母小写 → JSON 字段名
3. 调用该方法，把返回值写入 JSON

```java
public String getStatus()   → "status": "ok"
public boolean getIsLogin() → "isLogin": false
public Object getData()     → "data": null
public String getMsg()      → "msg": "登录成功"
```

**结论：** 字段可以是 `private`，只要有 public 的 getter，Jackson 就能读到。

---

### 知识点 7：静态内部类 vs 非静态内部类

| 类型 | 能访问外部类实例成员 | 能直接 `new` | 推荐场景 |
|------|---------------------|-------------|---------|
| `static class`（静态） | **不能** | 能，`new Result()` | 内部类不需要访问外部类 |
| `class`（非静态） | **能** | 不能，需先有外部类实例 | 内部类需要访问外部类的字段/方法 |

`Result` 加 `static` 的原因：它只是数据容器，不需要访问 `AuthController` 的任何字段，加 `static` 更简洁，可以直接 `new Result()`。

---

### 知识点 8：Java Bean 的 getter/setter 命名规范

Jackson 转 JSON 是按这个规范识别 getter 的：

| 方法名 | Jackson 识别为的字段名 |
|--------|------------------------|
| `getStatus()` | `status` |
| `getIsLogin()` | `isLogin` |
| `isActive()` | `active`（boolean 用 `is` 前缀） |
| `getData()` | `data` |

> **注意：** `boolean` 类型的字段，getter 通常写 `isXxx()` 而不是 `getXxx()`。Jackson 两种都认识。

## 第三部分：学习 Q&A

### Q1：为什么 `new Result()` 就能返回 JSON？没看到任何转换代码？

**A：** 这是 Spring Boot 内置的 Jackson 自动处理的，调用发生在框架内部。

流程：

```
你的代码                Spring Boot 内部               浏览器收到
return new Result()  →  Jackson 调用所有 getter  →  {"status":"ok","isLogin":false}
```

验证方式——在 getter 里加日志：

```java
public String getStatus() {
    System.out.println("Jackson 正在调用 getStatus()!");
    return "ok";
}
```

访问 `/auth`，控制台会打印这行，证明 Jackson 确实在内部调用了它。

---

### Q2：为什么没看到调用 `getStatus()` 和 `getIsLogin()`？

**A：** Jackson 在序列化时自动调用了它们，这个过程在框架内部发生，你的代码里不会显式出现。

Jackson 的逻辑伪代码：

```java
// Jackson 内部大概做了这些（你不用写）：
Result result = new Result();
String json = "{";
json += "\"status\":\"" + result.getStatus() + "\",";   // 自动调用
json += "\"isLogin\":" + result.getIsLogin();           // 自动调用
json += "}";
// 然后把 json 写进 HTTP 响应体
```

---

### Q3：为什么 `Result` 要声明为 `static`？不加行不行？

**A：** 不加也能运行，但加 `static` 更推荐。

不加 `static` 时，`Result` 是非静态内部类，内部隐含持有 `AuthController` 实例的引用，有额外内存开销，且语义上暗示"这个类依赖外部类"，容易误导。

`Result` 只是数据容器，完全独立于 `AuthController` 的实例，所以加 `static` 更清晰，且可以直接 `new Result()` 而不需要任何外部类实例。

---

### Q4：内部类需要访问外部类的实例字段时，能用 `static` 吗？

**A：** 不能，会编译报错。

```java
public class AuthController {
    private String controllerName = "Auth控制器";  // 实例字段

    // 静态内部类：无法访问外部类实例字段
    private static class StaticResult {
        public String test() {
            return controllerName;  // ✗ 编译错误！Cannot access instance field
        }
    }

    // 非静态内部类：可以访问
    private class NonStaticResult {
        public String test() {
            return controllerName;  // ✓ 可以
        }
    }
}
```

**原因：** `static` 成员属于类本身，不依附于任何实例，所以访问不了实例字段。

**决策：**
- 内部类需要访问外部类实例 → **必须用非静态（不加 static）**
- 内部类不需要访问外部类 → **推荐用静态（加 static）**

### Q5：`@Controller` 的方法不加 `@ResponseBody` 会怎样？

**A：** Spring 会把返回值当作**视图名**去找 HTML 模板文件，找不到就报错。

```java
@Controller
public class AuthController {

    @GetMapping("/auth")
    // 不加 @ResponseBody：Spring 去找名为"Result 对象的 toString"的模板，报 500
    public Object auth() {
        return new Result();
    }

    @GetMapping("/auth")
    @ResponseBody  // 加上：把返回值转成 JSON
    public Object auth() {
        return new Result();
    }
}
```

**更简洁方式：** 把 `@Controller` 换成 `@RestController`，全类的方法都默认 `@ResponseBody`，不用逐个加。

### Q6：`WebSecurityConfig` 读完感觉"似懂非懂"，是缺前置知识吗？

**A：** 不是你的问题。Spring Security 是 Spring 生态里**公认最难上手的模块之一**。

#### 为什么会"似懂非懂"

Spring Security 一次性塞给你 **5 个新概念叠在一起**：

```
① HTTP 请求权限控制（哪些路径要登录才能访问）
② 用户体系（UserDetails、UserDetailsService 是什么）
③ 认证流程（AuthenticationManager 怎么验密码）
④ CSRF 是什么
⑤ 上面这些怎么和 Spring 的 Bean / 注入机制联动
```

任何一个单独拿出来都需要时间消化，现在全堆在一个文件里，"看懂每一行"和"理解整体在干什么"是两回事。

#### 你真正缺的不是知识，而是一张"地图"

```
前端点击登录按钮
    │
    │  POST /auth/login
    │  body: { username: "user", password: "password" }
    ▼
AuthController.login()
    │
    ├─ ① 先查用户存不存在
    │     userDetailsService.loadUserByUsername("user")
    │     → 去 InMemoryUserDetailsManager 里找 → 找到了
    │
    ├─ ② 再验密码对不对
    │     authenticationManager.authenticate(token)
    │     → 内部调 passwordEncoder.matches()
    │     → 密码匹配 → 认证通过
    │
    └─ ③ 把认证结果存起来
          SecurityContextHolder.getContext().setAuthentication(...)
          → 后续请求就知道"当前用户是谁"了
    ▼
返回 { status: "ok", msg: "登录成功", isLogin: true, data: {...} }
```

`WebSecurityConfig` 只是在**提前配置**这个流程里用到的两个零件：

```
零件 ①：用户从哪来？
→ userDetailsService() 方法
→ 从内存里造一个用户（用户名 user，密码 password）

零件 ②：谁来验密码？
→ customAuthenticationManager() 方法
→ 把 Spring Security 内置的认证器暴露出来，AuthController 拿去用

配置 ③：哪些路径需要登录？
→ configure(HttpSecurity) 方法
→ /auth/** 不用登录，其他要登录
```

#### 你现阶段只需要记住的 3 句话

| 问题 | 一句话答案 |
|------|-----------|
| `UserDetailsService` 是什么 | 一个"查用户"的接口，Spring Security 登录时调它找用户 |
| `AuthenticationManager` 是什么 | 一个"验密码"的工具，把用户名+密码传给它，它告诉你对不对 |
| `SecurityContextHolder` 是什么 | 一个"当前登录是谁"的存储器，验证通过后把结果存进去 |

### Q7：`WebSecurityConfig` 里的方法是我手动调的吗？什么时候被触发？

**A：** 全部是 Spring Security 在启动时**自动调用**的，你不需要手动调。

#### 触发时机一览

```
Spring Boot 启动
    │
    ▼
扫描到 @EnableWebSecurity + @Configuration
    │
    ├─ 调用 configure(HttpSecurity)
    │    → 读取你的路径权限规则，注册进 Security 过滤器链
    │
    ├─ 调用 userDetailsService()（因为加了 @Bean）
    │    → 创建 InMemoryUserDetailsManager 并注册为 Bean
    │    → Spring Security 后续认证时自动使用它查用户
    │
    └─ 调用 customAuthenticationManager()（因为加了 @Bean）
         → 把 authenticationManager() 返回的实例注册为 Bean
         → AuthController 注入时从这里拿

Spring Boot 启动完成，对外提供服务
    │
    │  收到 POST /auth/login 请求
    ▼
AuthController.login()
    │
    │  authenticationManager.authenticate(...)
    │    → 内部调 userDetailsService.loadUserByUsername()
    │    → 找到用户，比对密码
    ▼
返回登录结果
```

#### 为什么你看不到"调用"的地方

Spring Security 在底层维护了一套**过滤器链（Filter Chain）**，每个请求进来都会经过这条链。`configure(HttpSecurity)` 配置的规则，是在过滤器里自动执行的，不是在你的 Controller 或 Service 里手动调的。

你能看到调用的地方只有：`AuthController.login()` 里主动调了 `authenticationManager.authenticate()`，其余的都是框架自动处理。

### Q7：`@Configuration` 是干什么的？没有它会怎样？

**A：** `@Configuration` 告诉 Spring：**"这个类不是普通业务类，而是一个配置类，里面 `@Bean` 方法返回的对象要注册进容器。"**

#### 有 `@Configuration` vs 没有 `@Configuration`

| | 有 `@Configuration` | 没有 `@Configuration`（只有 `@Bean`） |
|--|--------------------|------------------------------------|
| `@Bean` 方法能注册 Bean 吗 | 能 | 能（Spring 也会扫描） |
| 同一个 `@Bean` 方法被调用多次时 | 返回**同一个单例**（Spring 做了代理拦截） | 每次都 `new` 一个新对象（不是单例！） |

#### 具体例子

```java
@Configuration
public class Config {

    @Bean
    public A beanA() {
        return new A();
    }

    @Bean
    public B beanB() {
        return new B(beanA());  // ← 这里调了 beanA()
    }

    @Bean
    public C beanC() {
        return new C(beanA());  // ← 这里又调了 beanA()
    }
}
```

**有 `@Configuration`：** Spring 代理了这个类，`beanA()` 第二次调用时直接从容器里取，`beanB` 和 `beanC` 里拿到的是**同一个 A 实例**。

**没有 `@Configuration`（改成 `@Component`）：** `beanA()` 每次真正执行，创建新对象，`beanB` 和 `beanC` 拿到的是**两个不同的 A 实例**，破坏了单例。

#### 在 `WebSecurityConfig` 里的意义

`WebSecurityConfig` 继承了 `WebSecurityConfigurerAdapter`，内部的 `userDetailsService()` 和 `customAuthenticationManager()` 都是 `@Bean` 方法，且相互之间可能有引用。加 `@Configuration` 确保 Spring 代理拦截，保证单例语义正确。

**一句话记忆：** `@Configuration` = 让 `@Bean` 方法之间的调用走容器，而不是真的执行方法，保证单例。

### Q9：`@Controller` 到底有什么用？没有它会怎样？

**A：**

**有它：** Spring 启动时扫描到这个注解，把 `AuthController` 注册成 Bean，并且告诉 Spring MVC："这个类里有处理 HTTP 请求的方法，帮我接管进来。"

**没有它：** Spring 不认识这个类，`@GetMapping`、`@PostMapping` 全部失效，浏览器访问 `/auth` 和 `/auth/login` 会直接报 404。

**一句话记忆：** `@Controller` = 告诉 Spring"这个类负责接 HTTP 请求"，没有它请求就找不到这里。

### Q10：`userDetailsService.loadUserByUsername(username)` 干嘛的？

**A：** 它在做一件事：**去找这个用户名存不存在。**

`userDetailsService` 是在 `WebSecurityConfig` 里配置的，内部存了一个用户（用户名 `user`，密码 `password`，角色 `USER`）。

调 `loadUserByUsername("user")` 时，去内存里找有没有叫 `user` 的账户：

```
找到了 → 返回 UserDetails 对象（包含用户名、密码、角色）
找不到 → 抛出 UsernameNotFoundException
          → 被 catch 捕获 → 返回"用户不存在"，方法结束
```

**为什么用 try-catch 而不是 if-else：** 因为 `loadUserByUsername` 找不到时会**抛异常**，不是返回 null，这是 Spring Security 的设计，只能用 try-catch 处理。

**一句话记忆：** `loadUserByUsername` = "这个用户名有没有注册过"，有就返回用户信息，没有就抛异常。

### Q11：第 51 行的 `UsernamePasswordAuthenticationToken` 是什么意思？

```java
UsernamePasswordAuthenticationToken token = new UsernamePasswordAuthenticationToken(userDetails, password, userDetails.getAuthorities());
```

**① `UsernamePasswordAuthenticationToken`：** Spring Security 提供的一个"凭证包"，专门用来装"用户信息 + 密码 + 角色"，交给认证器去验证。

**② 3 个参数分别是：**

| 参数位置 | 传入的值 | 含义 |
|---------|---------|------|
| 第 1 个 | `userDetails` | 用户信息（上一步查到的） |
| 第 2 个 | `password` | 前端传来的密码字符串 |
| 第 3 个 | `userDetails.getAuthorities()` | 这个用户拥有的角色列表（如 `ROLE_USER`） |

**③ 注意：** 3 参数的构造器会直接把 token 标记为"已认证"，但此时密码还没验过。更正确的做法是先用 2 参数构造器（未认证），交给 `authenticationManager` 验完再标记已认证（见之前对代码的修复记录）。

**一句话记忆：** 第 51 行 = 把"用户信息 + 密码 + 角色"打包成一个凭证，准备交给下一步验证。

### Q12：第 54-55 行分别是什么意思？

```java
authenticationManager.authenticate(token);          // 第 54 行
SecurityContextHolder.getContext().setAuthentication(token);  // 第 55 行
```

**第 54 行 — 验密码：**

把凭证包交给 `authenticationManager`，它内部做：

```
取出 token 里的密码（password）
→ 找到存储的密码（WebSecurityConfig 里配置的 "password"）
→ 用 passwordEncoder 比对两个密码
→ 匹配：正常返回，继续往下走
→ 不匹配：抛出 BadCredentialsException → 返回"密码不正确"
```

**第 55 行 — 记录登录状态：**

走到第 55 行 = 密码已经验过了，是对的。这行把"当前用户已登录"这个状态存起来。

`SecurityContextHolder` 相当于服务器的**临时记事本**，记着"这次请求是谁在操作"：

```
setAuthentication(token)
→ 把 token（包含用户信息）写进记事本
→ 后续这次请求里，任何地方都可以从 SecurityContextHolder 读出"当前登录的是谁"
```

**一句话分别记忆：**
- 第 54 行：`authenticate` = **验密码**，错了抛异常，对了继续
- 第 55 行：`setAuthentication` = **存登录状态**，告诉系统"这个用户已登录"

### Q13：`InMemoryUserDetailsManager`（内存里的用户）是什么时候被创建的？

**A：** 在 **Spring Boot 启动时**就创建好了，不是收到登录请求时才创建。

```
你执行 mvn spring-boot:run
        │
        ▼
SpringApplication.run(Application.class, args)
        │
        ▼
Spring 扫描所有带注解的类
        │
        ├─ 扫到 WebSecurityConfig（有 @Configuration）
        │    └─ 看到 userDetailsService() 上有 @Bean
        │              │
        │              ▼
        │         调用这个方法，创建用户 + InMemoryUserDetailsManager
        │         注册进 Spring 容器 ✓
        │
        ▼
Spring Boot 启动完成，用户信息已经在内存里等着
        │
        │    ← 一直在内存里等着
        │
收到 POST /auth/login 请求时
        │
        ▼
loadUserByUsername("user") 从内存里查 → 找到了
```

**你之前不理解的根本原因：** 缺少"Spring Boot 启动时会扫描 `@Bean` 并提前创建对象"这个前置知识。你以为 `userDetailsService()` 是收到请求时才执行的，其实它在启动阶段就已经执行完了，对象一直活在容器里。

**一句话记忆：** `@Bean` 方法在**启动时**执行，不是在请求来时执行。请求来时只是从容器里**取出**已经创建好的对象去用。

### Q14：`User` 里的 `Instant createdAt` / `updatedAt` 是什么意思？

**缺的前置知识：** Java 里表示时间有多种类型，`Instant` 是 `java.time` 包里的其中一种。

#### 类比锚定

`Instant` 就像**时间轴上的一个精确刻度**。地球上的每一秒，不管你在北京还是纽约，时间轴上那个刻度是同一个点。`Instant` 存的就是这个「绝对时间点」，不带「北京几点」这种本地时区概念。

#### 常见时间类型对比

| 类型 | 是什么 | 适合干嘛 |
|------|--------|---------|
| `Date` | 老式时间类，有时区混乱问题 | 老项目，不推荐新写 |
| `LocalDateTime` | 本地时间，**不带时区**（如 `2026-04-02 14:00:00`） | 只在一个时区内用 |
| **`Instant`** | 时间轴上的绝对时间点，本质是**距离 1970-01-01 00:00:00 UTC 的纳秒偏移** | 数据库存储、跨时区、记录「什么时候发生的」 |

#### 在你项目里的用法

`User.java` 中：

```java
Instant createdAt;   // 账号创建时间
Instant updatedAt;   // 账号最后修改时间
```

构造器里：

```java
this.createdAt = Instant.now();
this.updatedAt = Instant.now();
```

`Instant.now()` = 拍下「此刻」在时间轴上的刻度，存进字段。

返回给前端的 JSON 可能形如（Jackson 序列化）：

```json
{
  "createdAt": "2026-04-02T06:20:00Z",
  "updatedAt": "2026-04-02T06:20:00Z"
}
```

结尾的 `Z` 表示 UTC（零时区），体现的是「绝对时间点」。

**一句话记忆：** `Instant` = 时间轴上的一个刻度，全球统一表示，适合记「事件发生在什么时候」。
