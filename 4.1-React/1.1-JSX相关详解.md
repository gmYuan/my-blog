## Q1: JSX 完整数据流程是什么？（⭐⭐⭐⭐⭐）

**考察点：** 对 React 编译和运行时的全局理解

### 回答要点

JSX 从编写到渲染经历三个核心阶段：**编译时、运行时、渲染时**。

### 完整流程图

```
1. 开发者写 JSX（你写的代码）
   ↓
   <div id="123">Hello</div>

2. Babel 编译（编译时转换）
   ↓
   jsx("div", { id: "123" }, "Hello")

3. jsx 方法处理（运行时转换）
   ↓
   ├── 提取 key = null
   ├── 提取 ref = null
   ├── 提取 props = { id: "123", children: "Hello" }
   └── 调用 ReactElement(type, key, ref, props)

4. 返回 ReactElement 对象（JavaScript 对象）
   ↓
   {
     $$typeof: Symbol.for('react.element'),
     type: "div",
     key: null,
     ref: null,
     props: { id: "123", children: "Hello" },
     __mark: "Ygm"
   }

5. 渲染阶段（React 内部处理）
   ↓
   React 遍历 ReactElement 树
   ↓
   生成 Fiber 树（虚拟 DOM）
   ↓
   更新真实 DOM
```

### 三个关键阶段

| 阶段 | 负责方 | 输入 | 输出 | 何时发生 |
|------|--------|------|------|---------|
| **编译时** | Babel | JSX 语法 | jsx() 函数调用 | 打包/构建时 |
| **运行时** | jsx 方法 | jsx() 调用 | ReactElement 对象 | 代码执行时 |
| **渲染时** | react-reconciler | ReactElement 树 | Fiber 树 → DOM | React 渲染时 |

### 面试追问

**Q: Babel 为什么要把 JSX 转成函数调用？**  
A: 因为 JSX 不是标准 JavaScript 语法，浏览器无法直接执行。Babel 把 JSX 转成标准的函数调用（jsx()），这样浏览器就能运行了。

**Q: jsx() 方法的核心作用是什么？**  
A: 创建 ReactElement 对象。它会：
1. 提取 key 和 ref（特殊属性）
2. 将其他属性整理到 props 中
3. 处理 children（区分单个/多个）
4. 返回一个标准的 ReactElement 对象

**Q: ReactElement 和真实 DOM 的关系？**  
A: ReactElement 是轻量级的 JavaScript 对象，描述"想要什么 DOM"。React 会对比新旧 ReactElement 树，计算出最小变更，然后更新真实 DOM（diff 算法）。

## Q2: $$typeof 和 Symbol 的完整防御机制？（⭐⭐⭐⭐⭐）

**考察点：** React 安全机制、Symbol 特性、XSS 防御

### 核心问题

**为什么 React 元素需要 `$$typeof` 字段？**

防止服务端注入假的 React 元素，这是 React 的 **XSS 防御机制**。

### 攻击场景

假设没有 `$$typeof`，攻击者可以这样做：

```javascript
// 服务端返回恶意数据
const serverData = {
  type: 'div',
  props: {
    dangerouslySetInnerHTML: {
      __html: '<script>alert("你的账号被盗了")</script>'
    }
  }
};

// React 直接渲染 → 💥 XSS 攻击成功
```

### 防御机制：Symbol 无法通过 JSON 传输

```javascript
// 攻击者尝试伪造 React 元素
const fakeElement = {
  $$typeof: Symbol.for('react.element'),  // ← 想要伪造这个
  type: 'div',
  props: { dangerouslySetInnerHTML: { __html: '<script>...</script>' } }
};

// 通过 JSON 传输
const json = JSON.stringify(fakeElement);
console.log(json);
// 输出：{"type":"div","props":{...}}
// ❌ $$typeof 消失了！JSON 无法序列化 Symbol

// 前端接收并解析
const parsed = JSON.parse(json);
console.log(parsed.$$typeof);  // undefined

// React 检查
if (parsed.$$typeof !== Symbol.for('react.element')) {
  throw new Error('这不是真的 React 元素！拒绝渲染！');
}
```

### 为什么 JSON 不支持 Symbol？

**技术原因：**
- JSON 诞生于 2001 年，Symbol 是 ES6（2015）才加入的
- JSON 是纯文本格式，Symbol 是 JavaScript 运行时概念
- JSON 需要跨语言通用（Java、Python、Go），这些语言没有 Symbol

**JSON 支持的类型（仅6种）：**

```javascript
// ✅ 支持
{ string: "hello", number: 123, boolean: true, null: null, array: [], object: {} }

// ❌ 不支持（会被忽略）
{ symbol: Symbol('test'), function: () => {}, undefined: undefined }
```

### Symbol.for 和 Symbol 的区别

```javascript
// Symbol() - 每次都不同（不能跨模块共享）
const s1 = Symbol('test');
const s2 = Symbol('test');
console.log(s1 === s2);  // false

// Symbol.for() - 全局唯一（可以跨模块共享）
const s3 = Symbol.for('test');
const s4 = Symbol.for('test');
console.log(s3 === s4);  // true
```

**React 为什么用 Symbol.for？**
- 确保不同包（react、react-dom）使用同一个 Symbol
- 即使打包成不同 bundle，也能正确识别 ReactElement

### 面试追问

**Q: 攻击者能否传一个数字来伪造？**  
A: 不能。React 内部会严格检查 `$$typeof === Symbol.for('react.element')`，数字、字符串、null 都不匹配。

**Q: 为什么不用字符串标记？**  
A: 字符串可以通过 JSON 传输，攻击者可以轻易伪造 `{"$$typeof": "react.element"}`。

**Q: 这种防御是否万无一失？**  
A: 针对服务端注入的 JSON 数据是安全的。但如果攻击者能直接在前端执行 JavaScript 代码（例如通过其他漏洞），就能绕过这个防御。这只是 React 的**纵深防御**之一。

## Q3: children 为什么要区分单个和多个？（⭐⭐⭐⭐⭐）

**考察点：** 性能优化意识、React 设计决策

### 两种方案对比

```javascript
// 方案 1：全部用数组（简单但浪费）
<div>Hello</div>
// props.children = ["Hello"]  ← 单个元素也创建数组

// 方案 2：区分单个和多个（React 的选择）
<div>Hello</div>
// props.children = "Hello"  ← 直接是字符串（节省内存）

<div>
  <span>A</span>
  <span>B</span>
</div>
// props.children = [<span>A</span>, <span>B</span>]  ← 多个才用数组
```

### 性能分析

**典型场景：列表渲染**

```jsx
function List({ data }) {
  return (
    <ul>
      {data.map(item => (
        <li key={item.id}>{item.name}</li>  // ← 每个 li 只有1个 text 子元素
      ))}
    </ul>
  );
}

// 假设有 1000 个 item
<List data={thousandItems} />
```

| 方案 | 创建的数组数量 | 内存开销 |
|------|--------------|---------|
| 全部用数组 | 1000 个 | 高（每个数组对象 + 数组元数据） |
| 区分单个/多个 | 0 个 | 低（直接用字符串） |

**内存节省计算：**
- 一个数组对象：~40 字节（V8 引擎）
- 1000 个数组：~40KB
- 如果页面有 10 个这样的列表：~400KB

在复杂应用中，这种优化积少成多。

### jsx 方法中的实现

```typescript
export const jsx = (type: ElementType, config: any, ...maybeChildren: any) => {
  // ... 其他处理
  
  const maybeChildrenLength = maybeChildren.length;
  if (maybeChildrenLength) {
    if (maybeChildrenLength === 1) {
      // ✅ 单个：直接赋值（快速路径）
      props.children = maybeChildren[0];
    } else {
      // ✅ 多个：用数组
      props.children = maybeChildren;
    }
  }
  // 如果没有 children，props.children 就是 undefined
};
```

### 使用时的处理

**如何统一处理 children？**

```javascript
function MyComponent({ children }) {
  // 方法 1：React.Children API
  React.Children.forEach(children, (child) => {
    console.log(child);  // 自动处理单个/多个
  });
  
  // 方法 2：转成数组
  const childArray = React.Children.toArray(children);
  childArray.forEach((child, index) => {
    console.log(`Child ${index}:`, child);
  });
  
  // 方法 3：手动判断
  const childArray = Array.isArray(children) ? children : [children];
}
```

### 面试追问

**Q: 为什么不用 null 表示没有 children？**  
A: `undefined` 更语义化，表示"这个属性不存在"。用 `null` 反而会让 `props.children` 总是存在，不符合"没有就不传"的原则。

**Q: 这种优化在现代浏览器中还重要吗？**  
A: 是的。虽然内存变便宜了，但：
1. 移动设备内存仍然有限
2. 大型应用中优化积少成多
3. 减少 GC（垃圾回收）压力，提升流畅度

**Q: 这会不会增加使用复杂度？**  
A: 会一点点，但 React 提供了 `React.Children` API 来统一处理。这是典型的"把复杂度留给框架，把便利性留给用户"的设计。

## Q4: 为什么要用 `{}.hasOwnProperty.call(config, prop)`？（⭐⭐⭐⭐）

**考察点：** 安全编程、原型链理解、边界case处理

### 三个问题场景

#### 问题 1：对象可能没有 hasOwnProperty 方法

```javascript
// 正常对象
const normalConfig = { id: '123' };
normalConfig.hasOwnProperty('id');  // ✅ true

// 没有原型的对象
const nullProtoConfig = Object.create(null);
nullProtoConfig.id = '123';
nullProtoConfig.hasOwnProperty('id');  // ❌ 报错：hasOwnProperty is not a function
```

**为什么会这样？**
- `Object.create(null)` 创建的对象没有原型链
- `hasOwnProperty` 方法定义在 `Object.prototype` 上
- 没有原型 = 没有 `hasOwnProperty` 方法

#### 问题 2：hasOwnProperty 可能被覆盖

```javascript
// 恶意覆盖
const hackedConfig = {
  id: '123',
  hasOwnProperty: function() {
    return false;  // 假装所有属性都不存在
  }
};

hackedConfig.hasOwnProperty('id');  // ❌ false（被骗了！）
```

#### 问题 3：继承的属性也会被 for...in 遍历

```javascript
const parent = { inherited: 'from parent' };
const config = Object.create(parent);
config.own = 'my property';

// for...in 会遍历继承的属性
for (const prop in config) {
  console.log(prop);  // 输出：own, inherited
}
```

### 安全的解决方案

```javascript
// ✅ 最安全的写法
{}.hasOwnProperty.call(config, prop)

// 等价于
Object.prototype.hasOwnProperty.call(config, prop)
```

**为什么这样写安全？**
1. `{}` 创建一个新的空对象，它有原始的 `hasOwnProperty` 方法
2. 用 `.call(config, prop)` 把方法应用到 `config` 上
3. 不管 `config` 是什么样的对象（没有原型、被篡改），都能正确检查

### 三种写法对比

```javascript
const config = Object.create({ inherited: 'parent' });
config.own = 'child';

// ❌ 写法 1：直接用 in（会包含继承的属性）
for (const prop in config) {
  console.log(prop);  // own, inherited（不想要 inherited）
}

// ❌ 写法 2：config.hasOwnProperty（可能崩溃或被篡改）
if (config.hasOwnProperty('own')) {  // 可能报错
  console.log('has own');
}

// ✅ 写法 3：{}.hasOwnProperty.call（始终安全）
if ({}.hasOwnProperty.call(config, 'own')) {  // 始终正常工作
  console.log('has own');
}
```

### jsx 方法中的实际使用

```typescript
export const jsx = (type: ElementType, config: any, ...maybeChildren: any) => {
  let key: Key = null;
  const props: Props = {};
  let ref: Ref = null;

  for (const prop in config) {
    const val = config[prop];
    
    // 跳过 key 和 ref
    if (prop === 'key' || prop === 'ref') {
      continue;
    }
    
    // ✅ 安全检查：只要自己的属性，不要继承的
    if ({}.hasOwnProperty.call(config, prop)) {
      props[prop] = val;
    }
  }
};
```

### 面试追问

**Q: 为什么不直接用 Object.hasOwn()（ES2022新方法）？**  
A: `Object.hasOwn(obj, prop)` 确实更简洁，但：
1. 需要 ES2022+ 支持（旧浏览器不支持）
2. React 需要兼容更多环境
3. `{}.hasOwnProperty.call()` 是经典写法，兼容性最好

**Q: 这种边界情况真的会遇到吗？**  
A: 会的，例如：
1. JSON 解析某些特殊数据可能产生 null 原型对象
2. 用户传入的 props 可能来自第三方库，无法保证结构
3. 恶意用户可能尝试注入特殊对象

React 作为底层框架，必须处理所有边界情况。

**Q: 这是否是过度设计？**  
A: 不是。这是**防御性编程**的最佳实践：
- 不假设输入一定合法
- 不依赖对象的默认行为
- 确保在任何情况下都能正常工作
