# React的VDOM DIFF算法原理是什么？

## 什么是DIFF算法？

**DIFF算法**是React协调（Reconciliation）过程的核心，用于对比新旧虚拟DOM树，找出最小变更集合，然后批量更新到真实DOM。

**注意：** 是**VDOM DIFF**，不是DOM DIFF！

---

## 为什么需要DIFF算法？

### 传统方案的问题

直接对比两棵树的差异，传统算法的时间复杂度是**O(n³)**：
- 遍历树A的每个节点：O(n)
- 遍历树B的每个节点：O(n)
- 计算编辑距离：O(n)
- 总复杂度：O(n³)

对于有1000个节点的树，需要10亿次比较，性能无法接受。

### React的优化策略

React通过**三个策略假设**，将复杂度降低到**O(n)**：

## React DIFF的三大策略

### 策略1：只对比同一层级

**假设：** 跨层级的DOM节点移动操作非常少，可以忽略不计。

**实现：**
- 只对比同层级的节点
- 不跨层级比较
- 如果节点不在了，直接删除该节点及其子树

```
旧树：              新树：
  A                   A
 / \                 / \
B   C               D   E
   / \                 / \
  D   E               F   G

对比过程：
第1层：A vs A ✅ 相同
第2层：B,C vs D,E ❌ 不同 → 删除B,C子树，创建D,E子树
第3层：不再对比C的子节点 → 即使D,E在新树中也存在，也会被重新创建
```

**优势：** 避免了跨层级搜索，复杂度从O(n³)降到O(n)

**注意：** 如果真的发生跨层级移动，React不会复用，而是删除后重建，性能会下降。

---

### 策略2：不同类型元素直接替换

**假设：** 不同类型的元素会产生不同的树结构。

**实现：**
- 如果节点类型不同（如`<div>`变成`<span>`），直接删除旧节点，创建新节点
- 不再对比子节点

```jsx
// 旧树
<div>
  <Counter />
</div>

// 新树
<span>
  <Counter />
</span>

// React的处理：
// 1. 删除整个 <div> 及其子树（包括 Counter 组件）
// 2. 创建新的 <span> 及其子树（重新挂载 Counter 组件）
// 结果：Counter 组件会被卸载后重新挂载，状态丢失
```

**优势：** 避免了深度对比不同类型节点的子树

**注意：** 会导致组件卸载重新挂载，状态丢失。

---

### 策略3：通过key标识列表元素

**假设：** 开发者可以通过`key`属性来暗示哪些子元素是稳定的。

**实现：**
- 使用`key`来快速识别列表中元素的移动、新增、删除
- 没有`key`时，按顺序逐个对比

#### 没有key的情况

```jsx
// 旧列表
<ul>
  <li>A</li>
  <li>B</li>
</ul>

// 新列表（头部插入C）
<ul>
  <li>C</li>
  <li>A</li>
  <li>B</li>
</ul>

// React的处理（没有key）：
// 对比第1个：A vs C → 不同 → 更新内容 A→C
// 对比第2个：B vs A → 不同 → 更新内容 B→A
// 对比第3个：无 vs B → 新增 → 插入B
// 结果：3次DOM操作（2次更新 + 1次插入）
```

#### 有key的情况

```jsx
// 旧列表
<ul>
  <li key="a">A</li>
  <li key="b">B</li>
</ul>

// 新列表（头部插入C）
<ul>
  <li key="c">C</li>
  <li key="a">A</li>
  <li key="b">B</li>
</ul>

// React的处理（有key）：
// key="a"：位置变了，移动
// key="b"：位置变了，移动
// key="c"：新增，插入
// 结果：1次DOM操作（插入C），A和B复用
```

**优势：**
- 准确识别节点的移动、新增、删除
- 最大化复用现有节点
- 减少不必要的DOM操作

**为什么key不能用index？**

```jsx
// 初始列表
[
  { id: 'a', name: 'Alice' },  // index=0
  { id: 'b', name: 'Bob' },    // index=1
  { id: 'c', name: 'Charlie' } // index=2
]

// 删除第一项后
[
  { id: 'b', name: 'Bob' },    // index=0 ⚠️ key变了！
  { id: 'c', name: 'Charlie' } // index=1 ⚠️ key变了！
]

// React认为：
// - index=0 从 Alice 变成 Bob → 更新内容
// - index=1 从 Bob 变成 Charlie → 更新内容
// - index=2 的 Charlie 被删除
// 结果：误判为内容变化，而非删除第一项
```

**正确做法：** 使用唯一稳定的业务ID作为key。

## DIFF算法的详细流程

### 1️⃣ 树的对比（Tree Diff）

```
从根节点开始，逐层对比：

层级1：对比根节点
  ↓
层级2：对比根节点的子节点
  ↓
层级3：对比子节点的子节点
  ↓
...
```

### 2️⃣ 组件对比（Component Diff）

**同类型组件：**
- 继续对比子节点（递归）
- 可以通过`shouldComponentUpdate`或`React.memo`优化

**不同类型组件：**
- 直接删除旧组件树
- 创建新组件树

### 3️⃣ 元素对比（Element Diff）

**同类型元素：**
```jsx
// 旧
<div className="old" title="old">

// 新
<div className="new" title="new">

// React的处理：
// - 保留DOM节点
// - 只更新变化的属性（className, title）
// - 继续递归对比子节点
```

**不同类型元素：**
```jsx
// 旧
<div>

// 新
<span>

// React的处理：
// - 删除旧的 <div> 及其子树
// - 创建新的 <span> 及其子树
```

### 4️⃣ 列表对比（List Diff）

使用**key**进行高效对比：

**算法步骤：**
1. 新旧列表都建立key→节点的映射
2. 遍历新列表：
   - 如果key在旧列表中存在 → 复用（可能需要移动）
   - 如果key在旧列表中不存在 → 新增
3. 遍历旧列表：
   - 如果key在新列表中不存在 → 删除

## React Fiber架构下的DIFF

React 16引入了Fiber架构，DIFF算法也相应优化：

### 传统Reconciliation的问题

```
旧版本（React 15及之前）：
- DIFF过程是递归的
- 一旦开始就无法中断
- 大型组件树DIFF时间过长 → 阻塞主线程 → 页面卡顿
```

### Fiber的优化

```
Fiber架构（React 16+）：
- DIFF过程是可中断的
- 使用链表结构（而非递归）
- 分片执行，每5ms让出主线程
- 高优先级任务可以打断低优先级DIFF
```

**工作流程：**
1. **Render阶段**（可中断）：
   - 遍历Fiber树
   - 对比新旧虚拟DOM
   - 标记变更（effectTag）
   
2. **Commit阶段**（不可中断）：
   - 一次性应用所有变更到真实DOM

## DIFF算法的性能优化建议

### 1️⃣ 合理使用key

```jsx
// ❌ 不要用index
{items.map((item, index) => <Item key={index} data={item} />)}

// ✅ 使用唯一ID
{items.map(item => <Item key={item.id} data={item} />)}
```

### 2️⃣ 避免不必要的类型变化

```jsx
// ❌ 类型变化导致重建
{isLoading ? <div>Loading...</div> : <span>Content</span>}

// ✅ 保持类型一致
{isLoading ? <div>Loading...</div> : <div>Content</div>}
```

### 3️⃣ 使用shouldComponentUpdate/React.memo

```jsx
// 类组件
class MyComponent extends React.Component {
  shouldComponentUpdate(nextProps) {
    return nextProps.value !== this.props.value;
  }
}

// 函数组件
const MyComponent = React.memo(({ value }) => {
  return <div>{value}</div>;
});
```

### 4️⃣ 避免在render中创建新对象

```jsx
// ❌ 每次render都创建新对象 → DIFF认为props变了
<Child style={{ color: 'red' }} />

// ✅ 提取到常量
const style = { color: 'red' };
<Child style={style} />
```

## 总结

| 策略 | 说明 | 复杂度优化 |
|-----|------|-----------|
| **同层级对比** | 只对比同一层级节点 | O(n³) → O(n) |
| **类型判断** | 不同类型直接替换 | 避免深度对比 |
| **key标识** | 通过key复用节点 | 高效列表更新 |

**React DIFF的核心思想：**
- 用**启发式算法**替代完美算法
- 牺牲理论最优，换取实际高效
- 通过**合理假设**将O(n³)降到O(n)

**协调过程（Reconciliation）：**
```
状态变化 → 生成新虚拟DOM → DIFF算法对比 → 标记变更 → 批量更新真实DOM
```
