# 【高频】为什么以函数组件 + Hooks 为主？

## 当前业界常见说法

新岗位与新技术栈里，**默认写法是函数组件 + Hooks**；面试重点多在 **Hooks 用法、规则与原理**，**很少**再单独深挖 Class 生命周期或 `this` 绑定。若问到 Class，多数是 **对比题**（为何迁移、有何不便），用下面几条概括即可。

## 常怎么问

- 为什么推荐函数组件 + Hooks？  
-（偶尔）和 Class 相比有什么优势？

## 答题要点（预习篇框架）

### 函数组件 + Hooks（主答）

- 无 **`super` / `this` 绑定** 一类样板与心智负担。  
- **`useState` / `useEffect` 等** 表达状态与副作用，逻辑更易聚在一处。  
- **自定义 Hooks** 复用有状态逻辑，比 mixin 更清晰（见 **1.4**）。

### Class 侧（对比时略提即可）

- 构造与继承样板多、事件回调易遇 **`this`** 问题、生命周期 API 多且历史版本有差异——当前业界 **更倾向 Hooks 作为默认**，Class 多用于 **维护旧代码**。

**为什么 Class 会有 super 与 this 的问题？（预习篇原文）**

1. **`super(props)` 必须调用的原因**  
   - Class 组件继承 `React.Component`，构造函数里必须先 **`super(props)`** 执行父类构造。  
   - 若不调或只调 `super()`（不传 `props`），父类构造里的 **`this.props` 赋值** 会失败 → 在 `constructor` 或早期生命周期里 **`this.props === undefined`**。  
   - 这是 **ES6 class 继承** 的语法强制，与 React 无关；但函数组件 **没有继承、没有 this**，自然不需要。

2. **事件回调要 `bind(this)` 或箭头函数的原因**  
   - React 的合成事件在触发时，**直接调用回调函数**（类似 `callback(event)`），不是 `obj.callback(event)`。  
   - Class 实例方法如果直接传给 `onClick={this.handleClick}`，执行时 **`this` 会丢失**（变成 `undefined` 或 window）。  
   - **解决方案：**  
     - `onClick={this.handleClick.bind(this)}` 或在 constructor 里预绑定；  
     - 或用 **类字段箭头函数**（ES2022+）：`handleClick = () => { ... }`，箭头函数 **继承外层 this**，自动指向组件实例。  
   - 函数组件 **没有 this**，状态与函数通过 **闭包** 捕获，直接 `onClick={handleClick}` 即可。

**一句话：** Class 的 `super` 和 `bind` 都是 **ES6 类语法 + React 事件机制** 的副作用；Hooks 用闭包替代，心智成本更低。

### 趋势（可一句带过）

- 官方与生态新能力默认面向 Hooks；不必把 Class 当作当前学习与面试的主线。

## 一句话收尾

「默认用 Hooks 写业务与复用逻辑；Class 对比题知道「样板多、this 与生命周期成本高」即可。」
