# 【高频】自定义 Hooks：做什么？和 HOC、mixin 怎么选？

## 当前业界常见说法

**「怎么复用带状态的逻辑」** 在多数团队与面试语境里，默认会指向 **自定义 Hook**（配合 `useState` / `useEffect` 等）。**mixin 已淘汰**；**HOC**（高阶组件，入参组件返回组件）在类库与旧代码里仍常见，但新需求里更常落到 **抽 `useXxx`**。本篇对应预习篇「自定义 Hooks 模式」，可与 **1.2**（为何以 Hooks 为主）对照阅读。

## 常怎么问

- 自定义 Hook 是什么？命名为什么以 `use` 开头？  
- 和 HOC、render props 比有什么优势？  
- Hooks 使用有哪些规则（面试顺口答）？

## 答题要点

### 是什么

- **以 `use` 开头的函数**，内部调用其他 Hooks，把 **可复用的状态与副作用** 从组件里抽出来。  
- **不是** React API 里的新组件类型，而是 **逻辑复用约定**。

### 相对 mixin / HOC（预习篇逻辑）

- **mixin**：隐式合并、易冲突、难维护，**已废弃**。  
- **HOC**：嵌套、props 注入与调试成本；很多场景可用 **一个 `useXxx()`** 平铺表达。  
- **自定义 Hook**：调用关系显式、可按功能拆分文件，和 **函数组件 + 组合** 一致。

**为什么 mixin 被废弃？（预习篇原文总结）**

1. **依赖 `React.createClass`**：官方移除 `createClass` 后 mixin 失去载体。  
2. **命名冲突**：多个 mixin 若定义同名方法/state，会相互覆盖，排查困难。  
3. **隐式依赖**：mixin 之间可能有调用顺序要求，但代码里看不出来，**维护成本像滚雪球**。  
4. **来源不透明**：组件里用了某个 state/method，不知道来自哪个 mixin，调试与重构都难。

**自定义 Hook 如何避免这些问题？**

- **显式调用**：`const [state, setState] = useMyHook()` 一眼能看出数据来源。  
- **无命名冲突**：每次调用返回独立状态，变量名自己定，不会和其他 Hook 撞。  
- **依赖清晰**：Hook 内部用 `useEffect` 等声明依赖数组，ESLint 可检查；调用顺序由你控制。

### 规则（八股顺口溜）

- 只在 **函数组件或自定义 Hook 顶层** 调 Hook，勿放循环、条件、嵌套函数里。  
- **React 插件 / ESLint `react-hooks/rules-of-hooks`** 辅助约束。

## 一句话收尾

「有状态逻辑抽成 `useXxx`，显式复用；别用 mixin，HOC 按需、Hooks 优先。」
