# React Reconciler 与 Fiber 架构 - 函数调用流程

## 第一部分：宏观鸟瞰 - 函数调用流程

### 1. 前置知识：ReactElement 和 Reconciler 的关系

```
阶段1：创建 ReactElement（数据准备）
   开发者写 JSX
   ↓
   Babel 编译成 jsx() 调用
   ↓
   jsx() 返回 ReactElement 对象
   ↓
   const element = { $$typeof, type, props, ... }
   
   ─────────── 这里是分界线，element 只是个对象 ───────────
   
阶段2：触发渲染（用户主动调用）
   ReactDOM.render(element, container)  或  createRoot(container).render(element)
   ├─ 创建/获取 FiberRootNode
   ├─ 将 element 挂载到 rootFiber.current.child（待实现）
   ├─ 标记需要更新
   └─ 调用 renderRoot(rootFiber)  ← 进入 Reconciler
   
   ─────────── 从这里开始是本文重点 ───────────
   
阶段3：Reconciler 协调（本文重点）
   renderRoot(rootFiber)
   └─ workLoop 遍历 Fiber 树...
```

**关键说明：**
- **阶段1和阶段2不是自动连接的**
- ReactElement 创建后只是个 JavaScript 对象，不会自动触发渲染
- 必须手动调用 `ReactDOM.render()` 或 `createRoot().render()` 才会进入 Reconciler
- **本文档只讲解阶段3：Reconciler 内部的 workLoop 流程**

### 2. 核心流程图

```
入口：createRoot(container).render(element)
  ↓
renderRoot(rootFiber):
  ↓
  prepareFreshStack(rootFiber)
    设置 WIP = rootFiber
  ↓
  do...while(true) {
    调用 workLoop()
  }
  ↓

┌──────────────────────────────────────────────────────────────────────┐
│                                                                      │
│  workLoop():  ←═══════════════════════①②══════════┐                  │
│                                                   │                  │
│    while (WIP !== null):                          │                  │
│                                                   │                  │
│      performUnitOfWork(WIP):                      │                  │
│                                                   │                  │
│        next = beginWork(fiber)                    │                  │
│        fiber.memoizedProps = fiber.pendingProps   │                  │
│                                                   │                  │
│        if (next 存在):                             │                  │
│          WIP = next  ═════════════════════════════╡ ← ① 有子节点      │
│          继续循环                                  │                  │
│                                                   │                  │
│        else:                                      │                  │
│          completeUnitOfWork(fiber):               │                  │
│                                                   │                  │
│            node = fiber                           │                  │
│                                                   │                  │
│            do:  ←─────────③ 内部循环                │                  │
│                           │                       │                  │
│              completeWork(node)                   │                  │
│                           │                       │                  │
│              if (node.sibling 存在):               │                  │
│                WIP = sibling                      │                  │
│                return  ══════════════════════════╡ ← ② 有兄弟节点     │
│                           │                       │                  │
│              node = node.return                   │                  │
│              WIP = node                           │                  │
│                           │                       │                  │
│            while (node 不为 null) ─────────────────┘                  │
│                                                   │                  │
│            跳出循环时：node = null, WIP = null      │                  │
│                                                                      │
│  while 循环结束：WIP = null，退出                                      │
│                                                                      │
└──────────────────────────────────────────────────────────────────────┘
  ↓
完成遍历，Fiber 树已标记副作用

说明：
① 路径1（═══）：有子节点，向下递，回到 workLoop
② 路径2（═══）：有兄弟节点，横向遍历，回到 workLoop  
③ 路径3（───）：无兄弟节点，向上归，do...while 内部循环
```

### 3. 完整遍历示例：`<div><h1>A</h1><p>B</p></div>`

#### 初始状态

```
Fiber 树：
      div
     /   \
   h1     p
   |      |
  "A"    "B"

初始：WIP = div
```


---
#### 函数作用

**workLoop**
- **作用**：
遍历整个 Fiber 树，为每个 FiberNode 创建子节点（beginWork）和 DOM（completeWork）
- **产物**：完整的 Fiber 树（FiberNode 对象树） + 完整的 DOM 树（每个 FiberNode.stateNode 存储对应的 DOM 元素）


**performUnitOfWork**
- **作用**：调用 beginWork 处理当前 FiberNode，根据返回值决定继续递（有子节点）或开始归（无子节点，调用 completeUnitOfWork）
- **产物**：WIP 更新为下一个要处理的 FiberNode（子节点、兄弟节点或 null）


**beginWork**
- **作用**：为当前 FiberNode 创建所有子 FiberNode（如 div 创建 h1、p），通过 child 和 sibling 连接
- **产物**：当前 FiberNode 的所有子 FiberNode
- **示例**：`beginWork(divFiber)` → 创建 `h1Fiber` 和 `pFiber`，`divFiber.child = h1Fiber`，`h1Fiber.sibling = pFiber`


**completeUnitOfWork**
- **作用**：循环调用 completeWork，从当前 FiberNode 向上完成父节点，直到找到兄弟节点或到达根节点
- **产物**：部分或完整的 DOM 链（取决于是否找到兄弟节点）
- **示例1（找到兄弟）**：`completeUnitOfWork(textFiber "A")` → `completeWork(textFiber)` → `completeWork(h1Fiber)` → 找到兄弟 pFiber → 停止，WIP = pFiber
- **示例2（无兄弟）**：`completeUnitOfWork(textFiber "B")` → `completeWork(textFiber)` → `completeWork(pFiber)` → `completeWork(divFiber)` → 到达根节点，WIP = null


**completeWork**
- **作用**：为当前 FiberNode 创建真实 DOM 元素，将子 DOM 挂载到它身上
- **产物**：一个真实 DOM 元素（存储在 `fiber.stateNode`）
- **示例**：`completeWork(h1Fiber)` → `h1DOM = document.createElement('h1')` → `h1DOM.appendChild(textDOM)` → `h1Fiber.stateNode = h1DOM`

---
#### 执行流程

**第1轮 | WIP = div**

```
workLoop → performUnitOfWork(div) → beginWork(div)
  
  beginWork(div) 做什么：
    - 创建 h1 FiberNode（pendingProps = { children: 'A' }）
    - 创建 p FiberNode（pendingProps = { children: 'B' }）
    - 建立连接：h1.sibling = p，div.child = h1
    - 返回：h1（第一个子节点）
  
  有子节点 → WIP = h1
```

**第2轮 | WIP = h1**

```
workLoop → performUnitOfWork(h1) → beginWork(h1)
  
  beginWork(h1) 做什么：
    - 创建 "A" FiberNode
    - 建立连接：h1.child = "A"
    - 返回："A"（第一个子节点）
  
  有子节点 → WIP = "A"
```

**第3轮 | WIP = "A"**

```
workLoop → performUnitOfWork("A") → beginWork("A")
  
  beginWork("A") 做什么：
    - 文本节点无子节点
    - 返回：null
  
  无子节点 → completeUnitOfWork("A")
    
    completeWork("A")：创建文本节点 "A"
    无兄弟 → node = h1，继续循环
    
    completeWork(h1)：创建 h1 元素，appendChild("A")
    有兄弟 p！→ WIP = p，return
  
  回到 workLoop
```

**第4轮 | WIP = p**

```
workLoop → performUnitOfWork(p) → beginWork(p)
  
  beginWork(p) 做什么：
    - 创建 "B" FiberNode
    - 建立连接：p.child = "B"
    - 返回："B"（第一个子节点）
  
  有子节点 → WIP = "B"
```

**第5轮 | WIP = "B"**

```
workLoop → performUnitOfWork("B") → beginWork("B")
  
  beginWork("B") 做什么：
    - 文本节点无子节点
    - 返回：null
  
  无子节点 → completeUnitOfWork("B")
    
    completeWork("B")：创建文本节点 "B"
    无兄弟 → node = p，继续循环
    
    completeWork(p)：创建 p 元素，appendChild("B")
    无兄弟 → node = div，继续循环
    
    completeWork(div)：创建 div 元素，appendChild(h1)，appendChild(p)
    无兄弟 → node = null，WIP = null
  
  退出 completeUnitOfWork
```

**第6轮 | WIP = null**

```
workLoop：WIP = null，条件不满足，退出
```

#### 总结

```
遍历路径：
  div → h1 → "A" → (完成 h1) → p → "B" → (完成 p, div)

关键点：
  ① beginWork(div) 创建 h1 和 p，建立 h1.sibling = p 和 div.child = h1
  ② h1 完成后找到兄弟 p → return 回 workLoop → p 进行 beginWork
  ③ "B" 完成后没有兄弟 → 循环连续完成 p 和 div

函数作用总结：
  - workLoop：遍历整个 Fiber 树，得到完整的 Fiber 树（FiberNode 对象树）和 DOM 树（每个 FiberNode.stateNode 存储对应的 DOM 元素）
  - performUnitOfWork：调用 beginWork 处理当前 FiberNode，根据返回值决定继续递或开始归，更新 WIP 指向下一个要处理的节点
  - beginWork：为当前 FiberNode 创建所有子 FiberNode，通过 child 和 sibling 连接
  - completeUnitOfWork：循环调用 completeWork，从当前 FiberNode 向上完成父节点，直到找到兄弟节点或到达根节点
  - completeWork：为当前 FiberNode 创建真实 DOM 元素（存储在 fiber.stateNode），将子 DOM 挂载到它身上
```

## 第二部分：深入理解 - 核心问题解答

### **问题1：递阶段和归阶段分别做什么？（宏观理解）**

在理解具体函数之前，先从整体上理解 Fiber 遍历的两个阶段：

```
Fiber 树遍历 = 递阶段（向下） + 归阶段（向上）
```

#### 递阶段（beginWork）：构建 Fiber 树结构

```
目标：根据 ReactElement，生成/更新 FiberNode 子节点

主要工作：
1. 接收当前 fiber 节点
2. 根据 fiber.tag 判断节点类型（div？function 组件？文本？）
3. 获取 ReactElement（从 props.children 或 函数组件返回值）
4. 执行 Diff 算法：
   - 对比新的 ReactElement 和旧的 fiber.child
   - 决定是复用、更新还是删除
5. 生成新的子 FiberNode
6. 标记副作用 flags（Placement、Update、Deletion）
7. 返回第一个子节点

返回值：
- 有子节点：返回 child，继续向下递
- 无子节点：返回 null，开始向上归

本质：处理"父 → 子"的关系
```

**补充说明：什么是"获取 ReactElement"？**

```javascript
// 场景1：普通 DOM 元素（HostComponent）
// JSX:
<div className="container">
  <span>hello</span>
  <p>world</p>
</div>

// 编译后的 ReactElement:
{
  type: 'div',
  props: {
    className: 'container',
    children: [                    // ← 子元素在 props.children 里！
      { type: 'span', props: { children: 'hello' } },
      { type: 'p', props: { children: 'world' } }
    ]
  }
}

// beginWork 处理 div 时：
// 1. 发现 fiber.tag = HostComponent（普通 DOM）
// 2. 从 fiber.pendingProps.children 获取子 ReactElement
// 3. 拿到 [span, p] 两个 ReactElement
// 4. 为它们生成对应的 FiberNode


// 场景2：函数组件（FunctionComponent）
// JSX:
function App() {
  return <div>hello</div>;       // ← 函数的返回值就是子元素！
}
<App />

// App 对应的 ReactElement:
{
  type: App,                       // type 是函数本身
  props: {}
}

// beginWork 处理 App 时：
// 1. 发现 fiber.tag = FunctionComponent
// 2. 执行函数：const children = App(props)
// 3. 拿到返回的 ReactElement: { type: 'div', props: {...} }
// 4. 为 div 生成对应的 FiberNode


// 场景3：文本节点（HostText）
// JSX:
<span>hello</span>

// 编译后：
{
  type: 'span',
  props: {
    children: 'hello'              // ← 文本直接在 children 里
  }
}

// beginWork 处理 span 时：
// 1. 从 props.children 获取 'hello'
// 2. 为文本生成 FiberNode，tag = HostText
```

**核心理解：**
- **ReactElement** 是数据（描述要渲染什么）
- **Fiber** 是节点（用于工作调度）
- **beginWork 的任务**：把 ReactElement 转换成 Fiber 子节点
- 不同类型的组件，ReactElement 的来源不同：
  - 普通元素：`props.children`
  - 函数组件：`执行函数的返回值`

### **问题2：beginWork 执行后为什么要保存 props？**

理解这个问题需要知道 Fiber 中 props 的两个字段：

```javascript
class FiberNode {
  pendingProps: any;   // 新的 props（本次更新要使用的）
  memoizedProps: any;  // 旧的 props（上次渲染使用的）
}
```

#### 为什么需要两个 props？

```
场景：组件更新

初始渲染：
  <Button color="red" size="large">点击</Button>
  ↓
  fiber.pendingProps = { color: 'red', size: 'large', children: '点击' }
  fiber.memoizedProps = null（首次渲染）

更新渲染：
  <Button color="blue" size="large">点击</Button>
  ↓
  fiber.pendingProps = { color: 'blue', size: 'large', children: '点击' }
  fiber.memoizedProps = { color: 'red', size: 'large', children: '点击' }
                        ↑ 保留了上次的 props

作用：
1. beginWork 可以对比新旧 props
2. 如果 props 没变，可以跳过更新（优化）
3. 如果 props 变了，知道哪些属性需要更新
```

#### 为什么在 beginWork 后保存？

让我们用更清晰的逻辑来理解：

```javascript
performUnitOfWork(fiber):
  // 步骤1：beginWork 使用 pendingProps
  next = beginWork(fiber)

  // 步骤2：保存 props
  fiber.memoizedProps = fiber.pendingProps

  // 步骤3：决定下一步
  if (next) WIP = next
  else completeUnitOfWork(fiber)
```

**核心问题：beginWork 到底做了什么，让我们可以放心保存？**

#### beginWork 的完整工作流程

```javascript
// 伪代码：beginWork 内部实现
function beginWork(fiber) {
  const newProps = fiber.pendingProps;  // ← 1. 拿到新 props
  const oldProps = fiber.memoizedProps; // ← 2. 拿到旧 props

  // 3. 根据 fiber.tag 判断类型
  switch (fiber.tag) {
    case FunctionComponent:
      // 执行函数组件
      const Component = fiber.type;
      const children = Component(newProps);  // ← 使用 newProps！
      // 根据 children 生成子 FiberNode
      reconcileChildren(fiber, children);
      return fiber.child;

    case HostComponent: // div、span 等
      // 从 newProps.children 获取子元素
      const nextChildren = newProps.children;  // ← 使用 newProps！
      reconcileChildren(fiber, nextChildren);
      return fiber.child;

    case HostText:
      // 文本节点没有子节点
      return null;
  }
}

// reconcileChildren 的作用
function reconcileChildren(fiber, newChildren) {
  // 4. Diff 算法：对比新旧子节点
  //    - 旧子节点在哪？fiber.child（上次生成的）
  //    - 新子节点在哪？newChildren（从 pendingProps 来的）
  
  // 5. 生成新的子 FiberNode
  //    - 根据 newChildren 创建/更新 FiberNode
  //    - 设置子节点的 pendingProps
  //    - 标记副作用 flags
  
  // 6. 连接关系
  fiber.child = newChildFiber;
}
```

#### 关键理解：beginWork 完成后发生了什么？

**重要区分：ReactElement vs FiberNode**
```
ReactElement = 数据描述（存在 props.children 里）
  例如：{ type: 'span', props: { children: 'hello' } }

FiberNode = 工作节点（存在 fiber.child 里）
  例如：{ tag: HostComponent, type: 'span', pendingProps: {...} }
```

**beginWork 的转换过程：**

```
执行前：
  fiber.pendingProps = { 
    color: 'blue', 
    children: [                           ← ReactElement 数组
      { type: 'span', props: {...} },     ← ReactElement（数据）
      { type: 'p', props: {...} }         ← ReactElement（数据）
    ]
  }
  fiber.memoizedProps = { color: 'red', children: [...] }
  fiber.child = 旧的 span FiberNode      ← FiberNode（工作节点）

beginWork(fiber) 执行：
  ✓ 读取 pendingProps.children（ReactElement 数组）
  ✓ 对比旧的 fiber.child（FiberNode）
  ✓ 根据 ReactElement 生成新的 FiberNode
  ✓ 设置新 FiberNode 的 pendingProps
  ✓ 更新 fiber.child 指向新的 span FiberNode

执行后：
  fiber.child = 新的 span FiberNode      ← 已更新
  fiber.child.sibling = 新的 p FiberNode ← 兄弟节点
  
  pendingProps 的信息已经"转移"到子 FiberNode 中：
  - pendingProps.children（ReactElement）→ fiber.child（FiberNode）
  - 子节点的 props 也设置好了

此时可以安全保存：
  fiber.memoizedProps = fiber.pendingProps
  ↓
  意思是："我已经根据这个版本的 props 生成了子节点，
          下次更新时，这个就是旧版本了"
```

#### 总结：为什么 beginWork 后可以保存？

```
原因1：pendingProps 已经被"消费"了
  └─ props.children 已经转换成子 FiberNode
  └─ 其他属性（color、style）会在 completeWork 用

原因2：标记"已处理"状态
  └─ memoizedProps = pendingProps 表示"我已经用这个版本生成了子节点"
  └─ 下次更新时，这个就是"旧版本"

原因3：为下次更新做准备
  └─ 下次更新：新的 pendingProps 来了
  └─ 对比 memoizedProps（旧）和 pendingProps（新）
  └─ 决定是否需要重新生成子节点
```

**类比：**
```
beginWork = 根据设计图纸（pendingProps）生产零件（子 FiberNode）
保存 props = 把图纸归档到"已生产"文件夹（memoizedProps）
下次更新 = 新图纸来了，对比"已生产"图纸，看是否需要重新生产
```

### **问题3：为什么 renderRoot 需要 do...while(true) 循环？**

#### 真实的 React 18 实现逻辑

```javascript
function renderRootSync(root, lanes) {
  // 只在需要时准备新栈（条件判断）
  if (workInProgressRoot !== root || workInProgressRootRenderLanes !== lanes) {
    prepareFreshStack(root, lanes);  // ← 在循环外面！
  }
  
  do {
    try {
      workLoopSync();  // 遍历 Fiber 树
      break;           // 成功完成，跳出循环
    } catch (e) {
      handleThrow(root, e);  // ← 关键：不是简单重置，而是智能处理
    }
  } while (true);
}
```

#### do...while(true) 的真实作用

**不是"完全重试"，而是"错误恢复和继续"**

```
handleThrow 的智能处理：

├─ Suspense 错误：
│  └─ 设置 workInProgress 为 Suspense 节点，继续循环渲染 fallback
│
├─ 错误边界：
│  └─ 设置 workInProgress 为最近的错误边界，继续循环渲染错误 UI
│
└─ 致命错误：
   └─ workInProgress = null，退出循环

核心：prepareFreshStack 在循环外，只初始化一次
  → 不是"从头再来"，而是"从中断点继续"
```

**类比：**
- 玩游戏遇到 Boss 失败 → 自动存档 → 从存档点继续（不是从头开始）

**我们的简化实现：**

```javascript
function renderRoot(root: FiberNode) {
  prepareFreshStack(root);  // ✓ 在循环外面，正确
  
  do {
    try {
      workLoop();
      break;
    } catch (e) {
      workInProgress = null;  // ⚠️ 简化版：直接退出，不会重试
    }
  } while (true);
}
```

- React 18：`handleThrow` 处理 Suspense、错误边界，可以继续
- 我们：直接 `workInProgress = null`，会退出循环
- 结论：学习阶段的简化版本

### **问题4：为什么 workLoop 需要循环调用 performUnitOfWork？**

#### workLoop 的代码

```javascript
function workLoopSync() {
  while (workInProgress !== null) {  // ← 为什么需要循环？
    performUnitOfWork(workInProgress);
  }
}
```

#### 如果不用循环会怎么样？

```javascript
// 错误尝试：只调用一次
function workLoopSync() {
  if (workInProgress !== null) {
    performUnitOfWork(workInProgress);
  }
  // ❌ 结束了！只处理了一个节点
}
```

**用具体例子说明：**

```
Fiber 树：
      div
     /   \
   h1     p

执行流程：
  1. WIP = div
  2. 调用 performUnitOfWork(div)
     - beginWork(div) → 返回 h1
     - WIP = h1
  3. 函数结束 ❌
  
结果：
  只处理了 div 节点
  h1 和 p 都没有被处理
```

#### 为什么必须是循环？

因为 **performUnitOfWork 只处理一个节点**：

```javascript
function performUnitOfWork(fiber) {
  const next = beginWork(fiber);  // 处理当前节点，返回子节点
  fiber.memoizedProps = fiber.pendingProps;
  
  if (next === null) {
    completeUnitOfWork(fiber);  // 无子节点，开始归
  } else {
    workInProgress = next;  // ← 设置下一个要处理的节点，但不处理！
  }
}
```

**关键理解：**
- performUnitOfWork 只处理**当前**节点
- 它会设置 `workInProgress = next`（指向下一个节点）
- 但它**不会**去处理 next 节点
- 需要 workLoop 的循环再次调用 performUnitOfWork 才能处理 next

**总结：**

workLoop 必须是循环，因为：
- performUnitOfWork 只处理一个节点
- 它设置下一个节点，但不处理
- 需要循环再次调用才能继续遍历整个树

### **问题5：为什么 completeUnitOfWork 需要循环？**

#### completeUnitOfWork 的代码

```javascript
function completeUnitOfWork(fiber) {
  let node = fiber;
  
  do {  // ← 为什么需要循环？
    completeWork(node);
    const sibling = node.sibling;
    
    if (sibling !== null) {
      workInProgress = sibling;
      return;  // 找到兄弟，退出循环
    }
    node = node.return;  // 向上到父节点
    workInProgress = node;
  } while (node !== null);
}
```

#### 如果不用循环会怎么样？

```javascript
// 错误尝试：不用循环
function completeUnitOfWork(fiber) {
  let node = fiber;
  
  completeWork(node);  // 只执行一次
  
  if (node.sibling !== null) {
    workInProgress = node.sibling;
    return;
  }
  node = node.return;
  workInProgress = node;
  // ❌ 结束了！只向上了一层
}
```

**用具体例子说明：**

```
Fiber 树：
      div
       |
      h1
       |
    "标题"  ← 叶子节点，无子节点

处理 "标题" 节点时：

不用循环的情况：
  1. completeWork("标题")  ✓
  2. 没有兄弟节点
  3. node = h1
  4. workInProgress = h1
  5. 结束 ❌
  
结果：
  只完成了 "标题" 节点
  h1 没有被 completeWork
  div 也没有被 completeWork
  DOM 树构建不完整！
```

#### 为什么必须是循环？

因为**可能需要连续向上处理多个父节点**：

```
使用循环的情况：

处理 "标题" 节点：
  
  第1次循环：
    completeWork("标题")  ✓
    没有兄弟
    node = h1
    继续循环  ← 关键！
  
  第2次循环：
    completeWork(h1)  ✓
    没有兄弟
    node = div
    继续循环
  
  第3次循环：
    completeWork(div)  ✓
    没有兄弟
    node = null（到达根节点）
    退出循环

结果：
  "标题"、h1、div 都被 completeWork 处理了
  DOM 树从下往上正确构建 ✓
```

#### 为什么是从下往上？

completeWork 的作用是创建 DOM 节点并组装：

```
DOM 构建顺序（必须从下往上）：

1. completeWork("标题")
   → 创建文本节点：document.createTextNode("标题")
   → 保存到 fiber.stateNode

2. completeWork(h1)
   → 创建 h1 元素：document.createElement("h1")
   → 把子节点的 DOM 挂载：h1.appendChild(文本节点)
   → 保存到 fiber.stateNode

3. completeWork(div)
   → 创建 div 元素：document.createElement("div")
   → 把子节点的 DOM 挂载：div.appendChild(h1)
   → 保存到 fiber.stateNode

如果不连续处理父节点：
  无法正确执行 appendChild
  DOM 树结构错误
```

**总结：**

completeUnitOfWork 必须是循环，因为：
- 需要连续向上处理多个父节点
- 每个父节点都需要被 completeWork（创建 DOM、执行 appendChild）
- 如果不是循环，只能处理一层父节点
- DOM 树无法从叶子节点向上正确构建

### **问题6：completeWork 的作用，为什么执行完后就能找 sibling 或 return？**

#### completeWork 做什么？

**真实的 React 18 中 completeWork 的核心工作：**

```javascript
function completeWork(current, workInProgress, renderLanes) {
  const type = workInProgress.type;
  const newProps = workInProgress.pendingProps;
  
  switch (workInProgress.tag) {
    case HostComponent: // div、span 等 DOM 元素
      // 1. 创建 DOM 节点
      const instance = document.createElement(type);
      
      // 2. 设置 DOM 属性
      setInitialProperties(instance, type, newProps);
      
      // 3. 将子节点的 DOM 挂载到当前节点
      appendAllChildren(instance, workInProgress);
      
      // 4. 保存 DOM 到 fiber
      workInProgress.stateNode = instance;
      return null;
      
    case HostText: // 文本节点
      const text = newProps;
      const textInstance = document.createTextNode(text);
      workInProgress.stateNode = textInstance;
      return null;
  }
}
```

**核心：completeWork 只负责"完成"当前节点**
- 创建 DOM
- 设置属性
- 收集子节点的 DOM（appendChild）
- 不关心下一步去哪里

#### 为什么执行完 completeWork 后就能找 sibling 或 return？

**因为 completeWork 已经完成了当前节点的所有工作！**

```javascript
do {
  completeWork(node);  // ← 完成当前节点的工作
  
  // 当前节点已完成，接下来该处理谁？
  if (node.sibling !== null) {
    workInProgress = sibling;  // 有兄弟，处理兄弟
    return;
  }
  
  node = node.return;  // 无兄弟，向上到父节点
  workInProgress = node;
} while (node !== null);
```

**具体例子：**

```
Fiber 树：
    div
   /   \
  h1    p
  |     |
 "A"   "B"

处理 "A" 节点（叶子节点）：

1. completeWork("A")
   - 创建文本节点
   - 保存到 fiber.stateNode
   - 工作完成 ✓
   
2. 找兄弟？
   - "A" 没有兄弟
   
3. node = h1（父节点）
   继续循环

4. completeWork(h1)
   - 创建 h1 元素
   - appendChild(文本节点 "A")  ← 挂载子节点
   - 工作完成 ✓
   
5. 找兄弟？
   - 有兄弟 p！
   - WIP = p
   - return ← 退出 completeUnitOfWork

回到 workLoop，处理 p 节点...
```

#### 为什么有兄弟就 return，没兄弟就继续循环？

**遍历策略：深度优先 + 兄弟优先**

```
优先级顺序：
1. 子节点（child）        → beginWork 处理
2. 兄弟节点（sibling）    → completeUnitOfWork 找到后 return
3. 父节点（return）       → completeUnitOfWork 循环处理

为什么这个顺序？
  - 子节点：先向下递，构建树结构
  - 兄弟节点：同一层级，找到后回到 workLoop 继续递
  - 父节点：本层和兄弟都处理完，才向上归
```

**总结：**

completeWork 的作用：
- 创建 DOM 节点
- 设置 DOM 属性
- 收集并挂载子节点的 DOM
- 只负责"完成"当前节点，不管下一步

为什么执行完就能找 sibling/return：
- completeWork 已完成当前节点的所有工作
- 接下来按优先级：先找兄弟（同层），再找父节点（向上）
- 找到兄弟就 return（回到 workLoop 继续递）
- 没有兄弟就继续循环（处理父节点）

### **问题7：为什么找到 sibling 后就 return，但父节点要继续循环？**

#### 代码中的两种路径

```javascript
do {
  completeWork(node);  // 完成当前节点
  
  if (node.sibling !== null) {
    workInProgress = sibling;
    return;  // ← 路径1：有兄弟，立即退出
  }
  
  node = node.return;  // ← 路径2：无兄弟，向上到父节点
  workInProgress = node;
  // 继续循环，执行 completeWork(父节点)
} while (node !== null);
```

#### 为什么有兄弟就 return？

**因为兄弟节点还没有被处理过（既没递也没归）！**

```
Fiber 树：
    div
   /   \
  h1    p  ← h1 的兄弟
  |
 "A"

当前处理到 h1：
  - h1 已经递过了（beginWork 处理过）
  - h1 的子节点 "A" 也递过了
  - 现在 "A" 和 h1 都归完了（completeWork 完成）
  
找到兄弟 p：
  - p 还没有被 beginWork 处理！
  - p 需要先"递"（生成子节点）
  - 所以要 return 回到 workLoop
  - 让 workLoop 调用 performUnitOfWork(p)
  - 然后 beginWork(p) 才能处理 p
```

**关键：Fiber 遍历是"递 → 归"的过程**
- beginWork（递）：必须在 performUnitOfWork 中执行
- completeWork（归）：在 completeUnitOfWork 中执行
- 找到兄弟节点时，要先让它"递"，所以必须 return 回到 workLoop

#### 为什么父节点要继续循环（不 return）？

**因为父节点已经递过了！只需要归就行。**

```
Fiber 树：
    div  ← 父节点
     |
    h1
     |
    "A"  ← 当前处理完了

执行过程：
  1. 之前 div 已经递过了（beginWork 处理过）
     - beginWork(div) 生成了子节点 h1
  
  2. 现在处理到 "A"（叶子节点）
     - "A" 没有子节点，开始归
     - completeWork("A") ✓
     - 没有兄弟，node = h1
  
  3. h1 之前也递过了
     - beginWork(h1) 生成了子节点 "A"
     - 现在只需要归
     - 继续循环，completeWork(h1) ✓
     - 没有兄弟，node = div
  
  4. div 之前也递过了
     - 现在只需要归
     - 继续循环，completeWork(div) ✓
```

**为什么不需要 return？**
- 父节点已经在之前的 beginWork 中处理过了
- 现在只需要执行 completeWork
- completeWork 可以在 completeUnitOfWork 的循环中直接执行
- 不需要回到 workLoop

#### 对比两种情况

```
有兄弟节点：
  当前节点：已递 ✓，已归 ✓
  兄弟节点：未递 ❌，未归 ❌
  
  → 兄弟节点需要先递
  → return 回到 workLoop
  → 让 performUnitOfWork 调用 beginWork

无兄弟节点：
  当前节点：已递 ✓，已归 ✓
  父节点：  已递 ✓，未归 ❌
  
  → 父节点只需要归
  → 继续循环执行 completeWork
  → 不需要回到 workLoop
```

**总结：**

找到 sibling 后 return 的原因：
- 兄弟节点还没递过，需要先 beginWork
- beginWork 必须在 performUnitOfWork 中调用
- 所以要 return 回到 workLoop

父节点继续循环的原因：
- 父节点已经递过了，只需要归
- completeWork 可以在循环中直接执行
- 不需要回到 workLoop

### **问题8：退出 completeUnitOfWork 循环后会发生什么？**

#### completeUnitOfWork 的退出条件

```javascript
function completeUnitOfWork(fiber) {
  let node = fiber;
  
  do {
    completeWork(node);
    
    if (node.sibling !== null) {
      workInProgress = sibling;
      return;  // ← 退出方式1：找到兄弟节点
    }
    
    node = node.return;
    workInProgress = node;
  } while (node !== null);  // ← 退出方式2：node = null
}
```

**两种退出方式：**
1. **找到兄弟节点**：return 退出
2. **到达根节点**：node = null，循环条件不满足

#### 退出方式1：找到兄弟节点

```
Fiber 树：
    div
   /   \
  h1    p
  |
 "A"

处理完 h1 时：
  do {
    completeWork("A") ✓
    无兄弟，node = h1
    
    completeWork(h1) ✓
    有兄弟 p！
    workInProgress = p
    return  ← 退出 completeUnitOfWork
  }

退出后发生什么？
  → 回到 performUnitOfWork 的调用处
  → 回到 workLoop 的循环
  → while (workInProgress !== null)  // WIP = p
  → 继续循环，调用 performUnitOfWork(p)
  → beginWork(p)  // 开始处理兄弟节点
```

**关键：找到兄弟后，回到 workLoop 继续递**

#### 退出方式2：到达根节点

```
Fiber 树：
  div  ← 根节点
   |
  h1
   |
  "A"

处理完整个树后：
  do {
    completeWork("A") ✓
    无兄弟，node = h1
    
    completeWork(h1) ✓
    无兄弟，node = div
    
    completeWork(div) ✓
    无兄弟，node = null  ← div 的 return 是 null
    
    workInProgress = null
  } while (null !== null)  // 条件不满足，退出循环

退出后发生什么？
  → 回到 performUnitOfWork 的调用处
  → 回到 workLoop 的循环
  → while (workInProgress !== null)  // WIP = null
  → 条件不满足，退出 workLoop
  → renderRoot 结束
```

**关键：WIP = null，整个 Fiber 树遍历结束**

#### 完整的执行流程

```
renderRoot(root):
  prepareFreshStack(root)  // WIP = root
  
  do {
    workLoop()  ← 开始遍历
  } while (true)

workLoop():
  while (workInProgress !== null) {  ← WIP = root 开始
    performUnitOfWork(workInProgress)
    
    // 经过多次循环...
    // 最终 completeUnitOfWork 设置 WIP = null
    
  }  ← WIP = null，退出循环
  
  // workLoop 结束

renderRoot 继续执行：
  - 标记渲染完成
  - 返回退出状态
  - 进入 commit 阶段（将 DOM 变更提交到页面）
```

**总结：**

退出 completeUnitOfWork 后的两种情况：

1. **找到兄弟节点**（return 退出）：
   - WIP = sibling
   - 回到 workLoop 继续循环
   - 开始处理兄弟节点（beginWork）

2. **到达根节点**（循环条件不满足）：
   - WIP = null
   - 回到 workLoop，条件不满足
   - workLoop 退出
   - renderRoot 结束
   - 进入 commit 阶段
